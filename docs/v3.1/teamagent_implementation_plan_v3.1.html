1.本書について

1.1目的・関連文書（v3.1 群）

本書は TeamAgent v3.1 の実装計画を、7ヶ月・14 Sprint × 2週間で具体化するものである。v3.1 は v3.0 の Phase 構造を踏襲しつつ、Agent ホストを OpenClaw（オープンソース、Anthropic Agent SDK 互換）にフル採用、Skill は ClawHub のホワイトリスト を活用することで、2026年12月本格運用開始を実現する。S2 末（6月後半）に Go/No-Go ゲート①（OpenClaw 採用継続判断）、S10 末（10月後半）に Go/No-Go ゲート②（提案書 20h→8-12h 実証）を新設。コスト構造は「開発予算 ¥80K 総額上限 / 運用予算 月¥100K-1M レンジ」に分離。

v3.1 の意図
v3.0 までは Claude Agent SDK をベースに Skill Registry も含めて自前構築する方針だったが、(a) OpenClaw が Agent SDK 互換のオープンソース実装として既に存在し、Skill Hub（ClawHub）も整っていること、(b) 創業者の OpenAI 移籍でプロジェクト健全性のリスクは残るが Apache-2.0 で fork 可能なため致命的でないこと、(c) Bedrock 直接接続で Anthropic 政策変動からも遮断できること、を踏まえてフル採用に転換した。これにより MVA を S6（M3 末）で出して 5 名ベータ、S14（M7.5）= 2026年12月後半に本格運用開始を実現する。S2 末ゲート①で「OpenClaw 採用継続 or 自前ラップに戻すか」を再判断する撤退路を確保。

3.Phase 別計画

各 Sprint は 目的・成果物・DoD・テスト工程・リスク の 5 観点で記述する。

v3.1 の Sprint 再マッピング
v3.1 では 14 Sprint × 2週間 = 7ヶ月 に圧縮しているため、以下の Sprint 詳細表は v3.0 の 17 Sprint ベースの記述を含む。v3.1 の正式 Sprint 配置は 第 2.2 節の S1〜S14 詳細表を参照し、第 3 章の Sprint 詳細は 各 Skill の DoD / テスト工程の参考資料として扱うこと（OpenClaw 採用により S1-S2 の Agent ホスト構築工数は削減され、データ層着手が前倒し）。

3.1MVA Phase 1（M0.5〜M1.5 / S1〜S3）OpenClaw + データ層連携

目的：初期 Skill セット5個すべてが依存する 共通基盤と 3 データ層（Gmail / Drive / Slack）を pgvector に流し込む。MVA で「検索 Skill」を成立させる最小データを揃える。

Sprint S1（M1 W1-W2）— インフラセットアップ + pgvector 構築

目的AWS インフラ（Single-AZ）+ RDS PostgreSQL with pgvector + IaC（Terraform / CDK）一式
成果物(1) VPC / ECS / RDS pgvector / S3 / Secrets Manager。(2) IaC リポジトリ。(3) CI/CD（GitHub Actions）。(4) CloudWatch ベースライン
DoD・terraform apply で全リソース作成可能
・pgvector 拡張が RDS で有効化されている
・GitHub Actions の CI が green
・CloudWatch メトリクスが収集されている
・脅威モデリング初版（DREAD）が完了
テスト単体（IaC 構文 lint）／結合（apply→destroy）／セキュリティ（IAM 最小権限チェック）
リスクR-S1a: AWS IAM 権限不足で apply 失敗 → 事前に運用部門と権限確認

Sprint S2（M1 W3-W4）— Gmail OAuth + 取り込み

目的Gmail Pub/Sub watch + 差分取り込み + Embedding 化 + pgvector 格納
成果物(1) Gmail OAuth 同意フロー（社内 GWS 管理者経由）。(2) Pub/Sub watch + Cloud Functions or Lambda。(3) Embedding pipeline（text-embedding-3-large 候補）。(4) opt-out スイッチ
DoD・営業 1 名の Gmail がリアルタイムで pgvector に格納される
・本人 / 他人ラベル付き
・opt-out で取り込み停止可能
・監査ログが CloudWatch に出力される
テスト単体（パーサ）／結合（OAuth → Pub/Sub → Embedding → pgvector）／受入（営業 1 名 1 週間運用）／セキュリティ（OAuth scope 監査）
リスクR14: OAuth スコープの選定（Domain-wide vs 個別） — 子会社ヒアリングを並行

Sprint S3（M2 W1-W2）— Drive OAuth + 取り込み

目的Google Drive の社内資料を差分同期し、PDF/PPTX/DOCX を抽出して pgvector へ
成果物(1) Drive watch（changes.list）。(2) PDF/PPTX 抽出 pipeline（Tika / pdfplumber / python-pptx）。(3) チャンク化 + Embedding
DoD・社内 Drive 共有ドライブの全資料がインデックス化される
・PPTX のスライド単位でチャンク化される
・抽出失敗率 < 3%
・更新差分が 1 時間以内に反映される
テスト単体（抽出ロジック）／結合（Drive→抽出→pgvector）／回帰（PPTX バリエーション 20 件）／性能（5,000 ファイル取り込み時間）
リスクR-S3a: 暗号化 PDF が抽出失敗 → 「要確認」化、エラーログで運用検知

Sprint S4（M2 W3-W4）— Slack OAuth + 取り込み + メタデータ層

目的Slack Events API + 案件メタデータ統合スキーマの確立
成果物(1) Slack Bolt App + Event subscription。(2) public/private チャンネルの取り込み。(3) メタデータ層スキーマ（案件 ID / 顧客 / フェーズ）
DoD・営業の主要チャンネルがインデックス化される
・案件メタデータが Gmail / Slack / Drive を横断結合できる
・DM は opt-in 設定で取り込み
テスト単体（Event parser）／結合（3 データ層横断検索）／受入（営業 2 名で「自分の過去案件」検索が動く）
リスクR-S4a: Slack rate limit → backoff + キュー化

3.2MVA Phase 2（M2.5 / S5）Agent SDK 基本ループ + Skill Registry 雛形

目的：Claude Agent SDK を組み込み、「Slack で呼ばれる → Skill Registry から Skill を選択 → pgvector を検索する → 回答する」のループを成立させる。Skill Registry の構築は本 Phase に含む（拡張 Skill 受け入れの基盤）。

Sprint S5（M2.5）— Agent SDK 組み込み + Skill Registry 雛形 + pgvector 検索 Tool + Slack 受け口

目的Claude Agent SDK の基本ループを完成させ、Skill Registry 雛形を組み込み、Slack から呼べる状態に
成果物(1) Agent SDK ベースの Agent ランタイム（Orchestrator）。(2) Skill Registry 雛形（Skill 登録 / 発見 / 実行インターフェース）。(3) pgvector 検索 Tool（hybrid search：BM25 + embedding）。(4) Slack Bolt の app_mention handler。(5) 引用付き回答テンプレート
DoD・@TeamAgent にメンション → 3 秒以内に応答開始
・Skill Registry に Skill を登録／発見できる
・回答に Tier ラベル付き引用が含まれる
・hybrid search の recall@10 ≥ 0.85（手動評価 30 件）
・捏造率（引用に無い数値）= 0%
テスト単体（Tool 呼び出し）／結合（Slack→Agent→検索→Slack 応答）／性能（応答 SLO 3 秒）／受入（営業 2 名で 30 問評価）
