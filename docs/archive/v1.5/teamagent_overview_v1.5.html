1.本書について

1.1目的

本書は、ショート動画 PR 施策を主業とする営業組織（20名規模）に対し、社内専用の Slack AI エージェント「TeamAgent」を導入する構想の全体像とシステム概要を説明することを目的とする。提案書作成業務（1案件あたり約20時間）を対象とし、Slack 上での自然言語インタフェースを介して、Gmail・Slack・Google Drive・TikTok 動画ナレッジを横断する組織記憶と外部リサーチ（Gemini Deep Research）を統合し、提案書作成時間を 8〜12 時間に短縮する。

本書はあくまで 構想と概要 の説明文書であり、詳細な実装仕様は別途「SDD（System Design Document）」に分冊する。本書を読了することで、経営層は意思決定の論点を、実装担当者は構築物の輪郭を、それぞれ把握できる粒度で記述する。

1.2対象読者

本書は以下の3つの役割を兼ねる。役割ごとに章構成を読み進めることを推奨する。

営業メンバー（システム初心者を含む）第2章（使い方）のみで業務イメージを掴める。理解を深めたい場合は第4章（目指す姿）→ 第6章（ユースケース）
経営層・取締役第3章（背景）→ 第4章（目指す姿）→ 第8章（投資効果）→ 第9章（リスク管理）
推進オーナー（高林氏）第4章 → 第5章（システム構成）→ 第6章（ユースケース）→ 第7章（実装計画）
実装担当（社内エンジニア・SES）第5章 → 第6章 → 第7章 + 関連 SDD 文書

1.4用語定義

TeamAgent
本書で扱う Slack AI エージェントの総称。営業16名が @TeamAgent メンションで呼び出し、提案書生成・類似事例サジェスト・動画ナレッジタグ付けを行う。
Phase 1 / Phase 1.5
Phase 1 は高林構想（個別 Claude.ai ライセンスによる手動オーケストレーション）。Phase 1.5 は TeamAgent による組織レイヤーへの統合実装。両者は並走し、9月時点で統合判断を行う。
Step05
高林構想で定義する全21工程のうち、提案書自動生成工程。Gemini Deep Research → Claude Projects → Google Colab の3ステップで構成される。本構想の中核。
提案書 FMT
67スライド・84プレースホルダで構成される提案書フォーマット。｛N：ラベル｝ 形式の placeholder を Composer が値で埋め、Slidewriter が python-pptx で置換する。
Co-Authoring
Gemini DR 用プロンプトの共同執筆方式。TeamAgent が過去成功プロンプトを基に3案を提示し、営業が選択・編集する。完全自動化と完全手動のあいだの中庸案。
VSEO Analytics
本構想着手時点で本番稼働中の TikTok 分析システム。vseo.manus.space でホスティング。Puppeteer による TikTok スクレイピング、Bedrock Claude による LLM 分析、3シークレットによる重複度分析を提供する。TeamAgent は tRPC API で疎結合接続する。
Champion User
営業16名のうち、5/29 ロールアウト時に先行稼働開始する2名。残り14名のオンボーディングを支援する役割を持つ。
Drive RAG
Google Drive 内の過去提案 PDF（1,500件）を pgvector に格納し、類似度検索で関連事例を引く仕組み。「直近重視」の重み付けスコアリングを採用する。
サジェスト / ephemeral
Slack 上で「あなたにのみ表示」される一時的なメッセージ。他のメンバーには見えないため、Bot からの提案で会話が荒れない。

2.TeamAgent の使い方（営業向け解説）

本章は、システムに詳しくない営業メンバーや経営層を主な読者として、TeamAgent を実際にどう使うのかを Slack の画面イメージとともに具体例で説明する。技術的詳細は第6章以降を参照。

📌 一言で言うと

TeamAgent は Slack の中にいる「営業の右腕 AI」です。 商品名と URL を伝えるだけで、過去の提案資料や市場調査、TikTok 動画分析を統合した提案書（パワポ）を 10〜20 分で作って Slack に届けてくれます。 使い方は @TeamAgent と打つだけ。Excel やパワポを開く必要はありません。

2.1TeamAgent でできる3つのこと

営業の皆さんから見ると、TeamAgent は次の3つの機能を持つ「Slack の中の助手」です。

📝

① 提案書を作る

商品名と URL、目的を Slack で伝えると、過去事例 + 市場調査 + 競合分析 + TikTok 動画分析を統合した パワポ提案書 を生成して Slack に届けます。

所要 10〜20分
💡

② 過去事例を教えてくれる

普段通り Slack で議論していると、Bot が「これに似た過去事例ありますよ」と 過去の提案 PDF や勝ちパターン動画 をこっそり教えてくれます（自分にだけ見える形）。

瞬時
🎬

③ TikTok 動画を分析

気になった TikTok の URL を /teamagent-tag で投げると、フック・CTA・訴求軸など5項目を自動でタグ付け してナレッジに保存。次回以降の提案書で引用できます。

所要 30秒

2.4よくある質問（FAQ）

営業の方々から想定される疑問に、事前に回答しておきます。

Q1. AI が間違った内容を提案書に書いてしまったら、どうなりますか？

まず予防として、Bot は 出典のない記述を書かない 設計です。提案書の各セクション末尾には引用元 URL（Drive PDF / 過去 Slack 議論 / Gemini DR）が必ず併記されます。出典が見つからなかった項目には自動的に「[要確認]」マークが付きます。

それでも誤りに気づいた場合：提案書の該当メッセージを 右クリック → 「ハルシ報告」を選ぶだけ。Modal が開くので「P5 の競合データが事実と違う」のように一行書いて送るだけで、1時間以内に Ops 担当が一次応答、24時間以内に「原因と対策」レポートが #teamagent-ops チャンネルで共有されます。

Q2. 自分の Gmail や Slack の DM・プライベートチャンネルも見られちゃうの？

いいえ、見られません。TeamAgent がアクセスできるのは以下に限定されています：

  • Bot 専用の Gmail 受信箱（bot@org.com）— 営業個人のメールは閲覧不可
  • Slack の Bot が招待されたパブリックチャンネルのみ — DM、プライベートチャンネル、Bot がいないチャンネルは見えない
  • Google Drive の指定フォルダ内 PDF（過去提案書 1,500件）— 個人の Drive は見えない

全 NDA 締結済の社内資料のみが対象です。個人情報が漏れる構造的リスクはありません。

Q3. 慣れるのに時間かかりますか？操作が難しそう…

基本操作は @TeamAgent と打つだけです。Excel やパワポを開く必要も、特別な画面を覚える必要もありません。

オンボーディング 1on1（30分）で Champion + Ops が同伴して初回案件を一緒に作るので、独学で習得する必要はありません。1回やれば次回からひとりでできます。慣れない場合は何度でも 1on1 を予約できます。

Q4. もし全然使わなかったら、怒られますか？

怒られません。利用は強制ではなく、各自のペースで使うことを前提にしています。週次で「今週の利用状況」DM が来ますが、うざい場合は /teamagent-mute で停止、隔週化、opt-out も可能です。

ただし経営は KPI（提案書作成時間）を見ているので、使った人と使わなかった人の差は中長期的には可視化されます。「自分で全部やる方が速い」と感じるなら、それも有効な判断です。

Q5. 既存の Gemini Advanced や Claude.ai の個人ライセンスはどうなる？

9月の本番化判断までは そのまま継続使用できます。TeamAgent と並走させ、両方使える状態で実利用データを集めます。

9月時点で利用実績を見て、16名 → 5〜8名（営業マネージャ層のみ）に縮減する方針です。ただし「壁打ち用途」（Bot に頼まず思考整理に使いたい）は残します。CFO + 外部評価者の2名委員会で最終判断します。

Q6. 提案書の質が悪かったら、どう改善されますか？

.pptx 投稿後の 「修正した」ボタンを押すと、修正後の .pptx をアップロードできます（任意）。修正前後の差分が S3 に保存され、月次で Ops + Champion がレビューします。

特定のセクションがいつも修正される場合、Composer のプロンプトが改善されます。リリースは毎週火曜 22:00。1ヶ月使えば徐々に "あなた専用" の品質に育っていきます。

Q7. 急にシステムが落ちたり、変な動きをしたらどうすれば？

Slack の #teamagent-status チャンネルでステータスを確認できます。Ops 担当が15分以内に一次応答する SLA で運用されます（営業時間内）。

深刻な不具合の場合は、/teamagent-rollback Slash コマンドで30秒以内に前バージョンに戻せます。インシデントは月次レポートで全営業に共有されるので、隠蔽リスクはありません。

Q8. 個人にお金がかかったり、契約書にサインさせられたりしますか？

かかりません・サインしません。TeamAgent の運用費（インフラ ¥35K + Claude API ¥150K = 月¥250K 上限）は会社が一括負担します。営業個人には課金は発生しません。

PoC（試行）期間中は会社の負担で実験するので、気軽に触ってみるのが正しい姿勢です。

Q9. Bot のアウトプットを「自分の手柄」にしていいの？

むしろ推奨されます。TeamAgent は「営業の右腕」であり、最終的な提案・受注は 営業の判断と顧客との信頼関係で決まる。Bot は資料準備を肩代わりするだけです。

一方で、Co-Authoring で良いプロンプトを編集したエース営業は、その貢献が gemini_dr_prompts.contributed_by で記録され、月次レポートで「プロンプト師範」として顕彰されます。プロンプト経由でも組織貢献が可視化される設計です。

Q10. 「Bot に仕事を奪われる」って感じない？

奪われるのは「過去事例を Drive で全文検索する作業」「競合動画を一本ずつ観る作業」「パワポのテンプレに値を流し込む作業」など、本来の営業価値（顧客理解・提案・関係構築）ではない部分です。

高林構想の言葉を借りれば「AIが業務の7割を代行、人は判断と創造に集中」。削減した時間で、これまで手が回らなかった新規開拓や顧客フォローに振り向けるのが目的です。

5.4既存資産の活用

本構想の特筆すべき点は、ゼロベース構築ではなく、既に稼働中の2つの社内システムを Slack に統合する 点である。これにより成功確率は構造的に押し上げられ、PoC 工数も大幅に圧縮される。

5.4.1 高林氏 Step05 資産

高林構想の Step05「提案書自動生成3ステップ」は、既に高林氏のもとで運用されている。本構想の着手時点で利用可能な資産は以下のとおりである。

  • Gemini DR マスタープロンプト（運用中）：A〜H の8カテゴリ（市場・社会潮流・TikTokハッシュタグ・パブリシティ・界隈・競合・インサイト・イベント）を構造化 JSON として収集する。
  • Claude Projects プロンプト（完成済）：Gemini DR の出力 JSON を入力として、提案書 FMT を生成する。
  • 提案書 FMT 雛形 .pptx（完成済）：67スライド・84プレースホルダ構造。｛N：ラベル｝ 形式で値が埋まる。
  • 文脈設計フレーム：「メディア・界隈 → 頻出ワード → 右脳・左脳 → 訴求軸 → 表現」の5段階フレーム。

TeamAgent はこれらを継承して自動化する。具体的には PromptDrafter の初期ライブラリ、Composer の基盤プロンプト、Slidewriter の置換テンプレートとして移植する。

5.4.2 VSEO Analytics

VSEO Analytics は vseo.manus.space で本番稼働中の TikTok 分析システムである。Puppeteer + Bright Data 日本プロキシによる TikTok 内部 API スクレイピング、Bedrock Claude による LLM 分析、3シークレットコンテキストによる重複度分析（パーソナライズ排除・勝ちパターン特定）を提供する。

TeamAgent は VSEO の MySQL を直接参照せず、tRPC API（analysis.runAnalysis 等）経由で疎結合接続する。これにより、VSEO の本番運用を停止せず、両システムの障害が相互に波及しない設計を維持する。VSEO の出力は提案書 FMT のうち C_tiktok（Slide 40 の界隈マトリクス）と F_competitor セクションに直接マッピングされる。

特筆すべき点として、VSEO の 3シークレット重複度分析（勝ちパターン特定） は高林構想の Step20 を超える機能であり、提案書に「業界の勝ちパターン動画 TOP5」セクションを追加できる。これは TeamAgent の独自セールスポイントとなる。

PromptDrafter の3案提示は、完全自動化ではなく「営業の判断は残す」設計の核である。FDE 経験則によれば、3案提示は営業ペルソナによって「選択肢疲労」を起こすケースが多いため、ラベルは「A/B/C」ではなく意味的タグ（「競合動画分析重視」「ターゲット深掘り」等）を採用する。さらに「おまかせ」ボタンを併設し、選びたくない営業はワンクリックで実行できる。

6.2類似事例サジェスト（パッシブ）

TeamAgent は全パブリックチャンネルに自動 join し、メッセージイベントを監視する。直近5発言を embedding 化し、cosine 類似度 ≥ 0.82 の過去事例（提案書 PDF / VSEO 動画 / Slack 過去議論）が見つかった場合、ephemeral でサジェストする。

営業の心理的反発を避けるため、以下の制約を課す。

  • 同一チャンネルでは 30分のクールダウン
  • 3回連続「不要」評価で 24時間自動ミュート
  • /teamagent-mute 2h で手動ミュート可
  • サジェストは ephemeral（投稿者のみ可視）

6.3動画ナレッジ収集

TikTok 動画ナレッジは、VSEO Analytics の本番稼働システムを活用する。営業が /teamagent-tag <tiktok_url> を実行すると、TeamAgent は VSEO API で既存メタデータを参照、未登録の場合は yt-dlp で動画を取得し、Sonnet 4.6 ビジョンで以下の5項目をタグ付けする。

  1. メディア・界隈（プラットフォーム + ジャンル）
  2. フック・頻出ワード（冒頭3秒のフックパターン）
  3. 訴求軸（右脳訴求 / 左脳訴求）
  4. CTA 形式（プロフィールリンク誘導 / コメント誘導 等）
  5. 表現（編集スタイル・テンポ）

タグ付け結果は pgvector に embedding として格納され、提案書生成時に Researcher が search_tiktok_kb ツールで類似動画を引いて引用する。

12.1背景：なぜ複数データソースの併用が必要か

TikTok 広告分析の世界では、単一のデータソースで完結する分析はほぼ不可能である。理由は、TikTok 公式 API が公開している指標が限定的（再生・いいね・コメント・シェア）であり、保存数（save count）・広告消化額・3シークレット重複度といった「効果の質」を見極める指標は、それぞれ別のサードパーティ分析サービスを横断する必要があるためである。

本構想では既に VSEO Analytics（自社運用）が稼働しており、加えて運用中のスクレイピングパイプラインにより dpro と Buzzmiru の2つの商用分析サービスからもデータを取得している。3者は重複ではなく 役割分担 の関係にある。

12.2Buzzmiru の独自価値：保存数（save count）

Buzzmiru の最大の独自価値は 保存数の可視化 である。これは TikTok 公式 API では基本的に取得不可能で、自分の動画の Creator Center か、Buzzmiru のような逆算分析ツールでしか見えない指標である。dpro / VSEO とも保存数は取得していない。

保存数が営業の提案精度に効く理由
  • 記憶定着の質的指標：保存は「後で見返したい / 共有したい」という意思の表明であり、いいねよりも購買意欲・記憶定着の強いシグナル。
  • 「再生は伸びたが効果が薄い動画」と「再生は普通だが刺さった動画」を分離可能：いいね率だけでは判別できない隠れた当たり動画を発見できる。
  • 競合分析の差別化要素：他社が見ていない指標で評価することで、提案書の説得力が一段上がる。
  • 提案書 FMT への直接マッピング：C_tiktok セクション（界隈マトリクス）および F_competitor セクションの根拠データを強化する。

すなわち、サーバー落ちの頻度と引き換えにしても Buzzmiru は維持すべきデータソース である。本章の AWS 移行案は、不安定さを前提として「落ちている時の付き合い方」を仕組みで吸収する設計を提案する。

12.5不安定サイトへの耐性設計（Buzzmiru 対策）

Buzzmiru のサーバー落ちは こちら側で根本解決できない外部要因 である。AWS 移行の本質的価値は、「落ちている時の挙動」を賢くする仕組みを24時間自動で回せる点にある。具体的には3層の耐性設計を提案する。

① 死活監視 Lambda（5分おき HEAD）

② 指数バックオフ + 諦めライン（Step Functions）

取得失敗時は Step Functions で 30分後 → 1時間後 → 3時間後 → 6時間後 とリトライ。24時間ダメなら諦めて翌日朝 9:00 に再挑戦。リトライ間隔・最大試行回数はパラメータ化する。

12.6Claude Agent SDK による複数ソース横断分析

AWS 移行の真の狙いは、定期取得そのものではなく 取得したデータを Claude Agent SDK が横断的に読み込んで分析できる素地を作る ことにある。現状は3ソースが Sheets タブとして分離しているため、TeamAgent が「保存数の高い動画 × 消化額」「いいね率は低いが保存数 Top5%」のような横断クエリを実行することが困難である。

SDK 横断分析の典型例（提案書生成中の挙動）
  1. 営業が @TeamAgent 提案書つくって 商品=ACME美容液 を投げる
  2. Composer サブエージェントが pgvector に問い合わせ：「美容液カテゴリ・直近30日・保存数 Top5%」
  3. SDK が dpro データ（消化額・配信期間）と Buzzmiru データ（保存数）と VSEO データ（動画内容分析）を 動画 ID で JOIN して取得
  4. 「再生は中位だが保存数が異常に高い隠れ当たり動画」3本を競合事例として提案書 C_tiktok セクションに埋め込み
  5. 営業に「この3本は保存数が高い＝記憶定着が強い動画です」と根拠付きで提示

これが成立するには、3ソースのデータが 動画 ID をキーとして同一 DB（pgvector）に集約 されている必要がある。AWS 移行と同時に Apps Script + Sheets だけでなく RDS PostgreSQL + pgvector への二重書き込み を行うことで、Sheets は人間用ダッシュボード、pgvector は SDK 用データ層、という役割分担が成立する。
