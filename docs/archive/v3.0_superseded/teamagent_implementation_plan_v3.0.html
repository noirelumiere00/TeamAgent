1.本書について

1.1目的・関連文書（v3.0 群）

本書は TeamAgent v3.0 の実装計画を、9ヶ月・17 Sprint で具体化するものである。v3.0 は v2.2 の Skill 設計を継承しつつ、MVA（Minimum Viable Agent）を最初の 3 ヶ月で立ち上げる構成に切り替えた。具体的には、データ層連携 → Agent SDK ループ（Skill Registry 構築を含む）→ 超軽量検索 Skill という最短経路で「Slack で動く Agent」を最速で見せ、それから 初期 Skill セット5個を Skill Registry に登録するフェーズ（Phase 4）・本番品質の UX/Guardrails（Phase 5）へと進む。Phase 5+ では 拡張 Skill（業務固有 / ユーザー定義 / 高林プロンプト Skill 等）の追加余地を残す。

構想・概要書teamagent_overview_v3.0.html — Skill Registry + 初期 Skill セット5個 × 6 データ層 × 4 共通基盤 の v3.0 整理
要件定義書teamagent_requirements_v3.0.html — FR / NFR / DR / SR を v3.0 で再整理
アーキテクチャ図teamagent_architecture_v3.0.html — システム構成と Slack 入口の設計
MVA 仕様書mva_spec_v1.html — 本書 Phase 1-3 の確定仕様（S1〜S6 の DoD 詳細）

初期 Skill B（動画分析）確定仕様書teamagent_kinou4_spec_v1.html — Phase 4 S7-S8 で参照
初期 Skill D（提案生成）Phase 0-E 仕様書teamagent_phase0E_spec_v1.html — Phase 4 S11-S12 で参照

v3.0 の意図
v2.2 までは「Skill B（動画分析）から作り、最後に Skill A（検索）を仕上げる」順序だったが、運用初期に 営業に見える成果物が薄いという課題があった。v3.0 では 「データ層 → Agent ループ（Skill Registry 構築を含む）→ 超軽量な検索 Skill」を最初の 3ヶ月で完成させ、M3 末に営業 5 名ベータを開始することで、価値仮説の検証を前倒しする。重い Skill（動画・Mail・提案生成）は Phase 4 で初期 Skill セットとして Skill Registry に順次登録、UX / Guardrails / 本番化は Phase 5 でまとめる。Phase 5+ では拡張 Skill（業務固有 / ユーザー定義 / 高林プロンプト Skill 等）を追加可能。

3.Phase 別計画

各 Sprint は 目的・成果物・DoD・テスト工程・リスク の 5 観点で記述する。

3.1MVA Phase 1（M1〜M2 / S1〜S4）データ層連携

目的：初期 Skill セット5個すべてが依存する 共通基盤と 3 データ層（Gmail / Drive / Slack）を pgvector に流し込む。MVA で「検索 Skill」を成立させる最小データを揃える。

Sprint S1（M1 W1-W2）— インフラセットアップ + pgvector 構築

目的AWS インフラ（Single-AZ）+ RDS PostgreSQL with pgvector + IaC（Terraform / CDK）一式
成果物(1) VPC / ECS / RDS pgvector / S3 / Secrets Manager。(2) IaC リポジトリ。(3) CI/CD（GitHub Actions）。(4) CloudWatch ベースライン
DoD・terraform apply で全リソース作成可能
・pgvector 拡張が RDS で有効化されている
・GitHub Actions の CI が green
・CloudWatch メトリクスが収集されている
・脅威モデリング初版（DREAD）が完了
テスト単体（IaC 構文 lint）／結合（apply→destroy）／セキュリティ（IAM 最小権限チェック）
リスクR-S1a: AWS IAM 権限不足で apply 失敗 → 事前に運用部門と権限確認

Sprint S2（M1 W3-W4）— Gmail OAuth + 取り込み

目的Gmail Pub/Sub watch + 差分取り込み + Embedding 化 + pgvector 格納
成果物(1) Gmail OAuth 同意フロー（社内 GWS 管理者経由）。(2) Pub/Sub watch + Cloud Functions or Lambda。(3) Embedding pipeline（text-embedding-3-large 候補）。(4) opt-out スイッチ
DoD・営業 1 名の Gmail がリアルタイムで pgvector に格納される
・本人 / 他人ラベル付き
・opt-out で取り込み停止可能
・監査ログが CloudWatch に出力される
テスト単体（パーサ）／結合（OAuth → Pub/Sub → Embedding → pgvector）／受入（営業 1 名 1 週間運用）／セキュリティ（OAuth scope 監査）
リスクR14: OAuth スコープの選定（Domain-wide vs 個別） — 子会社ヒアリングを並行

Sprint S3（M2 W1-W2）— Drive OAuth + 取り込み

目的Google Drive の社内資料を差分同期し、PDF/PPTX/DOCX を抽出して pgvector へ
成果物(1) Drive watch（changes.list）。(2) PDF/PPTX 抽出 pipeline（Tika / pdfplumber / python-pptx）。(3) チャンク化 + Embedding
DoD・社内 Drive 共有ドライブの全資料がインデックス化される
・PPTX のスライド単位でチャンク化される
・抽出失敗率 < 3%
・更新差分が 1 時間以内に反映される
テスト単体（抽出ロジック）／結合（Drive→抽出→pgvector）／回帰（PPTX バリエーション 20 件）／性能（5,000 ファイル取り込み時間）
リスクR-S3a: 暗号化 PDF が抽出失敗 → 「要確認」化、エラーログで運用検知

Sprint S4（M2 W3-W4）— Slack OAuth + 取り込み + メタデータ層

目的Slack Events API + 案件メタデータ統合スキーマの確立
成果物(1) Slack Bolt App + Event subscription。(2) public/private チャンネルの取り込み。(3) メタデータ層スキーマ（案件 ID / 顧客 / フェーズ）
DoD・営業の主要チャンネルがインデックス化される
・案件メタデータが Gmail / Slack / Drive を横断結合できる
・DM は opt-in 設定で取り込み
テスト単体（Event parser）／結合（3 データ層横断検索）／受入（営業 2 名で「自分の過去案件」検索が動く）
リスクR-S4a: Slack rate limit → backoff + キュー化

3.2MVA Phase 2（M2.5 / S5）Agent SDK 基本ループ + Skill Registry 雛形

目的：Claude Agent SDK を組み込み、「Slack で呼ばれる → Skill Registry から Skill を選択 → pgvector を検索する → 回答する」のループを成立させる。Skill Registry の構築は本 Phase に含む（拡張 Skill 受け入れの基盤）。

Sprint S5（M2.5）— Agent SDK 組み込み + Skill Registry 雛形 + pgvector 検索 Tool + Slack 受け口

目的Claude Agent SDK の基本ループを完成させ、Skill Registry 雛形を組み込み、Slack から呼べる状態に
成果物(1) Agent SDK ベースの Agent ランタイム（Orchestrator）。(2) Skill Registry 雛形（Skill 登録 / 発見 / 実行インターフェース）。(3) pgvector 検索 Tool（hybrid search：BM25 + embedding）。(4) Slack Bolt の app_mention handler。(5) 引用付き回答テンプレート
DoD・@TeamAgent にメンション → 3 秒以内に応答開始
・Skill Registry に Skill を登録／発見できる
・回答に Tier ラベル付き引用が含まれる
・hybrid search の recall@10 ≥ 0.85（手動評価 30 件）
・捏造率（引用に無い数値）= 0%
テスト単体（Tool 呼び出し）／結合（Slack→Agent→検索→Slack 応答）／性能（応答 SLO 3 秒）／受入（営業 2 名で 30 問評価）
リスクR12: Embedding モデル選定の不確実性 → S5 中に 3 モデル A/B 比較

3.3MVA Phase 3（M3 / S6）超軽量・検索 Skill（Skill Registry に登録）

目的：「過去事例を検索して引用付きで返す」 1 Skill だけを完成させ、Skill Registry に登録し、営業 5 名にベータ提供する。MVA の中核 Sprint。

Sprint S6（M3）— 検索 Skill 実装（Skill Registry 登録）+ 営業 5 名ベータ + FB 収集

目的Agent SDK 上に「検索 Skill」を最小実装して Skill Registry に登録し、営業 5 名でベータ運用
成果物(1) 検索 Skill（Tier 0-3 引用ラベル必須、Skill Registry 登録済み）。(2) Slack ephemeral 応答 UI（カード形式）。(3) FB 収集フォーム（thumbs up/down + 自由記述）。(4) ベータ運用ドキュメント
DoD・営業 5 名が 2 週間日常運用
・1 日平均 5 回以上の利用
・回答満足度 ≥ 70%
・捏造率 = 0%、引用率 ≥ 90%
・FB 30 件以上を分類して次 Phase 4 にインプット
テスト受入（5 名 2 週間）／監査ログ（全クエリの記録）／カオス（pgvector 障害時のフォールバック）
リスクR11: MVA 期間中の見える成果物の薄さ → 検索 Skill で価値を可視化（本 Sprint で対応）
MVA リリースゲート（M3 末）
S6 完了時点で Go/No-Go ゲートを設置。満足度 ≥ 70% + 捏造率 0% + 1 日 5 回以上利用 の 3 条件を満たせば Phase 4 へ進む。未達なら S6 を 2 週間延長してプロンプト / 検索チューニング。

3.4Phase 4（M4〜M6 / S7〜S12）初期 Skill セット5個を Skill Registry に登録

目的：MVA で検証した Agent 基盤・Skill Registry の上に、初期 Skill セット5個（Skill A 検索＝MVA で登録済み、Skill B 動画分析・Skill C Mail・Skill D 提案・Skill E 自動サジェスト・出力整形 Skill）を順次実装し Skill Registry に登録。営業 10 名展開まで進める。

Sprint S7（M4 W1-W2）— 初期 Skill B ①：動画ナレッジ分析 MVP（Skill Registry 登録）

Skill B（動画分析）仕様書 v1 を参照。Gemini 2.5 Flash の動画ネイティブ機能を主軸に。

目的Slack に貼られた動画 URL を Gemini で分析 → 構造化レポート
成果物(1) URL リゾルバ（YouTube Shorts / IG / TikTok）。(2) Gemini 動画 prompt + パーサ。(3) Slack カード出力
DoD・3 プラットフォーム対応
