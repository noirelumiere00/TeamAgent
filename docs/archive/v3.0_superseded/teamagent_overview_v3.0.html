1.本書について

1.1目的

本書は TeamAgent の構想とシステム概要を v3.0 として再定義するものである。 v2.2 までは「営業向け 5 機能を順番に開発する」資料自動化プロダクトとして設計されていたが、 v3.0 では Skill レジストリ型の社内 AI Agent プラットフォーム として再構成する。すなわち、 Claude Agent SDK を中核に据えた汎用 Agent 基盤と Skill レジストリを先に構築し、その上に初期 Skill セット（5 個）を段階展開する構成へ転換する。 Skill は core を改修することなく無限に追加可能とする。

本書が再定義するのは以下の 4 点である：

  • ① 設計思想：プロダクト（5 機能固定）→ プラットフォーム + Skill レジストリへの再構成
  • ② アーキ：Claude Agent SDK 単独（OpenClaw 不採用）+ レイヤード構成（Skill レジストリ / データ層 / Guardrails / Memory）による役割分離
  • ③ 開発戦略：MVA（Minimum Viable Agent）— データ層連携を先に固め、Skill は段階展開
  • ④ 営業 UX：メンション → Slash Command → Workflow Builder の 3 段階で習熟に合わせて進化

本書は技術仕様書として、Skill レジストリ / Tools (MCP) / データ層 / Memory・Session / Permissions・Guardrails の標準的なレイヤー名で統一して記述する。

1.2対象読者

  • 経営層・監査役：プロダクト構想の再構成意図を理解する
  • 営業部門マネージャー・営業 16 名：初期 Skill セット（5 個）の使い方と段階展開計画を理解する
  • 開発・インフラ担当：Claude Agent SDK 中心の単独構成と MVA 戦略を理解する
  • 情報セキュリティ・コンプライアンス：Guardrails と Permissions の役割を理解する

1.3v2.2 からの主要変更点

v3.0 は単なるバージョンアップではなく 設計思想の再構成 である。以下 5 点を改訂履歴として明示する。
変更カテゴリ v2.2 までの設計 v3.0 の再定義
① プロダクト定義 営業向け資料自動化 5 機能の集合体（機能③ 提案コンテンツ生成を中心） Skill レジストリ型の社内 AI Agent プラットフォーム。Agent 基盤 + Skill レジストリを先に構築し、初期 Skill セット（5 個）を順次登録。Skill は core 改修なしに無限拡張可能
② Agent 構成 Claude Agent SDK (Planner) + OpenClaw (Executor) の 2 スタック想定 Claude Agent SDK 単独。Planner + Executor を兼ねるシングルスタック（OpenClaw 不採用）
③ アーキ表現 システム機能ベース（機能①〜⑤、データ層、共通基盤の機械的記述） レイヤード Agent アーキ。User / Frontend / Orchestrator（Agent）/ Skill レジストリ / Tools (MCP) / データ層 / Memory / Guardrails の役割分離で記述
④ 開発戦略 Phase ④ → ② → ③ → ① の機能直列開発（17 Sprint / 8.5 ヶ月） MVA 戦略。データ層連携を P1〜P3（〜3 ヶ月）で固め、Skill は Phase 4 以降に段階展開
⑤ 高林プロンプト 機能③ 提案コンテンツ生成の Skill 化を Phase A〜E で進める前提 現運用と並走。MVA には含めず、Phase 4 以降の Skill 化候補として温存

1.4用語定義

用語定義
AgentLLM を中核とし、自律的に Tool を選択して目標達成するソフトウェア。本書では Claude Agent SDK 上で動作する個体を指す。
SkillAgent が特定のタスクに特化して持つ処理パイプライン。プロンプト + Tool 呼び出しシーケンス + 出力フォーマットの組。本書の初期 Skill セット（5 個）はそれぞれ 1 Skill として実装される。
Skill レジストリAgent が呼び出し可能な Skill を登録・管理する仕組み。新規 Skill は core を改修することなくレジストリに登録できる。Skill カタログ（利用可能機能リスト）として Planner が参照する。
ToolAgent が呼び出す具体的な機能（Gmail API、Slack API、pgvector 検索、HTML 生成など）。Tool は MCP として実装される場合が多い。
MCPModel Context Protocol。Agent と外部システムの接続を標準化するプロトコル（MCP 接続層）。Anthropic が主導。
MemoryAgent が過去の会話・タスク履歴（Session 履歴）を保持する仕組み。
SessionMemory のうち、現在進行中の会話文脈。短期記憶。
GuardrailsAgent の出力・行動に対する制約（Permissions、出力フィルタ、コンプラチェッカーなど）。
pgvectorPostgreSQL の拡張で、ベクトル検索を可能にする。embedding ベースの意味検索基盤として採用。
Tier 0-3引用ラベル。Tier 0=社内一次情報、Tier 1=社内分析、Tier 2=外部公式、Tier 3=外部リサーチ。データドリブン度の担保に用いる。
MVAMinimum Viable Agent。最小実装で動く Agent。v3.0 で採用する開発戦略の核心概念。MVP の Agent 版。

2.AI Agent としての TeamAgent — Skill レジストリ型アーキテクチャ

v3.0 のアーキは Skill レジストリ型 Agent プラットフォームとして記述する。 Orchestrator（Agent）が中核で、Planner / Executor の両方を兼ね、Skill レジストリ・Tools (MCP 接続層)・Memory / Session・Permissions / Guardrails を統括する。 データ層（入力データ）は分離独立し、出力はレンダリングレイヤを経て Slack に返る。

Skill は Skill Registry（プラグイン基盤） 上で管理され、初期 Skill セット（5 本）以外にも、業務固有 Skill や利用者定義 Skill など、無限に拡張可能なアーキテクチャを採用する。

2.3「5 機能固定」から「Skill レジストリ + 初期 Skill セット」への再フレーム

v2.2 では「5 機能 = 5 つのプロダクトモジュール（固定）」だったが、v3.0 では 「Skill レジストリ型 Agent プラットフォーム + 初期 Skill セット 5 個」として、Skill を無限に拡張可能な構造に再定義する。初期 Skill セットは Phase 4 で展開する 5 個（動画分析 / Mail / 提案生成 / Slack サジェスト / 進捗サマリー）。

再フレームすることで以下のメリットが生まれる：

  • 基盤側（Agent SDK・データ層・Memory・Guardrails・Skill レジストリ）を 1 度だけ作れば良い。機能ごとに重複実装しない。
  • Skill の追加・削除・更新が 独立して行える。Skill 同士が干渉せず、core を改修する必要がない。
  • 初期 Skill セット（5 個）の枠を超えた追加 Skill（議事録要約、商談ロープレ、業務固有 Skill、ユーザー定義 Skill、高林プロンプト Skill、外部連携 Skill 等）を組織のニーズに応じて段階的に追加可能。
  • 営業の利用パターンが固まる前にプロダクトを乱造せず、MVA で検索 Skill を先に出して観察 → ニーズに合った Skill から順次実装。

2.4Skill レジストリの拡張ポイント（プラグイン構造）

Skill レジストリは プラグイン構造として設計する。新規 Skill は次の 3 ファイルをレジストリ配下に追加するだけで Agent から呼び出し可能となり、core（Agent SDK、データ層、Guardrails）を改修する必要はない：

  • skill.yaml — Skill 名、説明、必要 Tool、必要 Permissions、入出力スキーマ
  • prompt.md — Skill 固有のシステムプロンプト / 処理パイプラインの記述
  • handler.py — Tool 呼び出しシーケンスと出力整形ロジック

将来の拡張パスとしては、(a) 業務固有 Skill（部門特化）、(b) ユーザー定義 Skill（個人テンプレート）、(c) 高林プロンプト Skill（並走運用フェーズの後に Skill 化）、(d) 外部連携 Skill（CRM / SFA / BI 等）を想定する。 レジストリは内部 Skill カタログとして Planner に提示され、Agent は User のリクエストに応じて適切な Skill を選択する。

3.営業向け使い方ガイド（初期 Skill セットと検索 Skill）

v3.0 では MVA で 検索 Skill（超軽量・全データ層横断検索）が最初に登場し、その後 Phase 4 で初期 Skill セット（5 個）が段階展開される。 営業から見ると、Slack で同じ TeamAgent に話しかける窓口は 1 つで、内部的に Agent がレジストリからどの Skill を使うかを判断する。

3.1🔍 検索 Skill（MVA で最初にレジストリに登録される Skill）

🔍 検索 Skill MVA Phase 3 でリリース
使い方： @TeamAgent A 社の前回提案、何だっけ？
動作： Mail / Drive / Slack / 過去提案 HTML を横断検索し、関連情報を Tier ラベル付きで返す。
意義： 「資料を作る」前段の 「思い出す・調べる」 をカバー。MVA 期間中の見える成果物を担保。

3.2初期 Skill セット（5 個）の体験例

🎬 動画ナレッジ分析 Skill Phase 4 で実装、別冊あり
使い方： @TeamAgent このインフルエンサーの動画から訴求軸を抽出して
動作： YouTube / TikTok / Instagram の動画を分析し、訴求軸・ベネフィット・NG ワードを整理。
別冊： 『機能④ 確定仕様書 v1』（teamagent_kinou4_spec_v1.html）参照。
📧 Mail ワークフロー Skill
使い方： 朝出社して Slack を開くと、自動で前日 Mail の要約と返信案が届く。
動作： Gmail の未読を Tier ラベル付きで分類、重要案件は深掘りモードで返信ドラフトを生成。
📝 提案コンテンツ生成 Skill
使い方： @TeamAgent B 社向けに化粧品 PR の提案書を、過去 A 社案件を踏まえて
動作： 5 フェーズ（要件把握 → リサーチ → 骨子 → 本文 → 仕上げ）で HTML 提案書を生成。対話型で 5 割→8 割完成度に。
💬 Slack 自動サジェスト Skill
使い方： Slack の DM やチャンネルで「提案書作って」と書くと、TeamAgent が ephemeral でサジェストを出す。

📊 営業進捗サマリー Skill
使い方： @TeamAgent 今週の自分の案件進捗まとめて
動作： Mail + Slack + Drive を横断し、進行中案件 / 停滞案件 / 次アクションを HTML サマリーで出力整形。

4.目指す姿

4.1俯瞰：4 レイヤー × 4 要素

v3.0 ではプロダクト全体を 4 レイヤー で捉える。各レイヤーは Agent 基盤の中で異なる責務を持つ。

レイヤー 役割 v3.0 の主要構成要素
① データ層 入力データの供給（Tier 0-3） Gmail OAuth / Slack Bolt / Drive API / 動画分析パイプ / Gemini DR / X / pgvector index
② Agent SDK Orchestrator（思考・実行） Claude Agent SDK（Planner + Executor 兼務）/ Tool 呼び出し / Memory / Session
③ Skill レジストリ層 処理パイプライン（Skill）のカタログと拡張ポイント 初期 Skill セット 5 個（動画分析 / Mail / 提案生成 / Slack サジェスト / 進捗サマリー）+ 検索 Skill + 将来追加 Skill（業務固有 / ユーザー定義 / 外部連携 等）
④ Guardrails 層 Permissions / 横断制約 Permissions / 出力フィルタ / 薬機法・景表法チェッカー / 監査ログ / PII マスキング

各レイヤーは 独立して進化 させられる。データ層の拡張（例：CRM 連携）は Skill に影響を与えず、 新 Skill の追加（例：商談ロープレ）はデータ層・Guardrails の改修を伴わない。

5.システム構成

5.1全体アーキテクチャ（Claude Agent SDK 単独構成）

  • ① 運用複雑度：2 スタック構成は障害切り分け・バージョン管理が複雑化。
  • ② Claude Agent SDK の機能拡充：2026 年現在、SDK 側がサブエージェント呼び出し・並列実行をネイティブサポート。
  • ③ コスト：Executor を別スタックで持つ意義が薄れた（料金体系が一本化された方が見通しが良い）。
  • ④ 子会社依存リスク：戦略 / 価格 / 機能優先順位が子会社判断で変動する不確実性を排除。

6.MVA 戦略

v3.0 で新規導入。MVA（Minimum Viable Agent）は、機能を絞った最小実装で Agent を「とにかく動かす」開発戦略。MVP の Agent 版。

6.1MVA の哲学：データ層先行

Agent の品質は 入力データ（データ層）の質と取り回しやすさ でほぼ決まる。 どれほど Orchestrator（Agent）が優秀でも、データ層が劣化していたり取り出しにくければ、Skill がどれだけ揃っていても期待する出力は得られない。 そこで v3.0 では、Skill を先に作らず、まず データ層を整備する。これが MVA の哲学である。

v2.2 までは「機能④（動画分析）が一番派手なので先に作る」という機能優先の論理だったが、 v3.0 では「データ層が整っていれば後から Skill はいくらでも乗せられる」という基盤優先の論理に転換する。

8.HTML 出力と編集・PDF/PPTX 変換

v3.0 で新規追加。Skill の出力形式を HTML を標準 とし、PDF / PPTX への変換を二系統で持つ運用を明文化する。

8.1出力形式の標準：HTML

TeamAgent の Skill が生成する成果物（提案書・進捗サマリー・動画分析レポート）は、すべて HTML を一次フォーマット として出力する。 これは v2.0 以降の方針を継承するもので、以下の理由による：

  • ① LLM との親和性：Claude は HTML をそのまま生成可能で、Markdown 経由の劣化を避けられる。
  • ② Tailwind / CSS 変数でテーマ統一が容易。社内ブランドの一貫性を担保。
  • ③ contenteditable 属性で営業が直接編集できる WYSIWYG エディタ化が可能（§ 8.4）。
  • ④ PDF / PPTX への変換ルートが両方確立されている。

8.2HTML → PDF：Puppeteer / Playwright 経由

HTML を PDF に変換するには Puppeteer（Chrome ヘッドレス）または Playwright を使用する。 印刷用 CSS（@media print）を HTML 側に書いておけば、ページ区切り・ヘッダー / フッターも制御可能。

項目仕様
変換エンジンPuppeteer（軽量 / 既定）/ Playwright（マルチブラウザ検証用）
呼び出し方Skill から MCP 経由で html-to-pdf Tool を呼び出す
用紙A4 / Letter 切替可能。デフォルトは A4 縦
埋め込みフォントNoto Sans JP / Noto Serif JP を base64 埋め込み

8.3HTML → PPTX：2 層テンプレ + python-pptx

PowerPoint 出力は HTML テンプレ と PPTX テンプレ を 2 層で持って同期する設計を採る。 HTML 側で構造・テキストを生成し、PPTX 側のプレースホルダに python-pptx で値を流し込む。 

      Agent が生成した構造化データ（JSON）
              │
      ┌───────┴───────┐
      ▼               ▼
  HTML テンプレ    PPTX テンプレ
  （表示・編集用） （Excel 入稿対応・社内配布用）
      │               │
      ▼               ▼
   Slack 表示      python-pptx で
                  プレースホルダ流し込み
                       │
                       ▼
                  営業 / 顧客送付用 .pptx

2 層を 同じ JSON データ から生成することで、HTML 側を編集しても PPTX を再生成すれば同期される。 PPTX の単独編集は推奨しない（差分管理が困難になる）。

8.4営業向け WYSIWYG エディタ（Claude Artifacts 型）

営業が Agent の出力をそのまま渡すのではなく、「ちょっと直したい」 要求に応えるため、 Claude Artifacts 型の contenteditable HTML + Tailwind エディタを Slack 内 or Web で提供する。

  • 編集 UI：HTML の各要素を contenteditable で直接編集（テキスト変更・段落追加・順序変更）。
  • 変更検知：DOM の差分を JSON 化して保存。再変換時に PPTX 側にも反映。
  • エクスポート：「PDF でダウンロード」「PPTX でダウンロード」「Slack に再投稿」の 3 ボタン。
  • 導入時期：Phase 4-c（蒸す Skill リリース）と同時。

9.UX 階段：メンション → Slash Command → Workflow Builder

営業の習熟度に合わせて、Slack 入口の UI 形態を 3 段階で進化 させる。 最初は柔軟な自然言語、慣れたら構造化入力、最終的にフォーム UI と、利用者の負荷と精度をバランスさせる。

9.1MVA 期間：@TeamAgent メンション

最初の UX は 自由な自然言語。営業が @TeamAgent A 社の前回提案、何だっけ？ と書けば、 Agent が意図を解釈して検索 Skill を呼び出す。

  • 長所：学習コストゼロ、何でも聞ける。
  • 短所：意図検出が不安定な場合、Agent が何度も聞き返す。
  • 導入時期：MVA Phase 3 から。検索 Agent と同時。

9.2Phase 4：Slash Command

Skill が増えてくると、自然言語だけでは Agent が誤解する確率が上がる。そこで Slash Command で意図を明示する形を追加する。

  • /teamagent-create-proposal クライアント=B 社 商材=化粧品 PR
  • /teamagent-summarize-mail 範囲=昨日 重要度=高のみ
  • /teamagent-analyze-video url=https://...

長所：意図が明確になるので Agent の往復が減る。Skill を直接指定できる。
短所：コマンド名を覚える必要がある。導入時期：Phase 4 開始時。

9.3最終形：Slack Workflow Builder + Block Kit ボタン UI

最終的に、よく使うパターンは Workflow Builder でフォーム化する。 営業はチャンネルの「ショートカット」から起動し、Block Kit のフォームに入力するだけで、Agent が裏で Skill を呼び出す。

  • 例：「提案書作成 Workflow」— クライアント名 / 商材 / 過去案件参照を select で選択 → Agent が蒸す Skill を実行
  • 例：「動画分析 Workflow」— URL 貼り付け / 分析観点をチェックボックス → Agent が焼く Skill を実行
  • 長所：誰でも迷わず使える。誤入力が減る。
  • 導入時期：Phase 5（全社展開）と同時。

11.プライバシー & セキュリティ

アーキテクチャ上の Guardrails 層。v2.2 の方針を継承しつつ、 v3.0 では薬機法／景表法／ステマ規制チェッカーと KOL 二次利用権の期限管理を Phase 4 以降の Skill 関連で明文化する。

11.1会社方針

  • 営業活動由来の社内データは 社内利用に限定。Anthropic / Google 等への学習提供は行わない。
  • 外部 LLM API 利用時は No Train 契約（Enterprise tier）。
  • 個人特定情報（PII）は出力前に Guardrails でマスキング。
  • 監査ログは 1 年保存、四半期ごとに監査役レビュー。

11.2監査・フィルタ・opt-out

  • 全 Tool 呼び出しと Agent 出力をログ化（pgvector とは別 DB）。
  • 営業個人で「Mail 共有プールから外す」「Slack DM を Agent に渡さない」の opt-out が可能。
  • 出力に Tier 0-3 ラベル必須。Tier 3（外部リサーチ）の比率が一定以上の場合は WARNING。

11.3薬機法／景表法／ステマ規制対応のチェッカー（Phase 4 以降）

v3.0 で新規追加。提案コンテンツ生成（蒸す Skill）・動画ナレッジ分析（焼く Skill）の出力に対し、コンプライアンスチェッカーを Guardrails 層に組み込む。
対象法令チェック観点実装
薬機法化粧品・健康食品の効果効能の過剰表現NG ワード辞書 + LLM ダブルチェック
景表法「No.1」「最大級」など優良誤認・有利誤認の表現NG パターン正規表現 + 根拠提示要求
ステマ規制PR 表記の欠如、KOL 投稿の関係性開示「#PR」「広告」明記の自動挿入

11.4KOL 二次利用権・肖像権の期限管理

動画ナレッジ分析（焼く Skill）で使用する KOL（インフルエンサー）動画には、 二次利用権・肖像権の 契約期限 がある。期限切れデータが提案書に混入しないよう、 pgvector のメタデータに license_expires_at を持たせ、Agent 検索時に自動フィルタする。
