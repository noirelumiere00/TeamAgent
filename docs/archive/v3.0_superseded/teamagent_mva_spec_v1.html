1.本書について

1.1目的：3 ヶ月で「動く Agent」を立ち上げる最短ルート

本書は、TeamAgent v3.0 ドキュメント群（構想・概要書／要件定義書／アーキテクチャ図書／実装計画書）の 最も実装に近い別冊 として、MVA（Minimum Viable Agent）の構築を サーバー到着 Day 0 から 3 ヶ月で営業 5 名がベータ利用できる検索 Agent へ着地させるための実装手順書である。

本書のゴールは抽象論ではなく以下の 1 点に集約される。

本書の存在意義
「サーバーが届いた日に terraform apply を叩いて、翌日からコードを書き始められる」状態を、実装担当本人（FDE）と関係者の間で揃えるための作業計画。Day 0 のチェックリスト・OAuth 登録手順・docker-compose・SQL DDL・疑似コードまで網羅する。

1.2構想・概要書 v3.0 第 6 章との関係

構想・概要書 v3.0 の第 6 章「MVA 戦略」は、MVA の 哲学・全体像・3 Phase の位置づけ を語る章である。本書はその第 6 章の 詳細実装版 にあたる。

観点構想・概要書 v3.0 第 6 章本書（MVA 仕様書 v1）
抽象度経営層・営業向けの構想説明実装担当向けの作業指示書

2.MVA の哲学

2.1「資料自動化先行」から「データ層先行」への転換理由

v2.2 までの TeamAgent は「5 機能（動画分析／メール／提案資料／Slack 自動投稿／進捗まとめ）を順番に作る」設計だった。これを v3.0 で根本から見直し、「データ層（Mail / Drive / Slack）を先に固める」+「Skill Registry に初期 Skill セットを順次登録する」方針へ転換した。理由は以下のとおり。

論点資料自動化先行（旧 v2.2）データ層先行（MVA / 新 v3.0）
初動 3 ヶ月の成果物 提案資料 1 機能の β 版。データソース未連携のため出力品質が低い Mail/Drive/Slack 横断検索が動く Agent。全 Skill の基盤になる
2 度手間リスク 各機能ごとに個別にデータ取り込みを書き、後で共通化し直し 共通データ層 1 つに集約。各 Skill は SQL or Tool 呼び出しのみ
営業へのデモ素材 「ちょっとした自動生成」止まり。差別化が弱い 「過去類似案件・KOL・関連メール」を即提示。本質的価値が伝わる
Skill 拡張の伸び 5 機能 = 5 個の独立アプリ。横展開が線形 1 つ基盤 + Skill Registry に N 個 Skill を登録。Skill 追加が指数関数的に容易
監査・統制 各機能でログ・権限を個別実装 共通 Permissions / 監査ログを 1 箇所で完結

2.2ファウンデーション・ファーストの利点

個別 Skill から入ると「動くデモは早いが、後で必ず作り直し」になる根拠は以下の 4 点。

  1. データ重複の罠：機能 A が Mail を取り込み、機能 B も別形式で Mail を取り込み、機能 C が両方を参照しようとした瞬間に整合性破綻。pgvector に最初から集約しておけば全 Skill が同じビューを共有できる。
  2. Embedding コストの罠：各機能で別々の Embedding モデルを使うと、再 Embedding コストが Skill 数 × データ量で膨らむ。共通 Embedding なら 1 回投入で N Skill が使える。
  3. 権限・監査の罠：「誰が誰のデータを参照できるか」を Skill ごとに実装すると、4 つ目以降は確実に穴が空く。共通 Permissions 層を最初に置くことで一元統制できる。
  4. Agent の本質に合致：Claude Agent SDK は「自律的に Tool を呼び出す」モデル。Tool が呼ぶ先（データ層）が薄っぺらいと、Skill いくつ並べても賢くならない。

2.33 ヶ月で営業に見える成果物を担保する戦略（P3 で超軽量検索 Skill）

「データ層だけ作ってる 3 ヶ月」では営業に説明できない。そこで P3（Sprint 6 / M3）で超軽量の検索 Skill を必ず載せる ことで、3 ヶ月目には「Slack で @TeamAgent ACME 案件は？ と聞くと過去類似案件・KOL・メール・Drive リンクが返る」状態を作る。

営業に見せる時のデモシナリオ
営業 A：「@TeamAgent ◯◯食品の前回提案、誰が KOL だった？」
TeamAgent：「2024 年 11 月の提案では @yamada-san がリードしていました。当時 KOL は田中さん（インフルエンサー A）、起用結果は CV 1.4% 達成。関連資料：[Drive リンク] / [Slack スレッド] / [メール 3 通]」
→ これが M3（3 ヶ月目）の完成状態。Skill としては検索 1 個のみだが、データ層が揃っているため営業価値は十分。

# app/agent/skill_registry.py（疑似コード・MVA 最小実装） from typing import Callable, Type from pydantic import BaseModel class SkillSpec: """Skill Registry に登録される 1 個の Skill のメタデータ + 実体""" name: str # 例 "search", "draft_mail" description: str # Orchestrator が選択判断に使う説明 input_model: Type[BaseModel] # 入力 schema（Pydantic） output_model: Type[BaseModel] # 出力 schema（Pydantic） handler: Callable # 実行関数 required_scope: str # "mine" | "team" | "all" 等の最低必要権限 audit_tag: str # 監査ログに刻む識別子 class SkillRegistry: def __init__(self): self._skills: dict[str, SkillSpec] = {} def register(self, spec: SkillSpec) -> None: """Skill を登録。MVA では Phase 3 の search Skill 1 個のみ。""" assert spec.name not in self._skills, f"duplicate skill: {spec.name}" self._skills[spec.name] = spec def list(self) -> list[SkillSpec]: return list(self._skills.values()) def get(self, name: str) -> SkillSpec: return self._skills[name] def invoke(self, name: str, payload: dict, scope_filter: dict) -> BaseModel: spec = self.get(name) inp = spec.input_model(**payload) out = spec.handler(inp, scope_filter=scope_filter) return out # spec.output_model 型

6.3Tool スキーマ例（search_pgvector）

# app/agent/tools/search.py from pydantic import BaseModel, Field from typing import Literal, Optional class SearchInput(BaseModel): query: str = Field(description="自然言語の検索クエリ") scope: Literal["mine", "team", "all"] = Field(default="mine", description="検索範囲。mine=本人のデータのみ、team=同チーム、all=全社") source: Optional[Literal["gmail","drive","slack"]] = Field(default=None, description="特定ソースに絞る場合に指定。未指定なら横断") limit: int = Field(default=10, ge=1, le=50, description="返却件数の上限") class SearchResultItem(BaseModel): source: str title: str snippet: str # ヒット箇所周辺の本文 200 字 score: float # コサイン類似度 document_id: str link: str # Slack permalink / Drive URL / Gmail URL class SearchOutput(BaseModel): items: list[SearchResultItem] total_found: int def search_pgvector(inp: SearchInput, scope_filter: dict) -> SearchOutput: """ 自然言語クエリを Embedding 化し、pgvector で類似検索する。 scope_filter は呼び出し時に Permissions 層が注入する制約条件。 """ embedding = embedder.embed(inp.query) sql = """ SELECT d.id, d.source, d.title, d.metadata, e.chunk_text, 1 - (e.embedding <=> %s::vector) AS score FROM embeddings e JOIN documents d ON d.id = e.document_id WHERE (%s::text IS NULL OR d.source = %s) AND d.owner_email = ANY(%s::text[]) -- ← Permissions 注入 ORDER BY e.embedding <=> %s::vector LIMIT %s """ rows = db.fetch_all(sql, [embedding, inp.source, inp.source, scope_filter["owner_emails"], embedding, inp.limit]) return SearchOutput( items=[SearchResultItem(...) for r in rows], total_found=len(rows), )

  • フィードバック 20 件以上収集、Phase 4 で優先実装すべき Skill のヒントが得られている
  • 運用ダッシュボードで「クエリ数 / 平均応答時間 / Tool 呼び回数 / エラー率」が見える
M3 達成 = MVA 完了
ここまでくれば「動く社内 AI Agent 基盤」が稼働中。Phase 4 では同じ基盤の Skill Registry に 初期 Skill セット（残り 4 個）を register() していくフェーズ に入る。データ層・Agent ループ・Permissions・監査ログ・Skill Registry の最小実装は MVA で完成済み。
