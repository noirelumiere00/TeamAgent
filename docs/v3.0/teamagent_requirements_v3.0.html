1.本書について

1.1目的

本書は TeamAgent v3.0 の要件を、Claude Agent SDK を前提とした AI Agent 基盤要件として定義する。v2.2 までは「5機能の機能要件」を中心に組み立てていたが、v3.0 では TeamAgent を 拡張可能な Skill レジストリを持つ Agent プラットフォームとして再定義する。MVA（Minimum Viable Agent）P1〜P3 を最重要要件として頂上に据え、その下に Skill / Tool / Memory / Guardrails の4軸で要件を分類する。MVA で最初に動く Skill は 検索 Skill（1個）、Phase 4 で追加する初期 Skill セットは 5 個（Slack 自動サジェスト / Mail ワークフロー / 提案コンテンツ生成 / 動画ナレッジ分析 / 進捗サマリー）。Skill レジストリへの追加は無限であり、core 改修なしでプラグインとして拡張できることを要件化する。

v3.0 は v2.2 から大幅再構成。OpenClaw 関連要件を削除し、Claude Agent SDK 単独構成に統一。MVA P1〜P3 を要件番号付きで明示し、Phase 4 以降の Skill 要件は v2.2 の FR を継承。さらに HTML→PDF/PPTX 変換、ボタン UI、Workflow Builder などの UX 要件を新規追加。

2.システム概要

2.1アーキテクチャ俯瞰（6層）

v3.0 では TeamAgent を 6 層構成として整理する。最上位に Claude Agent SDK を据え、データ層 / Skills / Tools / Memory / Guardrails の5層がこれを支える。

① Claude Agent SDK
Agent ループ本体。Tool 呼び出し、Memory 参照、Skill 起動を司る
② データ層
Mail / Slack / Drive / 動画 / Gemini DR / X。pgvector に Embedding 投入
③ Skills（レジストリ）
プラグイン構造の Skill レジストリ。初期 Skill セット5個（SlackSuggest / MailWF / Proposal / VideoKB / Summary）を Phase 4 で登録。以降は無限に追加可能
④ Tools
pgvector 検索 / Gmail / Drive / Web 検索 / 動画分析 / 外部 MCP
⑤ Memory / Session
スレッド単位の短期、案件単位の中期、ユーザー単位の長期メモリ
⑥ Guardrails
アクセス制御、PII、薬機法・景表法・ステマ、引用必須化

3.MVA 要件（最重要章）MR-Px-n

MVA（Minimum Viable Agent）とは、TeamAgent の最小実用形態。Claude Agent SDK + pgvector + Slack Bolt + Gmail/Drive OAuth で構成され、営業16名が @TeamAgent [自然言語クエリ] 一行で過去ナレッジを横断検索できる状態。Sprint S1〜S6 の6スプリント（12週間）で構築する。MVA 要件はすべて MUST。

3.1MVA P1：データ層連携 MR-P1

S1〜S3 で実装。Gmail / Slack / Drive を pgvector に取り込むパイプライン。

ID優先度要件
MR-P1-1MUSTGmail OAuth 2.0 でドメインワイド委任を取得し、営業16名分の Gmail を取り込めること。スコープは gmail.readonly + gmail.metadata。
MR-P1-2MUSTDrive OAuth 2.0 で指定フォルダ配下の全ファイルを取り込めること。対応形式は PDF / Word（.docx）/ PowerPoint（.pptx）/ Excel（.xlsx）/ Markdown / Google Docs / Google Slides。
MR-P1-3MUSTSlack Bolt + OAuth で指定チャンネルのメッセージとスレッドを取り込めること。チャンネルは事前設定（営業共有 / 案件別 / DM は除外）。
MR-P1-4MUSTpgvector への Embedding 投入パイプラインを実装すること。Embedding モデルは Bedrock Titan v2（1024 次元）を初期採用、Voyage AI も A/B 比較できること。
MR-P1-5MUST増分同期：新着メール / Slack 投稿 / Drive ファイル変更を 5 分以内に検知し、追加 Embedding を投入すること。
MR-P1-6MUSTメタデータ層：取り込んだ各レコードに「誰の / いつの / どの案件 / どのチャンネル / Tier ラベル」のタグを付与すること。タグスキーマは DR-Tag-1 で定義。
MR-P1-7MUST取り込み監視：CloudWatch ダッシュボードで取り込み件数 / 失敗件数 / Embedding 投入レイテンシをリアルタイム表示すること。
MR-P1-8MUST取り込み失敗時アラート：OAuth トークン失効、API レート制限、Embedding 失敗を Slack #ops チャンネルにアラート通知すること。
MR-P1-9MUST全件初期同期：5万〜10万件相当の Mail / Slack / Drive を、2〜3日かけてバッチで初期取り込みできること。途中失敗時の再開機能を持つこと。
MR-P1-10MUSTシークレット管理：OAuth トークン、API キー、DB 接続情報はすべて AWS Secrets Manager に格納し、コード / 環境変数に平文で含まないこと。
MR-P1-11SHOULD添付ファイル（PDF / Word / PPT）も本文と同列に Embedding 化し、検索可能にすること（OCR 含む）。

3.2MVA P2：Agent SDK 基本ループ MR-P2

S3〜S4 で実装。Claude Agent SDK を組み込み、Slack から自然言語クエリを受けて Tool 呼び出しまでを動かす。

ID優先度要件
MR-P2-1MUSTClaude Agent SDK（Python or TypeScript）を組み込み、Anthropic Bedrock 経由で claude-3.5-sonnet 以上を呼び出せること。
MR-P2-2MUSTpgvector 検索 Tool を SDK の Tool として定義し、自然言語クエリ → Embedding → top-k 検索 → 構造化結果返却までを動かすこと。
MR-P2-3MUSTSlack Bolt の受け口：@TeamAgent メンションを受信し、Agent ループに渡せること。メンション以外のメッセージは応答しないこと。
MR-P2-4MUSTAgent ループ：ユーザークエリ → 思考 → Tool 呼び出し → 結果評価 → 最終応答までを SDK 標準の Tool Use ループで実行すること。
MR-P2-5MUSTMemory / Session 最小実装：スレッド ID 単位で会話履歴を保持し、同一スレッド内では文脈を引き継ぐこと。
MR-P2-6MUST監査ログ：すべての Agent 呼び出しを「ユーザー / クエリ / 使用 Tool / 応答 / トークン数」の形で CloudWatch Logs + S3 にアーカイブすること。
MR-P2-7MUST最小 Permissions：Agent は読み取り専用 Tool のみ呼び出せること（書き込みは P3 以降で段階的に解禁）。
MR-P2-8MUST応答 SLO：シンプルな検索クエリに対して 3 秒以内に最初のトークンを返すこと（NFR-1 と整合）。

3.3MVA P3：超軽量・検索 Skill MR-P3

S5〜S6 で実装。営業16名が実際に「検索エージェント」として日常使用できる状態。MVA 完成 = この Skill が動くこと。

ID優先度要件
MR-P3-1MUST@TeamAgent [自然言語クエリ] で pgvector を横断検索できること。例：「サントリー案件で過去に出した提案ある？」
MR-P3-2MUST引用付きサマリー返却：検索結果上位 5 件の要約と、引用元（Mail / Slack / Drive のリンクとタイムスタンプ）を併記すること。
MR-P3-3MUST関連ファイル / メールリンク提示：Drive ファイルは https://drive.google.com/...、Mail は https://mail.google.com/.../{messageId}、Slack は permalink を返却すること。
MR-P3-4MUSTスレッド内文脈保持：同一 Slack スレッド内で「もう少し詳しく」「別の案件で」などの追問に対し、前回検索条件を継承すること。
MR-P3-5MUST引用率 90% 以上 / 捏造率 0%：Skill が返す事実主張は全て引用ラベル（DR-Tier-x）付きであること。検証用 KPI ダッシュボードを設置。
MR-P3-6SHOULD検索結果のフィードバック：「役立った/役立たなかった」ボタンで品質ログを蓄積すること。

4.機能要件（FR）— Skill レジストリと初期 Skill セットFR-XXX-n

TeamAgent は 拡張可能な Skill レジストリを持つ Agent プラットフォームである。Skill は 入出力 schema + 必要 Tools + Guardrails 仕様を持つ再利用可能な処理単位として定義され、core 改修なしでプラグインとして無限に追加できる。本章 4.0 で Skill レジストリ自体の機能要件を定義し、4.1〜4.5 で Phase 4 で登録する初期 Skill セット5個（v2.2 の FR-1〜5 を継承して Skill 化したもの）を定義する。

4.0Skill レジストリ要件FR-Skill-Reg

Skill レジストリは Skill の登録／検索／実行を司る基盤コンポーネント。MVA P2（S4）で最低限の登録／実行 API を実装し、Phase 4 開始時点（S7 直前）にバージョン管理・依存解決・権限管理を含む完全形を満たすこと。

ID優先度要件
FR-Skill-Reg-01MUSTSkill の登録／削除／更新 API：Skill メタデータ（name / version / description / owner）と実装エントリポイントを登録できること。CLI または管理 UI から実行可能
FR-Skill-Reg-02MUST入出力契約の schema-based 定義：各 Skill は JSON Schema で入力／出力／エラーを宣言し、レジストリ側で実行時バリデーションを行うこと
FR-Skill-Reg-03MUSTSkill 間の依存解決：Skill が他 Skill または Tool を必要とする場合、レジストリが依存を解決し、未充足時は起動を拒否すること
FR-Skill-Reg-04MUSTSkill のバージョン管理：semver（major.minor.patch）で複数バージョンを並列に登録できること。デフォルトバージョンの切替・ロールバックが可能
FR-Skill-Reg-05MUSTSkill のアクセス権限管理：Skill 単位で実行可能なユーザー／ロール／案件を RBAC で制御できること（GR-1 と整合）
FR-Skill-Reg-06MUSTSkill 利用ログの Audit：全 Skill 呼び出し（入力／出力／所要時間／呼び出し元）を CloudWatch Logs + S3 に記録し、Audit 検索可能とすること（GR-3 / NFR-6 と整合）
FR-Skill-Reg-07MUSTプラグイン構造：新規 Skill は core 改修なしで追加可能であること。Skill コードは独立リポジトリ／ディレクトリで管理し、registry.register(skill) のみでホットロード／再起動なしで反映できること

4.1FR-VideoKB（初期 Skill 4：動画ナレッジ分析）FR-VideoKB

Phase 4 / Sprint S7〜S8。詳細仕様は別冊『機能④ 確定仕様書 v1』参照。本書では要件番号体系のみ。

ID優先度要件
FR-VideoKB-1MUSTYouTube / TikTok / Instagram の URL を Slack に投げると、動画分析を返すこと（リアクティブ）
FR-VideoKB-2MUST分析アウトプット：トランスクリプト、登場人物、商材、訴求軸、再生数 / engagement の構造化
FR-VideoKB-3MUSTGemini Video Understanding API を使った動画コンテンツの直接解析
FR-VideoKB-4MUST3 プラットフォーム × 2 段階（メタ + 中身）の分析パターンを提供
FR-VideoKB-5SHOULD類似動画のクラスタリング（VSEO Analytics 連携）
FR-VideoKB-6SHOULD競合分析モード：指定アカウントの直近10件の傾向分析
FR-VideoKB-7MAYdpro / Buzzmiru / SHORTBOOSTER 等の外部 SaaS の MCP 統合（API 確認後）
FR-VideoKB-8MUST分析結果は pgvector に保存し、後続 Skill（FR-Proposal）からも参照可能であること

4.2FR-MailWF（初期 Skill 2：Mail ワークフロー）FR-MailWF

Phase 4 / Sprint S9〜S10。朝サマリー / 返信深掘り / 進捗サマリー（初期 Skill 5 を内包）。

ID優先度要件
FR-MailWF-1MUST朝サマリー：毎朝 8:30 に当日対応すべきメールを優先度別に Slack DM 配信
FR-MailWF-2MUST優先度判定：差出人 / 件名 / 本文 + 過去文脈から「即返信 / 今日中 / 数日OK / 様子見」の4段階分類
FR-MailWF-3MUST返信深掘り：個別メール選択でスレッド全体・関連 Drive / Slack を統合分析
FR-MailWF-4MUST返信ドラフト生成：相手の文体・過去やり取りに沿った返信案を Gmail 下書き保存
FR-MailWF-5MUST下書き保存スコープ：gmail.compose。送信は人間が必ず確認
FR-MailWF-6MUST進捗サマリー（初期 Skill 5 / FR-Summary を内包）：マネージャー向け週次ダイジェスト（各営業の動き / 案件進捗 / リスク）
FR-MailWF-7MUST本人 / 他人メールの区別：本人 Mail = 要件定義用、他人 Mail = 過去事例ナレッジ DB
FR-MailWF-8MUSTopt-out：個別営業がメール共有を opt-out できる機能
FR-MailWF-9SHOULD会議招待 / カレンダー連携で「次のミーティング前にこれ見ろ」リマインダ
FR-MailWF-10SHOULD添付ファイルの自動要約（PDF / Word / Excel）
FR-MailWF-11MAYメール文面の語気・トーン分析（怒り / 不安 / 期待）

13.2用語集

Agent
LLM が Tool を動的に選び、目的達成まで自律的に動作するシステム。Claude Agent SDK は Anthropic 公式の Agent 実行フレームワーク。
Skill
Agent SDK 上で実行可能な再利用可能な処理単位。入出力 schema + 必要 Tools + Guardrails 仕様を持ち、Skill レジストリにプラグインとして登録される。core 改修なしで無限に追加可能。本書 v3.0 改訂以降では、Phase 4 で登録する初期 Skill セット5個（FR-SlackSuggest / FR-MailWF / FR-Proposal / FR-VideoKB / FR-Summary）を出発点として位置付ける。
Skill レジストリ
Skill の登録／検索／実行を司る基盤コンポーネント。バージョン管理（semver）、依存解決、RBAC、Audit ログ、プラグイン構造のホットロードを備える（FR-Skill-Reg-01〜07）。
初期 Skill セット
Phase 4 で Skill レジストリに最初に登録する5個の Skill（FR-SlackSuggest / FR-MailWF / FR-Proposal / FR-VideoKB / FR-Summary）。v2.2 までの「5機能」を Skill 化したもので、これ以降の Skill は無限に追加可能。
Tool
Agent が呼び出す関数の単位。pgvector 検索、Gmail 操作、Drive 書き込み、HTML→PDF 変換などが Tool。
MCP（Model Context Protocol）
Anthropic が定義した「Tool / リソース / プロンプトを外部サーバから LLM に提供する」標準プロトコル。外部 SaaS との接続に使用。
Memory
Agent が会話 / 案件 / ユーザー単位で保持する文脈情報。短期（Redis）/ 中期（pgvector）/ 長期（PostgreSQL）の3層。
Guardrails
Agent の挙動を制約するセキュリティ・コンプラ層。アクセス制御、PII 保護、薬機法・景表法・ステマ規制、引用必須化。
pgvector
PostgreSQL に Embedding ベクトル検索機能を追加する拡張。本システムの中心ストレージ。
Tier 0-3
データ信頼度ラベル。Tier 0 = 一次データ（動画/案件メタ）、Tier 1 = 本人由来、Tier 2 = 社内二次、Tier 3 = 外部リサーチ。
MVA（Minimum Viable Agent）
TeamAgent の最小実用形態。Claude SDK + pgvector + Slack + Gmail/Drive で「@TeamAgent で過去ナレッジ横断検索ができる」最小構成。S1〜S6 で構築。
DoD（Definition of Done）
各 Sprint の完了基準。本書 第12章で Sprint 毎に列挙。
引用率 / 捏造率
生成文の主張のうち Tier 引用ラベル付きの比率を引用率、出典のない事実主張を捏造として捏造率とする。KPI = 引用率 90% 以上 / 捏造率 0%。
意図ルーター
FR-SlackSuggest（軽い検索 Skill）と FR-Proposal（重い生成 Skill）を confidence ≥ 0.7 で振り分けるロジック。
ephemeral
Slack chat.postEphemeral による特定ユーザー向け一時メッセージ。FR-SlackSuggest の確認応答に使用。
Workflow Builder
Slack 標準のノーコードワークフロー構築機能。ボタン押下で Skill を起動する UI に使用。
Block Kit
Slack のリッチ UI 構築フレームワーク。ボタン、Modal、フォームを Skill から呼び出せる。
Bedrock Titan v2
AWS Bedrock の Embedding モデル（1024 次元、日本語対応）。本システムの初期採用モデル。
