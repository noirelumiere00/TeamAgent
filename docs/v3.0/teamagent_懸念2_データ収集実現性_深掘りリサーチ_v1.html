0.エグゼクティブサマリー

監査役懸念②「データが綺麗に集まるか」を、3観点（文書抽出 / LLM リサーチ品質 / マルチソース統合）に分けて Web リサーチ + 類似事例調査を実施した。結論：3観点とも「楽観論は危険だが、致命的でもない」。確立された手法・他社事例があり、現実的な割り切りと PoC を前提にすれば到達可能。

観点監査役の懸念リサーチ結論現実的な数値
① 文書抽出（PDF/PPTX）提案書から綺麗に抽出できるか「全件綺麗に」は不可能。ハイブリッド3層構成（PyMuPDF振り分け→LlamaParse/Azure DI/python-pptx）で実用化可能。PPTXネイティブグラフは数値が抜ける朗報あり約74%が実用レベル抽出、残26%は割り切り or 人手補完
② Gemini DR 品質・捏造AIがもっともらしい嘘を書くのでは懸念は技術的に正しい。「捏造率0%」は達成不可能。response_schema で構造は100%守れるが中身は守れない。URL機械検証＋件数API直取得＋事実照合の実装レイヤーが必須素朴実装で中身信頼は40-60%、フル実装で85-90%（残りは「要確認」とフラグ化）
③ マルチソース統合データは集まるが繋がらないのでは「集める」は枯れた技術。「案件IDで束ねる」が最大の難所だが、エンティティ解決（案件マスタをハブ）という確立分野。SAPですら専業Reltioを買収するほど＝難所は本物案件ID自動マッチ70-85%、残りは人手補完UI必須
⚠️ 3観点に共通する最重要メッセージ
3つの Agent が独立にリサーチした結果、同じ結論に収束した：
・「設計すればできる」「プロンプトでお願いすればできる」という楽観論は、いずれの観点でも他社事例によって否定される
・しかし「現実的な割り切り（74%・85%・80%という歩留まり）」＋「人手補完の仕組み」＋「機械検証レイヤー」を計画に織り込めば、いずれも到達可能
・監査役の「無料枠で試したら」は、3観点すべてに対する正しい処方箋。PoC で歩留まりを実測し、設計目標を現実値に補正することが本格投資の前提

1.文書抽出（PDF/PPTX）の実現性

対象：Google Drive の過去提案書（PDF/PPTX 約1,500件）→ pgvector 取り込み。監査役懸念リスト上の「高リスク」筆頭。

1.1 リサーチ結論

「全件を綺麗に」は不可能。だが現実的な切り分けをすれば実用レベルの取り込みは十分可能。約74%が実用レベル・残り26%は部分抽出または人手補完が必要、というのが正直な見積もり。

1.2 抽出ツール比較（2026年時点）

ツール複雑レイアウト図表・グラフ内数値日本語推奨用途
PyMuPDF△グラフ内数値✗◯一次振り分け（テキスト/画像PDF判別）
LlamaParse V2◎ 最強表◎ / グラフ△◯テキストPDF・複雑レイアウト本命
Azure AI Document Intelligence◯表◎ / グラフ✗◎ 日本語OCR最高精度スキャンPDF・日本語の最有力

表 1.1どのツールも「画像PDF内のグラフ数値ラベル」は原理的に抽出困難。唯一の例外は PPTX のネイティブチャート（埋め込みExcelからデータを完全取得可能）。

1.3 推奨：ハイブリッド3層構成

[1] 一次振り分け（PyMuPDF） page.get_text() の文字量で「テキストPDF / 画像PDF / 混在」を自動判別 ↓ [2-a] テキストPDF・複雑レイアウト → LlamaParse（基本モード 1〜3クレジット/ページ） [2-b] スキャンPDF（画像PDF） → Azure AI Document Intelligence（日本語OCR最高精度） [2-c] PPTX → python-pptx（テキスト・表・ノート・ネイティブグラフ数値） ↓ [3] S3 中間キャッシュ（.md）に保存 → 信頼度スコア低い文書は人手レビュー → pgvector に UPSERT

1.4 「抜けるもの」と「抜けないもの」の現実的な切り分け

抜ける（実用レベル）部分的にしか抜けないほぼ抜けない（諦める判断）
テキストPDF の本文・見出し多段組みの読み順（崩れる）画像PDF内のグラフの数値ラベル
PPTX のテキストボックス・表・ノート複雑/ネストした比較表（構造が壊れる）スクリーンショットとして貼られた表
PPTX ネイティブチャートの数値スキャンPDF（OCR誤りが残る）手書きの赤入れ・注釈
罫線のある単純な表図解（ポンチ絵）内のテキスト断片グラフから「傾向」を読む

2.Gemini DR 品質・捏造対策の実現性

対象：Gemini Deep Research に「8セクション構造化 JSON」を生成させる。監査役の本質的懸念「AIがもっともらしい嘘を書くのでは」。

2.1 リサーチ結論

「JSON の構造（スキーマ）」は技術で 100% 守れる。しかし「中身の正しさ（URL 実在・件数の正確さ）」はプロンプトでは守れない。監査役の懸念は技術的に正しく、「捏造率 0%」は現行アーキテクチャでは達成不可能。機械検証レイヤーを実装に組み込むことが必須。

2.2 構造保証と中身の正しさは「別レイヤー」

手法何を解決するか効果（2026年実測）
response_schema（Constrained Decoding）JSON の構造崩れスキーマ準拠を100%保証（無効トークンをマスク）
プロンプトで「JSON だけ出せ」（旧世代の手法）失敗率 5〜20%。非推奨
「不明」と言わせるプロンプト自信のない捏造回答有効だがLLM がこの指示を無視する頻度は一定数ある
URL 機械検証（HTTP チェック）存在しない URL の引用「存在しない URL を引用しようとしたケースの100%を阻止」した実例あり
Span-level 事実照合（LLM-as-judge）「URLは実在するが内容が主張を裏付けない」捏造ハイブリッド検証で引用精度92%
件数の API 直取得LLM のカウント能力の脆弱性LLM は約5%が数値すら返さない → API 直取得で捏造の余地を排除

表 2.1Constrained Decoding が保証するのは「構文的に正しい JSON」だけ。「スキーマに合致した有効な JSON が、意味的にナンセンスな値（でっち上げ URL・件数）を含むこと」は防げない。

2.3 類似事例（Web 調査）

事例内容TeamAgent への示唆
「Cited but Not Verified」研究（arXiv 2026）主要 Deep Research のリンク有効率94%以上だが、事実的正確性は39〜77%。ツール呼び出しを増やすと精度がむしろ低下「検索を増やせば正確になる」は誤り。リンクは生きてても内容が主張を裏付けない
DeepResearch Bench（100タスク・22分野）「検索品質が改善しても、どのモデルも正確な出典帰属を保証できない」出典の正確さは構造的に未解決問題
Gemini 自己説明の検証（2026年2月）Gemini Deep Research に自分の仕組みを説明させたら約半分が不正確自社製品についてさえもっともらしい誤りを生成する
Citation-Grounded 機械検証（arXiv 2026）機械検証で引用精度92%。「存在しないファイル・無効行を引用しようとしたケースの100%を阻止」LLM に検証させず決定論的な機械チェックで弾くのが低コストで効果的
LLM のカウント能力研究（arXiv 2026）「件数を数える」タスクは LLM が構造的に苦手。約5%の出力が有効な数値を返さないTikTok ハッシュタグ件数は LLM 生成を禁止し API 直取得すべき

2.4 推奨アプローチ

対策具体策
JSON Schema 遵守Gemini API の response_schema に Pydantic スキーマを渡す。「Deep Research に調査」と「結果を JSON 整形」を2段階に分ける
URL 実在確認LLM 出力後、全 URL に HTTP リクエスト実行。リンク切れは「要確認」に自動置換 or 再生成
件数厳格化件数は LLM に生成させない。TikTok 件数は専用スクレイパー API から機械取得し JSON にプログラム挿入
空欄禁止ルールスキーマ側で各フィールドを「値 or 『要確認』文字列」の union 型にして、空・null を構造的に弾く

2.5 「捏造率0%」KPI の修正提案

⚠️ KPI を「達成不可能な0%」から「機械検出率」へ
「捏造率0%」は現行アーキテクチャ（LLM に事実・URL・件数を生成させる）では達成不可能。フロンティアモデルでさえ事実的正確性39〜77%。
→ 現実的な KPI への置き換え：
・「構造崩れ0%」（Constrained Decoding で達成可能 — これは約束してよい）
・「リンク切れ URL 0%」（HTTP 機械検証で達成可能）
・「件数フィールドは LLM 生成を禁止し API 取得に限定」（捏造の余地をアーキから排除）
・「事実的正確性 ≥ 90%、未検証項目は100%フラグ化」（検証パイプライン込みで現実的）

2.6 現実的な品質見積もり

実装レベル構造中身（事実的正確性）
プロンプトのお願いのみ（素朴な実装）80〜95%40〜60%
+ response_schema 導入ほぼ100%40〜60%（変わらず）
+ URL HTTP 検証 + 件数 API 取得ほぼ100%65〜80%
+ Span-level 事実照合（フル実装）ほぼ100%85〜92%（残りは「要確認」と正しくフラグ化）

表 2.2何もしなければ「8セクション中、中身まで信頼できるのは3〜5セクション程度」。フル実装で初めて「8セクション中7セクション相当が信頼でき、残りは『要確認』と正直にフラグされる」水準に届く。「8セクション全部が無修正でそのまま使える」はどんな実装でも約束できない。

3.マルチソース統合の実現性

対象：6データソースを束ねる共通キー設計・pgvector 組織ナレッジ DB・Gmail/Slack/Drive 横断・スクレイピング系（dpro/Buzzmiru）。監査役の懸念「データは集まるが繋がらないのでは」。

3.1 リサーチ結論

監査役の懸念は正しい。「6ソースを集める」ことは技術的に枯れているが、「案件IDで束ねる」部分は他社事例でも最大の難所。ただし解決不能ではなく、「エンティティ解決（Entity Resolution）」という確立された分野の手法を適用すれば現実的に到達可能。

3.2 3つの要点

要点内容
① 共通キーは「推定して付与する」もの「最初から存在する」ものではない。Salesforce Data Cloud / Glean / Snyk 等、全成功事例が「マッチルール＋人手補完」で golden record を後付け生成
② pgvector は想定規模なら実績十分6レイヤー・数万〜数十万件なら pgvector の得意領域。ただし「案件IDフィルタ後にベクトル検索」は post-filtering の弱点に直撃 → 設計注意
③ スクレイピング系は「落ちる前提」で隔離設計必須業界実測で「クローラの10〜15%が毎週修正を要する」。コア機能の依存先にしてはいけない

3.3 類似事例（Web 調査）

事例共通キー設計TeamAgent への示唆
Snyk（Slack RAG）キーで JOIN せず、Gemini API で各スレッドを「質問＋回答」構造に変換Snyk 総括「本当の工学的成果は API 統合や embedding ではなくデータを丹念に構造化したこと」
Luna（会議メモ+Slack RAG）メッセージに「アンカー（どの案件か）」と「ラベル（種別）」を付与。equivalence check で曖昧参照を解決「動画ID/案件IDで JOIN」は equivalence check 層がないと成立しない
Salesforce Data Cloud「マッチルール＋リコンサイルルール」。高信頼ID優先、なければ exact match、最後に fuzzy match公式が「fuzzy match は慎重に使え」と警告。乱用すると別案件を誤結合
Glean（Gmail/Slack/Drive 横断検索）「アイデンティティグラフ」でエイリアスを連結。コンテンツと権限マップを両方取得Gmail 私信・Slack 閲覧権限を権限マップで弾く設計が必須
SAP × Reltio 買収（2026年3月）SAP が MDM 専業 Reltio を買収。「異種ソースの ID 統合」を内製しきれず業界の巨人ですら難所と認める＝難所が本物である最強の傍証。スコープを絞ることが現実的
スクレイピング型データ取得の破綻パターンScraperAPI「10分キャッシュで100%成功率」、Bright Data「99.99% uptime SLA」「スキーマドリフト」（ジョブ成功なのに中身が劣化）はエラーログに出ず静かに汚染

3.4 推奨：案件マスタをハブにした統合設計

project_master（案件マスタ = golden record） project_id（TeamAgent 内部発番 or 既存CRM IDを正とする） project_name / client_name / status / aliases（別名・略称配列） project_source_link（紐付けテーブル = エンティティ解決の成果物） project_id → project_master source_type（gmail/slack/drive/video/dr/x） source_object_id（Gmail threadId, Slack channel+ts, Drive fileId 等） match_method（deterministic / auto_llm / convention / manual） confidence（0.0〜1.0） verified_by（人手確定者・NULL = 未確定） ★ポイント：「案件ID」を Gmail/Slack/Drive 側に書き込まない（書けない）。 代わりに TeamAgent 側に「紐付け表」を持ち、そこに蓄積する。

3.5 共通キーの自動マッチ歩留まり

ソース付与方法信頼度
動画分析データ層既存の動画IDをそのまま使用。決定論的に解決高
Gemini DRDR 実行時に「どの案件のリサーチか」を入力 → 発番時に確定高
Google Driveフォルダ命名規則（新規のみ）+ 既存は LLM がファイル名・本文から案件名抽出中
Slackチャンネル単位で初回マッピング。DM・横断は LLM でメッセージ単位に推定中〜低
Gmailスレッド単位で LLM が件名・本文から推定（私信判定を先に通す）低〜中

表 3.1命名規則＋LLM 推定で 70〜85%が自動で紐付き、残り15〜30%は人手補完が必要。この人手補完用の簡易 UI（未確定リンク一覧→案件選択→確定）を機能③の一部として必ず作ること。「自動100%」を前提にした計画は破綻する。

3.6 本当の難所（工数とリスクが集中する5箇所）

難所なぜ難しいか
① Gmail/Slack/Drive → 案件ID の紐付け共通キーが存在せず「推定」が必要。自動マッチ70〜85%止まり、誤マッチは「データ汚染」を生む
② データの「構造化」（集めた後）Slack/Gmail は非構造テキスト。「集める」より「Q&A化・サマリ化・案件タグ付け」が本体の工数（Snyk が明言）
③ pgvector の post-filtering 問題「案件IDで絞り込んでからベクトル検索」は pgvector の弱点。母集合が小さいと HNSW の精度が大幅劣化
④ スクレイピング系の継続運用dpro/Buzzmiru は「常に一部壊れている前提」。スキーマドリフトはエラーに出ず静かに汚染
⑤ Gmail 私信の混入業務メールと私信を完全自動分離はできない（Glean も権限マップ＋人手前提）

4.統合：3つの本当の難所と現実解

3観点のリサーチを統合すると、懸念②の「本当の難所」は以下の3つに集約される。いずれも「楽観論は他社事例で否定されるが、現実解は存在する」。

#本当の難所楽観論（危険）現実解歩留まり目標
難所1提案書 PDF/PPTX からの構造化抽出「抽出ツールに通せば綺麗に取れる」ハイブリッド3層構成 + グラフ内数値は割り切り + S3 中間キャッシュで人手レビュー実用レベル抽出 74%
難所2Gemini DR の中身の正確さ（捏造防止）「プロンプトで遵守ルールを書けば守られる」response_schema + URL機械検証 + 件数API直取得 + 事実照合の4層実装。KPI を「捏造率0%」から「機械検出率」へ修正事実的正確性 85-90%（残りは要確認フラグ）
難所36ソースを案件IDで束ねる（共通キー）「設計図で JOIN すれば繋がる」案件マスタをハブに、自動マッチ70-85% + 人手補完UI を最初から作る + pgvector の post-filtering 劣化対策案件ID自動マッチ 70-85%

4.1 3観点に共通する「成功の条件」

3 Agent が独立に到達した同一結論
① 「集める」より「構造化する」が本体 — Snyk「本当の工学的成果はデータを丹念に構造化したこと」
② 「自動100%」を前提にしない — 文書抽出74%、案件マッチ70-85%、DR品質85-90%。残りは必ず人手補完 or 割り切り
③ 機械検証レイヤーを実装に組み込む — プロンプトのお願いでは守られない。URL の HTTP チェック、スキーマバリデータ、スキーマドリフト検知
④ スコープを絞る — 欧州銀行「データソースを絞ることが成功条件」、SAP ですら ID 統合を内製せず買収
⑤ 「取り込めた」と「検索で使える」は別物 — RAG 実装の40-60%が本番未到達。end-to-end の検索精度まで見る

4.2 監査役への回答（懸念②について）

監査役への回答
ご懸念の通り、現状の v2.2 仕様書は「設計はあるが検証がない」状態です。3観点を Web リサーチ + 類似事例調査した結果、3つの本当の難所（文書抽出・DR の捏造防止・共通キー統合）が特定されました。

ただし、これらはいずれも他社で解決事例があり、致命的ではありません。Snyk・Glean・Salesforce・欧州大手銀行など、同種の統合を実現した企業の手法（エンティティ解決・ハイブリッド抽出・機械検証レイヤー）を適用すれば到達可能です。

重要なのは「設計目標（取り込み率80%・引用率90%・捏造率0%）を、PoC で実測した現実値に補正する」こと。特に「捏造率0%」は技術的に達成不可能なため、「機械検出率」という測定可能な KPI に修正すべきです。

PoC で3つの難所の歩留まりを実測 → 設計目標を現実値に補正 → 本格投資の最終判断、という段階を踏むことを推奨します。

1〜2回の試行で「動いた」と判断しない。特に LLM 系（D4〜D7）はプロンプトの言い回しだけで精度が大きくブレるため、各項目20〜50回の反復で「率」として評価する。
