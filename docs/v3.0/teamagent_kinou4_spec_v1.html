1.背景と経緯

本章は、機能④が v2.2 までの「外部サービス連携」設計から「TeamAgent への吸収・自前構築」へ 確定再設計されるに至った経緯を記録する。設計判断の理由を残すことで、実装担当（FDE）が背景を理解したうえで構築に入れるようにする。

1.1当初設計（v2.2まで）

機能④は当初、VSEO Analytics（外部サービス・自社運用）＋ 動画広告プロ（dpro）＋ Buzzmiru の3サービスを横断参照し、動画を6観点（EG情報／メディア・界隈／フック／訴求軸／CTA／表現）で構造化する設計だった。VSEO Analytics は tRPC 連携で動画内容分析を担う中核コンポーネントとして位置づけられていた。

1.2VSEO 継続不能の判断

VSEO Analytics は EC2 上の Web アプリケーションとして運用されていたが、サーバーが貧弱で運用・拡張が困難になり、利用継続ができなくなった。これは 法的な理由ではなく、技術・運用上の理由 である。

ここで判断したのは、VSEO の中核機能である ショート動画 Creative Analyzer は機能④とほぼ同義 であるという点である。であれば、これを「外部サービス連携」として外に置くのではなく、TeamAgent に吸収して自前構築する ほうが構成として素直になる。これは新規機能の追加ではなく、既存資産の統合（吸収）である。

判断の要点
VSEO の停止は機能④の縮小ではない。VSEO Creative Analyzer ＝ 機能④ であり、外部依存を1つ減らして TeamAgent 本体に取り込む統合判断である。

1.3VSEO リポジトリ精査の知見

吸収にあたり、VSEO リポジトリ（github.com/noirelumiere00/vseo-analytics-web）を精査した。判明した構成は以下のとおり。

項目VSEO の実装
分析エンジンAmazon Bedrock 経由の Claude Haiku 4.5
LLM への入力動画 mp4 を直接 LLM に渡してはいない。フレーム抽出 ＋ 音声文字起こし ＋ サムネイル ＋ スクレイピングしたメタデータ（再生数・いいね・コメント・シェア・保存）を Claude に渡す構成
スクレイピングPuppeteer（stealth プラグイン）＋ Apify ＋ YouTube Data API v3 を併用

つまり VSEO は、Claude が動画ファイルをネイティブに扱えないことを 「ffmpeg でフレーム抽出 → Whisper 等で文字起こし → 静止画とテキストにして Claude に渡す」という前処理パイプラインで回避 していた。この前処理が運用負荷の一因でもあった。

1.4再設計の核心：Gemini 採用

再設計の核心は、動画コンテンツ分析エンジンを Gemini 2.5 Flash に変更する ことである。

理由内容
Claude の制約Claude は動画ファイル（mp4）をネイティブに受け取れない（静止画と PDF のみ）。VSEO の前処理パイプラインはこの制約の回避策だった。
Gemini の能力Gemini 2.5 系は 動画をネイティブ入力でき、内部でフレームサンプリング＋音声処理を行う。
得られる効果「ffmpeg でフレーム抽出 → Whisper で文字起こし」という前処理ステップが 丸ごと不要 になる。VSEO がやっていた面倒な前処理を Gemini が肩代わりする。
再設計の一行サマリー
VSEO を「連携」から「吸収」へ。そして動画コンテンツ分析エンジンを Claude（前処理必須）から Gemini 2.5 Flash（動画ネイティブ・前処理不要） へ。これにより構成がシンプルになり、運用負荷が下がる。

2.22段階構成（④-A / ④-B）

機能④は、リスク・工数の性質が異なる2つの機能を分離し、2段階構成 で実装する。

段階名称内容着手条件
機能④-A Phase 1 / MVP URL → 動画取得 → Gemini で分析 → Slack 返却 ＋ pgvector 格納。確実に作れる範囲。 Phase 1 で着手
機能④-B Phase 2 / 露出度分析 該当ハッシュタグをスクレイピングし、実際のスマホ UI を模したモック画面で自社動画の立ち位置を可視化する機能。 データ収集 PoC（Sprint 0）でハッシュタグスクレイピングの実現性が確認できてから着手

④-B は ④-A とリスク・工数の性質が異なるため分離する。フォールバック方針は Apify 経由 → それも不可なら手動補完。継続的なハッシュタグスクレイピングは、監査役懸念②で 🔴 高リスクと判定された Instagram スクレイピングそのもの である点を明記しておく。

🔴 ④-B の PoC ゲート
④-B は「データ収集 PoC（Sprint 0）」でハッシュタグスクレイピングの実現性が確認できるまで着手しない。PoC ゲートを通過できない場合、④-B はスコープから外す、または手動補完運用に切り替える判断を行う。

3.分析アウトプット仕様（MVP ライン）

3.1Gemini 構造化アウトプット

機能④-A（MVP ライン）では、Gemini 2.5 Flash が動画から以下の構造化アウトプットを生成 する。

#アウトプット内容
①タイムライン分解フック / 展開 / 本編 / CTA の4区分で時間軸を分解する
②シーンスコア各シーンの評価
③構成サマリー動画全体の要約
④テロップ文字起こし画面内に焼き込まれた文字を Gemini のビジョンで抽出する
⑤セリフ・ナレーション文字起こし音声をテキスト化する
⑥総合スコア全体評価

表 3.1Gemini 2.5 Flash が動画から生成する6つの構造化アウトプット（MVP ライン）。

3.2pgvector 化の方針

上記アウトプットは すべて pgvector 化する。理由は検索性にある。

アウトプットの性質検索性
ビジョン分析結果（構造・スコア）検索しづらい
テキスト化されたテロップ・セリフ・構成サマリー検索性が高い。 機能⑥（社内 AI コーチ）からの自然言語検索の主役になる

テキスト化された要素が検索の主役となるため、構造・スコアも含めて全アウトプットを pgvector に格納し、機能①③⑤⑥から横断参照できるようにする。

3.3「良いクリエイティブ」の判定基準

確定した判定基準
「良いクリエイティブ」の判定は スコアではなく、媒体エンゲージメント（実際に伸びた動画） で行う。スコアは構造分析の補助であり、良し悪しの最終判断は媒体パフォーマンスに置く。

Gemini が出す総合スコア・シーンスコアは、動画の構造を理解・分類するための補助指標である。「この動画は良いのか」という最終判断は、再生数・いいね・コメント・シェア・保存数といった 実際の媒体パフォーマンス に基づいて下す。スコアが高くても伸びていない動画は「良いクリエイティブ」とは扱わない。

4.アーキテクチャ（ハイブリッド構成）

機能④は マルチベンダーのハイブリッド構成 になる。Gemini と Claude が混在するため、その線引きを明示する。

4.1ベンダー線引き

ベンダー線引き（最重要）
機能④の動画コンテンツ分析だけ Gemini、TeamAgent の他のすべては Claude（Claude Agent SDK + Bedrock Claude）。

Gemini の利用範囲を「機能④の動画コンテンツ分析」という1点に閉じ込めることで、マルチベンダー化の影響範囲を管理する。サブエージェントのオーケストレーション、Slack 対話、統合・最終評価のロジックはすべて Claude 側が担う。

4.2処理コンポーネント

工程担当内容
動画本体の取得yt-dlpURL から mp4 をダウンロードする
動画コンテンツ分析Gemini 2.5 Flashダウンロードした mp4 を Gemini File API にアップロード → Gemini 2.5 Flash で分析（タイムライン分解・シーンスコア・構成サマリー・テロップ＆セリフ文字起こし・総合スコア）
媒体パフォーマンス取得Apify / スクレイピングハッシュタグ・エンゲージメント指標（再生数・いいね・コメント・シェア・保存数）を取得する。Apify は他 PJ に相乗りで利用可能
統合Claude SDKGemini の構造化出力 ＋ 媒体エンゲージメント → 最終評価（エンゲージメント基準）
格納pgvector（RDS）機能①③⑤⑥から参照可能
出力Slack / pgvector(1) Slack に分析結果を返す、(2) pgvector に格納 → 機能⑥（社内 AI コーチ）に聞いたとき答えが返ってくる

表 4.2機能④の処理コンポーネントとベンダー分担。動画コンテンツ分析のみ Gemini、それ以外は Claude / 既存基盤。

4.3マルチベンダーのコストと管理

ベンダー混在のコストとして、API キー・課金・SDK が2系統になる。ただし、これは完全な異物の追加ではない。

  • Pattern Bravo+ で既に Vertex AI 経路を想定済みのため、Gemini / GCP は完全な異物ではない。
  • 影響範囲を 「機能④の動画コンテンツ分析」に閉じ込める ことで管理する（4.1 のベンダー線引き）。

4.4モデル名注記

モデル名注記
Gemini 2.5 Flash を採用する。 動画ネイティブ対応は Gemini 2.5 系の確実な能力である。正式なモデル名・後継バージョンは実装時に FDE が最新を確認すること。分解精度が不足する場合は Gemini 2.5 Pro へのフォールバックを用意する。

5.動作フロー（機能④-A）

機能④-A（MVP）の動作フローを示す。営業が Slack に動画 URL を投げると、プラットフォーム判定 → 動画取得とエンゲージメント取得の並行処理 → Gemini 分析 → Claude 統合 → pgvector 格納 → Slack 返却、という流れで進む。

営業：@TeamAgent これ分析して https://... （YTS / IG / TT のいずれか） ↓ Slack Bolt [Lambda: Video Analyzer サブエージェント起動] ↓ URL からプラットフォーム判定（YTS / IG / TT） ├─ yt-dlp で動画本体をダウンロード → Gemini File API へアップロード └─ （並行）Apify / スクレイピングでハッシュタグ・エンゲージメント取得 ↓ Gemini 2.5 Flash で動画コンテンツ分析 （タイムライン分解 ＋ シーンスコア ＋ 構成サマリー ＋ テロップ&セリフ文字起こし ＋ 総合スコア） ↓ Claude SDK が Gemini 出力 ＋ 媒体エンゲージメントを統合 → 最終評価（EG基準） ↓ pgvector に動画ナレッジを格納（機能①③⑤⑥で参照可） ↓ Slack に構造化結果を投稿（所要 1〜3 分）

図 5.1機能④-A の動作フロー。動画取得（yt-dlp → Gemini File API）と媒体エンゲージメント取得（Apify / スクレイピング）を並行実行し、Claude が統合する。所要 1〜3 分。

6.データソース層の再定義

VSEO の吸収にともない、overview_v2.2.html が定義していた「動画分析データ層」の構成を再定義する。

旧（v2.2まで） VSEO Analytics（外部サービス）＋ dpro ＋ Buzzmiru
