0.エグゼクティブサマリー

監査役より「予算は承認するが、具体的な実行リスクを精査せよ」との指示を受け、4つの懸念を並列で精査した。結論：4懸念とも妥当な指摘であり、特に②データ実現性と③BU意向差は、本格実装前に対処すべき実リスク。共通する答えは「無料枠 PoC で未検証部分を実証してから本格投資する」である。

懸念監査役の問い精査結論リスク度
① スコープ営業活動の何をどこまでやる？「提案準備〜下書き作成」のみ。顧客接点（訪問・商談・交渉・送信）は全て人。「全部 AI」ではない。v2.3 でスコープ境界線の明文化を推奨低
② データ実現性データは綺麗に・欲しい形で集まる？自社系（VSEO・本人Gmail）は確実。Drive PDF抽出 / Gemini DR の品質 / 6ソース共通キー の3点は「仕様にあるが未検証」。数値はすべて設計目標で実測なし高
③ BU意向差BU1/BU2で意向が違わないか？違う。直クラ営業と代理店向き合いは提案先・サイクル・KPI が根本的に異なる。v2.2 は「営業16名」と一括りにしており BU 分岐がない。要設計対応中〜高
④ AWS無料枠無料枠で試せる？有料化は不可避？新規アカウントの $200 クレジットで機能④単機能 PoC は実質無料。ただし完全 $0 は不可能（API 実費は別）。本番（機能③・16名・Multi-AZ）は有料確定低
⚠️ 監査役への最重要メッセージ
現状の v2.2 ドキュメント群は「設計」としては十分に詳細だが、「設計したことを実際に試した形跡（PoC・実測）がない」。取り込み率80%・引用率90%・データ品質などの数値はすべて設計目標であり実測値ではない。監査観点では「PoC で未検証箇所を実証 → 実測値で本予算の根拠を固める」という段階を踏むことを強く推奨する。

1.懸念① 営業活動スコープの明確化

監査役の問い：「TeamAgent が営業活動の何をどこまでやろうとしているのか全体像が見えない」

1.1 結論

TeamAgent は「営業を AI に置き換えるシステム」ではない。営業が使う Slack の中に「補助ツール」を5つ置くもので、顧客接点（訪問・商談・送信・交渉・クロージング）は一切やらない。あくまで「机上の準備作業」の時短が狙い。

1.2 営業活動 × TeamAgent カバレッジマップ

営業工程カバー度対応機能内容
リード獲得・新規開拓✗ スコープ外なしリスト作成・アプローチ先探索はしない
アポ取り✗ スコープ外なしテレアポ・打診メール送信は対象外

1.3 全機能共通の「越えない一線」

TeamAgent が構造的にやらないこと
・顧客への送信・提出は全て人が手動（メール・提案書とも）
・AI の出力は社内（Slack / Gmail 下書き / MD・HTML ファイル）で必ず止まる
・AI は「下書き」「調査結果」「判定の参考」までしか作らず、最終成果物・最終判断は人
・gmail.send 権限を技術的に付与しないため「うっかり送信」が構造的に起きない

1.4 スコープ曖昧箇所（v2.3 で明確化を推奨）

曖昧な点監査役への説明推奨する明確化
「5割版」「8割版」の定義完成度の目安。確定基準ではなく営業が使って調整する前提「5割版＝8章＋データ引用済みドラフト」など状態で定義
機能②と機能③の境界機能②はメール1通、機能③は提案書1式。成果物が違う成果物（Gmail下書き vs MD/HTMLファイル）で線引き明記
機能③のデータ引用の精度9割に根拠を付ける目標。100%ではなく人の最終確認が前提「正確性の最終責任は営業本人」と免責的に明記
機能⑤の受注角度判定マネージャーの気づき補助。評価には使わない「人事評価・予算管理には使用しない」と用途限定を明記

1.5 監査役向け 一言サマリー

スコープ 一言サマリー
TeamAgent は営業活動のうち「提案準備（調査・リサーチ）」と「資料・メールの下書き作成」を支援するツール。AI は情報収集を完全に肩代わりし、文章は「下書き」までを作る。一方、顧客接点を持つ全工程（訪問・商談・提案・交渉・クロージング・受注・納品）と、あらゆる顧客への送信・提出は、すべて営業本人が担う。「全部 AI」システムではなく「営業の机上準備を時短する」システム、というのが正確な理解。

2.懸念② データ収集の実現性

監査役の問い：「データが綺麗に・欲しい形で集まるのか確認できているのか」

2.1 結論

v2.2 仕様書は「どのデータソースを・どの API で取るか」は十分書けている。しかし「実際に綺麗に取れたか」を試した形跡（PoC・サンプル検証・精度実測）が一切ない。取り込み率80%・引用率90%などはすべて設計目標であり実測値ではない。

2.3 「綺麗に集まらない」リスク TOP5

順位リスク箇所なぜ危険か緩和策
1位Drive 提案書 PDF/PPTX の構造化抽出抽出エンジンが未定。複雑レイアウト・グラフ内数値・スキャンPDF の抽出可否が不明。機能③の「過去提案を参照」という看板が成立しない恐れPoC で実提案書 20〜30 件を抽出させ、抽出率・数値正確性を実測
2位6ソースを束ねる共通キー（案件ID）の不在Gmail/Slack/Drive に「案件ID」フィールドが存在しない。JOIN の鍵が設計されていない＝「データは集まるが繋がらない」案件マスタテーブルを新設し、案件ID ⇔ Slack ch ⇔ Gmail ラベル ⇔ Drive フォルダ を明示紐付け
3位Gemini DR の JSON 遵守 & 捏造8セクションJSON・実在URL・件数厳格化を LLM が毎回守る保証なし。「もっともらしい嘘」が混入し提案書の信頼性が崩壊JSON スキーマ検証 + 自動リトライ + URL の HTTP 実在確認を必須化。PoC で遵守率を実測（最低50回）
4位Gmail 他人16名の OAuth 網羅性 + 私信分離16名全員が同意操作を完了しないとデータ欠損。私信フィルタ精度（目標80%）が他人メールで未検証オンボーディングで接続を必須タスク化。私信フィルタを本人/他人各100件で精度実測
5位Buzzmiru の構造的欠損 + Slack 案件チャンネル特定不能Buzzmiru は仕様書自身が「落ちる前提」。Slack は命名規則がなく案件チャンネルを機械的に同定できないBuzzmiru 欠損許容をステークホルダー合意。Slack 案件チャンネル命名規則を全社規約化

2.4 監査役向け 一言サマリー

データ実現性 一言サマリー
データ収集は自社運用の VSEO と本人 Gmail については確実に集まる見込み。一方、Drive 提案書からの数値抽出・Gemini DR の品質と捏造防止・6データソースを束ねる共通キー の3点は、仕様書に「できる」と書いてあるが一度も実地検証されていない。現状の数値はすべて設計目標で実測値ではない。本格実装の前に小規模 PoC でこれら未検証箇所を実データで叩いて確かめる必要がある。

3.懸念③ BU1/BU2 の意向差リスク

監査役の問い：「BU1/BU2 で意向が違うとかにならないか」
BU1/BU2 = 営業部署。一方は直クラ営業（事業会社に直接）、もう一方は代理店向き合い（電通・博報堂 等の広告代理店）。

3.1 結論

意向は明確に違う。そして現状の v2.2 はこの差を一切考慮していない（「営業16名」と一括り）。これは監査役の懸念として的確であり、本格設計前に BU 分岐を組み込む必要がある。

3.2 直クラ営業 vs 代理店向き合い：構造的な違い

観点直クラ営業（BU-直）代理店向き合い（BU-代理店）TeamAgent への影響
提案先事業会社の宣伝部・経営層代理店のプランナー・担当者提案書のトーン・粒度が変わる
提案書の役割最終提案物（顧客がそのまま見る）代理店が再加工する素材機能③の出力フォーマットが2系統必要な可能性
営業サイクル比較的長い（関係構築重視）速い・競合多い・コンペ形式機能②③のスピード要件が違う
メール・Slack の相手事業会社の人代理店の人機能②の「文体学習」「文脈解釈」が BU で異なる
KPI受注額・継続率代理店との取引額・案件数・関係性機能⑤の「受注角度判定」のロジックが BU で違う
過去事例の流用性直クラ事例が中心代理店経由事例が中心pgvector の事例検索に BU フィルタが必要
「目的」の意味商材の認知拡大・販促代理店の提案を勝たせる・代理店内での評価機能③ Phase A の「目的」解釈が BU で分岐

表 3.1直クラと代理店向き合いは、提案先・サイクル・KPI・事例の流用性がすべて異なる。同じ TeamAgent でも BU によって最適な動作が変わる。

3.3 v2.2 に潜む BU 非対応リスク

#リスク影響緩和策
R-BU1提案書 FMT の共通化が困難直クラ向け FMT と代理店向け FMT が違うと、機能③の出力テンプレが分岐。1つの FMT に統一すると両 BU から不満FMT を「共通骨格 + BU 別差分」の2層構造で設計。または BU タグで FMT を切替
R-BU2過去事例のクロス参照が的外れBU-直 の事例を BU-代理店 の提案に引用すると文脈がズレる（逆も同様）pgvector の事例検索に bu_type フィルタを追加。デフォルトは同一 BU 内検索、明示時のみクロス
R-BU3Phase A の「目的」解釈がブレる同じ「認知拡大」でも、直クラは「商材の認知」、代理店は「代理店内での提案通過」を意味しうるPhase A の解釈プロンプトに BU コンテキストを注入。営業の所属 BU を入力情報に含める
R-BU4要件定義が BU 分岐を持たないv2.2 要件定義書は全 FR が「営業16名」前提。BU 別の受入基準がないv2.3 で主要 FR（特に FR-3-XX）に「BU 別の受入基準」を追加
R-BU5機能⑤の進捗サマリーが BU 混在マネージャーが BU をまたいで管理している場合、受注角度の基準が混ざる機能⑤を BU 別にグルーピング表示。角度判定ロジックを BU 別にチューニング

3.4 推奨：BU 意向差の「実測」アプローチ

意向差は机上で議論しても結論が出ない。PoC で両 BU から1案件ずつ実データで試すのが最も確実：

BU 意向差 検証 PoC
・BU-直クラ から1案件、BU-代理店 から1案件を選定
・同じ機能③で提案書を生成させ、両 BU の営業に「これは使えるか」をレビューさせる
・「直クラ営業が代理店向けの出力を見て違和感を持つか」「その逆は」を実測
・結果次第で「共通設計でいける」or「BU 分岐が必須」を判断

3.5 監査役向け 一言サマリー

BU 意向差 一言サマリー
直クラ営業と代理店向き合いは提案先・サイクル・KPI・事例の流用性がすべて異なり、意向は明確に違う。現状の v2.2 は「営業16名」と一括りにしており、BU 分岐の設計が存在しない。これは監査役の懸念どおりのリスク。緩和策は (1) 提案書 FMT を「共通骨格＋BU 差分」で設計、(2) 事例検索に BU フィルタ追加、(3) Phase A に BU コンテキスト注入、(4) PoC で両 BU 1案件ずつ実測。本格実装前に BU 分岐方針を確定すべき。

4.懸念④ AWS無料枠プロトタイプの実現性

監査役の問い：「AWS の無料枠でプロトタイプを動かせないか。有料化しないと実現できないのかも、を確認したい」

4.1 結論

完全な「$0」は不可能。ただし新規 AWS アカウントの $200 クレジット（6ヶ月有効）を使えば、機能④単機能・利用者1〜2名の最小プロトタイプを実質ほぼ無料で動かせる。問題は AWS ではなく Anthropic / Gemini の API 実費（クレジット対象外）。

⚠️ 重要：2025年7月の AWS Free Tier 制度変更
2025年7月15日に AWS Free Tier の仕組みが変わった。2025/7/15 以降に作成する新規アカウントは「12ヶ月無料の RDS」が廃止され、代わりに「$200 分のクレジット（6ヶ月有効）」＋「常時無料枠（Lambda 等）」のみ。本精査は最も保守的な「新規アカウント前提」で記述。
