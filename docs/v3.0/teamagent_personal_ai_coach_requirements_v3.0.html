1.本書について

1.1目的

本書は、TeamAgent 機能⑥ "Personal AI Coach"（仮称）の機能要件・非機能要件・受入基準を体系的に定義する。v2.2 までで完成する「営業16名向け Slack AI エージェント（5機能）」の上に、全社員一人ひとりに専属の AI メンター・先輩・秘書を提供する新機能を追加することを目的とする。

v2.2 仕様書（5機能）と異なり、本書は v3.0 構想の 先行設計 として位置付ける。v2.2 完了後に経営判断を行う材料となるよう、FR / NFR / SEC / INT / TST / DR / DoD を v2.2 と同じ記法で記述する。

1.2対象読者

FDE / 実装担当第4〜10章（要件本体）を主に参照。v2.2 共通基盤との整合性、Sprint 別 DoD の達成可否を判定
人事担当第3章ペルソナ、第8章プライバシー（特に SEC-17 人事評価への流用禁止）、第13章リスクを参照
経営層ヒーローセクション、第2章スコープ、第11章コスト、第14章推奨実装順を参照
セキュリティ / 法務担当第8章セキュリティ・プライバシー要件、第13章リスクを参照

2.2含まないもの（Out of Scope）

項目除外理由 / 将来扱い
人事評価への直接利用個人 AI が記録する成長日記・対話履歴を、人事評価・査定・昇進判断に 直接 流用することは禁止（SEC-17）。本人が自発的に提示する場合のみ可。
AI 単独でのキャリア判断「異動すべき」「退職すべき」などのキャリア判断を AI が下すことは禁止。人間（上司・人事）の判断補助に留める。
機密級情報の取り扱いM&A・経営機密・人事評価・給与・賞与情報は SEC-16 の自動検出ルールで個人 AI から除外。誤って学習した場合は即時削除。
顧客側ユーザーへの提供本機能は社内利用専用。顧客向け AI チャットボットは含まない。
AI による自動メール送信v2.2 同様、送信は必ず本人実施（gmail.compose までで gmail.send は付与しない）。
採用候補者の評価FR-6-18（人事向け）では採用支援（候補者リストアップ、面接質問例の提示）は含むが、合否判定は人間が実施。
外部社員（業務委託）への展開v3.0 初期は正社員のみ。業務委託・派遣は SEC-13 のデータ分離要件確認後に検討。
多言語対応初期は日本語のみ。英語対応は v3.1 以降。
音声入出力テキストのみ。音声入出力（Slack Huddle 連携等）は v3.1 以降。
退職者の個人 context 引き継ぎ退職時は個人 context を削除（SEC-15）。後任者には引き継がない（プライバシー保護）。先輩パターンのみ匿名化して保持。

3.ユーザー要件（ペルソナ別）

3.15ペルソナ定義

機能⑥ では、応答トーン・距離感・前提知識を「ペルソナ」として5種類に分類する。役職・年次・部署から FR-6-21（ペルソナ自動選択）が初期値を決定し、FR-6-22（手動上書き）で本人が嗜好に合わせ調整できる。

🌱
Persona 1 / P-1
新卒・若手
入社 1年目以内
「先輩」「メンター」として AI を頼る。優しく励ます。基本用語の説明から始める。否定しない。
想定人数：年 10〜20名（新卒採用ベース） / トーン：丁寧語 + 励まし / 想定利用：1日 5〜10回
💼
Persona 2 / P-2
中堅
2〜5年目
「秘書」「壁打ち相手」として使う。対等な口調。要点だけ伝える。判断材料の提示にとどめ、結論は本人に委ねる。
想定人数：60〜80名 / トーン：丁寧語、簡潔 / 想定利用：1日 3〜5回
🎓
Persona 3 / P-3
ベテラン
5年目以上
「専門アドバイザー」として深い相談相手。専門用語を省略せず使う。本人の知見を尊重し、補足情報・最新動向を提供。
想定人数：30〜50名 / トーン：簡潔、専門的 / 想定利用：1日 2〜4回
📊
Persona 4 / P-4
管理職
課長・部長クラス
「戦略補助・チーム把握」AI。組織視点での提案。1on1 議題サジェスト、メンバーの活動傾向（匿名集計）を提示。
想定人数：10〜20名 / トーン：簡潔、戦略的 / 想定利用：1日 2〜5回
🛠️
Persona 5 / P-5
部署別
営業／開発／人事／経理
各専門ドメインに特化。営業：提案・顧客対応。開発：コード・技術。人事：採用・評価。経理：経費・予算。
P-1〜P-4 と並列で適用。トーンは P-1〜4 の上書き。1日 2〜10回（部署差）
ペルソナ適用ルール
  • 初期判定：FR-6-21 が人事システム（INT-12）から取得した役職・年次・部署で自動判定。例：入社 6ヶ月で営業 → P-1 + P-5（営業）。
  • 本人上書き：FR-6-22 で本人が「もっと対等に話してほしい」「敬語不要」など嗜好を上書き可能。
  • トーン継続：NFR-31 により、同じ人には同じトーンを継続（突然口調が変わらない）。
  • 変更履歴：ペルソナ変更は personal_profile.persona_history（DR-07）に記録し、監査可能とする。

3.2新卒・若手のユーザーストーリー（P-1）

US-P1-01 ／ 新卒（入社 2ヶ月）
先輩に聞きづらい 「経費精算の手順」を AI に聞いたら、図解付きでステップ 1-5 まで丁寧に教えてくれた。「分からなければまた聞いてね」と言ってくれて安心。
US-P1-02 ／ 新卒（入社 4ヶ月）
初めての商談前に 「先輩の佐藤さんはどう商談を進めてた？」と聞いたら、AI が佐藤さんの過去ログから「冒頭 5分のアイスブレイク → 課題深掘り 3問」というパターンを教えてくれた。
US-P1-03 ／ 若手（入社 8ヶ月）
上司から指摘を受けて落ち込んだとき、AI に 「今日の指摘の意図を整理したい」と話したら、感情を否定せず一緒に振り返ってくれた。
US-P1-04 ／ 新卒（入社 6ヶ月）
月末、AI から 「今月の成長記録です」と週次レポートが届いた。「Slack 投稿の語彙が広がった」「30分の商談を 5本完了」と具体的な進歩が見える化されて、自信になった。
US-P1-05 ／ 新卒（入社 1ヶ月）
「AI に何でも聞いていい」と分かっていても、つまらない質問をして人間メンターを煩わせたくなかった。AI なら気兼ねなく 1日 10回でも質問できる。

3.3中堅のユーザーストーリー（P-2）

US-P2-01 ／ 中堅（3年目営業）
朝、Slack を開くと AI が 「今日返信が必要なメール 3件、会議 2本の準備ポイント」を箇条書きで提示。判断は自分でする。AI は秘書として、まとめてくれるだけ。
US-P2-02 ／ 中堅（4年目）
提案書の構成で迷ったとき、AI に 「壁打ち相手になって」と頼む。AI が「価格訴求型」「価値訴求型」の2案を提示してくれ、自分は決めるだけ。
US-P2-03 ／ 中堅（3年目）
大きな商談前、AI から 「想定質問 5問、ネガ反応 3パターン」が事前資料の要約と一緒に届く。会議準備の時間が 1時間 → 15分に短縮。
US-P2-04 ／ 中堅（5年目）
月次の成長レポートで、同期の匿名化ベンチマーク（提案件数・受注率の分布上での自分の位置）が見える。誰と比較されているかは特定できない。
US-P2-05 ／ 中堅（4年目）
プライベートな悩み（家族事情で残業できない等）を AI に話したとき、個人 AI 同士でデータが漏れない仕組みが保証されているので、安心して相談できる。

3.4ベテランのユーザーストーリー（P-3）

US-P3-01 ／ ベテラン（10年目）
業界の最新動向について AI に聞いたとき、「ご存じだと思いますが」と前置きしつつ、自分が知らない最新事例を 3件提示してくれる。レベル感が合っていてストレスがない。
US-P3-02 ／ ベテラン（12年目）
自分のノウハウ（佐藤メソッド）を AI が学習し、若手の質問に答える際に 「佐藤さんはこう進めていました」と引用してくれる。引用前に「あなたのパターンを共有してよいか」の同意確認があった。
US-P3-03 ／ ベテラン（8年目）
後輩から 同じ質問を 3回も聞かれたとき、AI が「彼の AI に過去 3回同じ質問が来ています。マニュアル化を提案しますか？」と教えてくれる（匿名化された傾向情報）。
US-P3-04 ／ ベテラン（15年目）
提案書の表現で迷ったとき、AI が 過去 5年分の自分の提案から「あなたは A 業界では『価値訴求』、B 業界では『価格訴求』を使っています」と傾向を提示してくれる。
US-P3-05 ／ ベテラン（11年目）
AI に 「自分の弱点を 3つ挙げて」と頼んだら、活動データから「①長文メールが多い ②返信が深夜帯 ③Slack スレッドを途中で切る」と冷静に指摘。耳が痛いが本人だけが受け取る情報なので受け入れられる。

3.5管理職のユーザーストーリー（P-4）

US-P4-01 ／ 課長（営業課）
月曜朝、AI から 「チーム10名の今週の動き（匿名集計）」が届く。「メール返信が遅い人 2名」「会議準備が不十分な兆候 1名」など、個人特定できない形での傾向情報。
US-P4-02 ／ 部長（開発部）
1on1 の前に AI が 「田中さんと話すべき議題 3つ」をサジェスト。「最近のコードレビューでの指摘傾向」「先月の目標達成状況」「次のスキル開発提案」。
US-P4-03 ／ 課長（人事課）
採用面接の前、AI が候補者の応募書類を 「強み 3 / 懸念 2」に整理。質問例も 5問提示。合否判断は自分が下す。
US-P4-04 ／ 取締役
経営会議の前、AI が 「過去 3ヶ月の社内動向サマリー」を匿名集計で提示。AI が個別社員の評価をすることはない（SEC-17）。
US-P4-05 ／ 課長（営業課）
部下の AI が記録した個別データに 自分が直接アクセスできない（データ分離 SEC-13）。本人が共有した内容のみ 1on1 で共有される。

3.6部署別ペルソナのユーザーストーリー（P-5）

US-P5-01 ／ 開発エンジニア
コードレビュー前に AI が 「PR 内容の要約 + 過去類似 PR の指摘 3件」を提示。技術ドキュメント検索も同じ AI から可能。
US-P5-02 ／ 経理担当
経費申請の差し戻し対応中、AI に 「この経費は規程的に OK か」と聞くと、規程の該当条項と過去の判例（社内）を引用付きで提示。判断は本人。
US-P5-03 ／ 人事担当
採用候補者からの問い合わせメールに対する返信案を AI が下書き。過去の人事担当者の返信パターンを踏襲しつつ、本人がチェックして送信。
US-P5-04 ／ 営業 + 中堅（P-2 + P-5）
機能⑥ の AI が、v2.2 の機能①〜⑤ と 同じ Slack DM 上で連携。「@TeamAgent 過去事例ある？」（機能①）と「最近の私の傾向は？」（機能⑥）が一画面で完結。

12.用語集

AI コンテキスト（Personal Context）
個人ごとに保持される、AI の「その人についての理解」。役割・目標・嗜好・最近の活動・過去会話を含む。DR-07 / 08 / 09 / 12 が物理的な格納先。
ペルソナ（Persona）
応答トーン・敬語レベル・専門用語使用度・距離感を調整する 5種類のプロファイル（P-1 新卒〜P-5 部署別）。役職・年次・部署で自動判定し、本人嗜好で上書き可。
先輩パターン（Senior Pattern）
ベテラン社員のメール返信・提案進行・会議運営などの業務手法を AI が学習したもの。本人の明示的同意（オプトイン）が必須。同意撤回でいつでも削除可。
成長日記（Growth Journal）
AI が自動生成する個人の成長記録。週次・月次の活動量・成果・前期比・目標進捗を含む。本人のみアクセス可（管理職・人事はアクセス不可）。SEC-17 により人事評価への直接流用禁止。
データ分離（Data Isolation）
A さんの個人 AI と B さんの個人 AI で情報が漏れない技術設計。RDS RLS + Slack DM チャネル分離 + プロンプト混入検出の 3層で実現（SEC-13 / NFR-30）。
意図ルーター 4-way（4-way Intent Router）
v2.2 の 3-way（recall / compose / analyze）に「coach（個人 AI への質問）」を加えた 4-way 判定。Slack DM で受けた発話を 4分類し、適切な処理に振り分ける。FR-6-16 / DoD-S23 で実装。
Tier 0〜3（データ階層）
v2.2 DR-06 を継承。Tier 0：公開可、Tier 1：社内全体、Tier 2：部署内、Tier 3：機密（M&A・経営・人事評価・給与）。Tier 3 は personal_context / company_knowledge への格納禁止（SEC-16）。
匿名化ベンチマーク（Anonymized Benchmark）
同年次・同部署の集計から見た本人の位置を、個人特定不可な箱ひげ図風で提示するもの。集計母数 5名未満は表示禁止（FR-6-12）。
Domain-wide Delegation（DWD）
Google Workspace サービスアカウントが全社員のデータに代理アクセスできる仕組み。v3.0 では company_knowledge（DR-11）構築のため、drive.readonly スコープのみで取得（INT-11）。
オプトイン同意（Opt-in Consent）
「同意した人のみ対象」とする原則。デフォルトは「同意していない」。SEC-14 の先輩パターン抽出に適用。

13.リスクと留意事項

リスク影響緩和策
心理的安全性（AI に監視されてる感） 大：社員が「自分の AI が人事に筒抜け」と誤解すれば利用率が激減し、機能⑥ が空虚に。 SEC-17（人事評価への流用禁止）を技術 + 規約 + 経営層メッセージで三重に担保。SEC-18 監査ログを本人が自由に閲覧可能とする透明性。導入時の全社員説明会で丁寧に説明。
人間メンターとの役割分担 中：AI に頼りすぎて人間メンター（先輩・上司）との関係が希薄化。逆に AI 軽視で活用されない。 新卒研修で「AI は気兼ねない 1次相談、人間メンターは深い相談」という棲み分けを明示。AI が人間メンターへの相談を能動的にサジェスト（FR-6-11 学習計画に「先輩への弟子入り提案」を含める）。
先輩データの同意取得 大：ベテラン社員の同意（SEC-14）が取れないと FR-6-06 が機能しない。「自分のノウハウを AI に取られる」という抵抗感。 (1) 同意 UI で「名前付き引用 / 匿名化 / 同意しない」の 3択を明確化、(2) 撤回いつでも可、(3) パイロット 10名でメリット（後輩教育の負担減）を実感後に拡大。
機密情報の自動検出（HR・経営情報） 大：Tier 3 機密（M&A・査定・賞与）が誤って personal_context / company_knowledge に混入すれば情報漏洩リスク。 SEC-16 で 4 層検出（専用 channel / 専用 folder / キーワード辞書 / LLM 判定）。誤混入時の通報窓口 + 24h 以内削除 SLO。月次自動監査。
退職者の個人 context 取り扱い 中：退職時に context の取り扱いが不明確だと、本人にも会社にも遺恨を残す。 SEC-15 で退職処理 SOP を明文化：personal_* は 30日以内に全削除、senior_patterns は本人合意で匿名化保持 or 削除。退職前面談に組み込む。
スケール時のコスト超過 大：200名展開で月次 +¥350K を超過した場合、ROI が破綻。 Phase 制ロールアウト（営業16名 → 50名 → 100名 → 200名）。各 Phase でコスト実績を経営報告。利用量に応じた Claude モデル最適化（Sonnet → Haiku）も検討。
v2.2 既存機能との衝突 中：意図ルーター 4-way 化で v2.2 の 3-way が劣化。Slack DM の UX が複雑化。 FR-6-16 で v2.2 5機能との統合を必須化。回帰テスト（v2.2 DoD-Final のテストセット）を S23 で実施し、劣化 0 を確認後にリリース。
ペルソナ判定ミス 小：自動判定（FR-6-21）が外れると初期トーンが不適切。 本人による上書き（FR-6-22）を初日から提示。3日経過後の上書き率を観測し、≥ 30% の場合は判定ロジック見直し。
労務・法務観点（モニタリング規制） 大：従業員のメール・Slack を AI が読むことが、労働法上の「モニタリング」に該当する可能性。 法務 + 労務に事前レビューを依頼。利用規約改訂 + 全社員同意取得 + 労使協議（必要に応じて）。SEC-17 / SEC-18 を労務観点でも整合。
過剰依存（AI なしで判断できない社員の発生） 中：新卒・若手が AI 頼みになり、自分で考える力が育たない。 P-1 ペルソナの応答に「あなたはどう思いますか？」を意図的に挟む。月次レポートで「AI に聞いた回数」が突出して多い社員は人間メンターに連絡する仕組み（本人合意ベース）。

14.4並行開発の判断材料

v2.2 完了を待たずに機能⑥ と並行開発する選択肢もあり得るが、本書は 「v2.2 完了後の段階展開」を推奨 する。理由：

  1. v2.2 の共通基盤がプロダクション稼働してから機能⑥ を載せる方が、基盤側の不具合と機能⑥ 固有の問題を切り分けやすい。
  2. SEC-13〜18 は v2.2 完了時点の運用実績（特に SEC-01〜12 の実装結果）を踏まえて設計したほうが現実的。
  3. FDE 人員を 2系統（v2.2 と機能⑥）に分けると、v2.2 完了の見込みが後ろ倒しになるリスクが大きい。

ただし、人員に余力がある場合は FR-6-01 / 02 / 05 / 21 の MVP（質問応答 + プロファイル）のみを v2.2 後半（S15〜S17）で並行着手する選択肢はあり得る。経営判断と FDE 工数の余力に応じて判断する。
