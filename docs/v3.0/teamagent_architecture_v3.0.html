1本書について

1.1目的・対象読者

本書は TeamAgent v3.0 のシステムアーキテクチャを C4 model（Context / Container / Component） と Skill Registry / Plugin パターン の両軸で定義する技術設計文書である。実装計画書・要件定義書・MVA 仕様書の前提となる「どのコンポーネントがどう繋がるか」を一意に固定することを目的とする。

読者主たる利用箇所
開発リード第 4〜9 章（コンテナ／コンポーネント／Agent SDK／Skills／Tools）
インフラ担当第 12 章（AWS / Terraform / Docker Compose）
セキュリティ／監査役第 11 章（Permissions / Guardrails）
営業現場・上長第 2 章のみ（概念アーキテクチャ。技術詳細は省略可）
外部ベンダー第 9 章（MCP 接続層・認証）

1.2v2.2 からの主要変更

領域v2.2 までv3.0
Agent ランタイムOpenClaw を中核に据えた構成案を併記OpenClaw 関連設計を全面削除。Claude Agent SDK 単独構成
概念モデル「Slack + 4 共通基盤 + 6 データ層」の素朴3層User / Frontend / Agent SDK (Orchestrator) / Data Layer の 4 層を概念層として導入
Skills機能 ①〜⑤ という呼称のみ、コードレベルの抽象化なしSkill Registry + Plugin パターン化。Initial Skill Set（5 個）が独立 Python パッケージとして疎結合。新規 Skill は core 改修なしで追加可能
Tools暗黙的に Claude SDK に内包内部 Tool と外部 MCP を明確分離。MCP サーバー認証フローを独立設計
MemoryRDS にスレッド ID で保存（言及のみ）スレッド単位 / 案件単位 / ユーザー単位の 3 階層を技術設計化
GuardrailsOAuth + 監査ログ程度Permissions / Guardrails を 1 章割り当て。薬機法・景表法・ステマ規制チェッカーを含む
出力MD / HTML 生成のみ言及HTML → PDF（Playwright）／PPTX（python-pptx）変換層を新章。WYSIWYG エディタ設計も追加

1.3用語

用語本書での意味
Claude Agent SDKAnthropic 公式の Python SDK。ループ制御・Tool 呼び出し・Memory を提供する Agent ランタイム。本書では「Agent SDK」と略。
Skill RegistrySkill を Plugin として登録・検索・起動するレジストリ。Orchestrator（Reasoning Loop）から動的選択される。新規 Skill 追加は Registry への登録だけで完結し、core 改修は不要。
Skill独立した Python パッケージ（Plugin）として実装される処理単位。共通の Skill interface（name / description / input_schema / output_schema / required_tier / run）を実装する。本書では Initial Skill Set として 5 個（Slack サジェスト / Mail / 提案生成 / 動画分析 / 進捗サマリー）を初期登録。
Initial Skill SetSkill Registry に最初に登録される 5 個の Skill。製品仕様上の固定数ではなく、Phase に応じて追加・差し替え可能。
Tool / MCPAgent SDK が呼び出す外部機能。内部 Tool（pgvector・Gmail）と外部 MCP（Gemini DR・dpro 等）に分かれる。
Memory / Session会話履歴・案件情報・ユーザー嗜好を保持する 3 階層のストア。
GuardrailsOAuth・PII マスク・Tier 制御・法令チェッカーの総称。
Tier 0-3データ参照権限の 4 段階。0=本人のみ／1=案件担当／2=チーム／3=全社。
MVAMinimum Viable Architecture。社内検証用の最小実用構成。詳細は別冊。

2概念アーキテクチャ — User からデータまでの 4 層

v3.0 で最も重要な章。「AI Agent とは何か」を営業現場・上長・監査役が同じ言葉で語れるよう、User / Frontend / Agent SDK (Orchestrator) / Data Layer の 4 層モデルで定義する。

図 2.14 層概念図。User（営業）が Slack（Frontend）にリクエスト → Agent SDK（Orchestrator）が Skill Registry から最適 Skill を選択し、Tools / Memory を呼び出して処理 → 結果を Slack で返す。Guardrails は Agent SDK 全体に常時適用。

2.3Agent SDK の 4 構成要素 — 技術コンポーネントとしての独立化

┌──── Claude Agent SDK 内部の 4 層 ────────────────────────────────────────────┐ │ │ │ ① Skills │ Skill Registry │ Python パッケージ群（Plugin として登録）│ │ ─────────────────┼─────────────────┼────────────────────────────────────── │ │ ② Tools / MCP │ Tools Layer │ 内部 Tool + 外部 MCP サーバー │ │ ─────────────────┼─────────────────┼────────────────────────────────────── │ │ ③ Memory/Session│ Session Storage │ スレッド / 案件 / ユーザー の 3 階層 │ │ ─────────────────┼─────────────────┼────────────────────────────────────── │ │ ④ Guardrails │ Permissions Layer│ OAuth + Tier + PII + 法令チェッカー │ └──────────────────────────────────────────────────────────────────────────────┘

図 2.3v3.0 で独立した 4 層。v2.2 までは Agent SDK の中で混在していたが、v3.0 では各層を独立コンポーネントとして章割り当て（第 8 / 9 / 7 / 11 章）。Skills 層は Skill Registry によって Plugin パターンで拡張可能。

この 4 層分離は「営業現場が変更要求を出したときに、どの層に影響するか」を即答できる設計上のメリットを生む。たとえば「Slack のサジェスト条件を変える」は Skills 層（該当 Skill Plugin のみ）、「Buzzmiru の認証方式が変わる」は Tools 層、「過去スレッドを跨いだ要約が欲しい」は Memory 層、「個人情報を伏せたい」は Guardrails 層。新規 Skill 追加は core を触らず Registry 登録のみで完結する。

8Skill Registry アーキテクチャ（Plugin パターン）

v3.0 の中核章。Skill は Plugin パターンで構成され、Skill Registry が core レイヤーで Skill のライフサイクル（登録／検索／起動／終了）を管理する。Skill は schema-based の入出力契約（Plugin Contract）を持ち、新規 Skill は core 改修なしに追加可能。Initial Skill Set として 5 個（Slack 自動サジェスト / Mail / 提案生成 / 動画分析 / 進捗サマリー）が最初に登録されているが、これは Registry への初期登録分であって製品仕様上の固定数ではない。

8.1Skill インターフェース（Plugin Contract）

項目定義
name一意識別子（snake_case）。例：video_analysis / mail_workflow
descriptionLLM が選択に使う自然言語の説明（日本語、80〜200 文字）
input_schemaJSON Schema / Pydantic モデル（user_id, project_id, params など）
output_schemaJSON Schema / Pydantic モデル（content_md, attachments[], next_actions[]）
required_tierSkill 実行に必要な最小 Tier（0-3）
max_loop_iterationsAgent Loop 上限（既定 10、video_analysis 15、proposal_generator 30）
depends_on依存する他 Skill / Tool / MCP の宣言（依存解決に使用）
run(ctx, args)非同期実行メソッド。ctx は Memory / Tools / Guardrails への参照

8.2Skill Registry の動作（登録 / 検索 / 起動 / ライフサイクル）

┌── Skill Registry：起動時の Plugin 登録 ─────────────────────────────┐ │ │ │ [起動時] │ │ ① Skill パッケージを importlib.metadata で発見（entry_points） │ │ ② 各 Skill の interface（name / schema / required_tier）を検証 │ │ ③ depends_on を解析 → 依存 Skill / Tool / MCP の存在確認 │ │ ④ Registry に登録：registry[name] = SkillInstance │ │ ⑤ name の重複は起動失敗（ConflictError） │ │ │ │ [実行時] │ │ ⑥ list_skills(tier) → 現在ユーザーが起動可能な Skill のみ列挙 │ │ ⑦ get_skill(name) → Skill インスタンス取得 │ │ ⑧ ライフサイクル：start → run(ctx, args) → emit_event → finalize │ └─────────────────────────────────────────────────────────────────────┘

8.5Skill interface 疑似コード例

Skill interface（Plugin Contract）の Python 実装例。Pydantic でスキーマを定義し、Skill Protocol を実装するだけで Registry に登録される。下記は検索系 Skill（slack_suggest）の最小実装例。

# skills/slack_suggest/schemas.py from pydantic import BaseModel from typing import Literal class SlackSuggestInput(BaseModel): user_id: str channel_id: str thread_ts: str | None = None recent_message: str # トリガとなった直近メッセージ top_k: int = 5 class SlackSuggestOutput(BaseModel): suggestions: list[str] # 候補発言（最大 top_k 件） sources: list[dict] # 根拠：過去 Slack メッセージ ID 等 confidence: float # 0.0 - 1.0 # skills/slack_suggest/skill.py from teamagent.core.skill import Skill, SkillContext from .schemas import SlackSuggestInput, SlackSuggestOutput class SlackSuggestSkill(Skill): name = "slack_suggest" description = ( "Slack スレッドで進行中の会話を検知し、過去類似ケースから " "次の発言候補を 3 秒以内に提示する。Tier 1 以上で利用可能。" ) input_schema = SlackSuggestInput.model_json_schema() output_schema = SlackSuggestOutput.model_json_schema() required_tier = 1 max_loop_iterations = 3 depends_on = ["pgvector_search", "slack_history"] async def run( self, ctx: SkillContext, args: SlackSuggestInput ) -> SlackSuggestOutput: # ① 直近メッセージのベクトル化 embedding = await ctx.tools.embed(args.recent_message) # ② pgvector で類似過去ケース検索（Tier 制約を ctx が自動付与） hits = await ctx.tools.pgvector_search( embedding=embedding, top_k=args.top_k, filters={"source": "slack", "channel_id": args.channel_id}, ) # ③ Memory（スレッドスコープ）から会話履歴を取得 history = await ctx.memory.get(scope="thread", key=args.thread_ts) # ④ LLM に suggestion 生成を依頼（Guardrails を通る） suggestions = await ctx.llm.complete( system="過去類似ケースから次の発言候補を生成…", context={"hits": hits, "history": history}, ) return SlackSuggestOutput( suggestions=suggestions, sources=[{"id": h["id"], "score": h["score"]} for h in hits], confidence=min(1.0, sum(h["score"] for h in hits) / args.top_k), ) # skills/slack_suggest/pyproject.toml（抜粋） [project.entry-points."teamagent.skills"] slack_suggest = "skills.slack_suggest.skill:SlackSuggestSkill"

図 8.5Skill interface 疑似コード例（slack_suggest）。Pydantic でスキーマを定義し Skill Protocol を実装。ctx 経由で Memory / Tools / Guardrails / LLM にアクセスする。entry_points に登録された Skill は起動時に Registry が自動発見・登録する。

11.2Tier 0-3 アクセス制御

Tier定義参照可能データ典型 Skill
Tier 0本人のみ自分の Mail / DM / 個人 DriveMail ワークフロー
Tier 1案件担当当該案件チャンネル / 共有 Drive提案コンテンツ生成・進捗サマリー
Tier 2チーム営業16名共有データ / 動画ライブラリ動画ナレッジ分析・提案コンテンツ生成
Tier 3全社公開データ / 一般市場リサーチ提案コンテンツ生成（市場分析時）

Skill 実行時、required_tier と「現在のユーザー × データソース」を突き合わせ、不足する場合は tool_error を返す。pgvector 検索でも WHERE tier <= :user_max_tier の制約を必ず付与。

11.5薬機法 / 景表法 / ステマ規制チェッカー（Phase 4 以降）

規制チェック内容実装
薬機法「効果」「効能」「治る」等の禁止表現／医薬品的標榜禁止語リスト + LLM 二次判定（プロンプト分離）
景表法「最大」「最高」「業界 No.1」等の不当表示／優良誤認禁止語リスト + 根拠データ有無の確認
ステマ規制広告表記欠落 / 「個人の感想」抜けの宣伝テンプレに [PR] [広告] 明記の埋め込み + 検査

提案コンテンツ生成 Skill・進捗サマリー Skill 等、外部送付物を生成する Skill の出力直前、Guardrails ゲートで実行。違反検知時は LLM に修正版を再生成させ、3 回失敗で人間レビュー依頼（Slack DM）。
