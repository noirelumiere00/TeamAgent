0.エグゼクティブサマリー

ユーザーから提示された3つの認識（① 案件番号で施策内容を管理 ② 全PDF化でリスク減 ③ 子会社 OpenClaw が IG スクレイピング）について、2つの Agent で Web リサーチ + 複合再評価を実施した。

ユーザーの認識検証結果判定
① 案件番号で施策内容を管理するのが良い正解。Salesforce / Glean / Snyk など成功事例すべてが「案件マスタをハブ」にした設計。案件番号があると「どの案件のどのデータが欠けているか」を機械検知できる✅ OK
② 全PDF化でリスクは少ない半分正解。「テキスト埋め込みPDF」に統一すれば抽出率 74%→約80% に改善。ただし (a) DR の捏造リスクには無効 (b) PPTXグラフ数値の抽出ルートを失う (c) 既存1,500件の遡及付与という新難所が生まれる⚠️ 条件付き
③ 子会社 OpenClaw が IG スクレイピング高リスク。IG 規約違反（グレー〜アウト）+ OpenClaw が Google から無警告大量BANされた実績 + OSS エージェント自体の脆弱性（RCE・ボットネット化）の三重苦。業界は「無断スクレイピング → 公式API・ライセンス」へ移行中🔴 要見直し
⚠️ 最重要メッセージ
① 案件番号管理はそのまま進めてよい正解。② PDF統一は「テキスト埋め込みPDFであることを運用で保証し続けられれば」という条件付きで有効、「PDF化すれば全部解決」は楽観バイアス。③ 子会社の IG スクレイピングはTeamAgent とは別問題だが、同じ系譜のリスクとして最優先で見直すべき。

1.あなたの3つの認識への回答

1.1 「案件番号で施策内容を管理」→ ✅ 正解

これは前回リサーチ（Agent C）が出した「案件マスタをハブにした設計」そのもの。Salesforce Data Cloud・Glean・Snyk など成功事例が全てこの形。

project_master（案件マスタ = 案件番号ハブ） project_id 例：2026-BU1-0042（BUコードを内包させると懸念③のBU分岐にそのまま使える） project_name / client_name / bu / status aliases 略称・社内通称の配列（LLM推定マッチの精度を左右） project_source_link（紐付け表 = TeamAgent 側に持つ） project_id → Gmail スレッド / Slack チャンネル / Drive提案書 / 動画分析 / Gemini DR match_method（deterministic / auto_llm / convention / manual） confidence / verified_by

重要な原則：案件番号を Gmail/Slack/Drive 側に書き込むのではなく、TeamAgent 側に「案件番号 ⇔ 各データ」の対応表を持つ。案件番号は紐付けの「正解ラベル」を与えるもの。

1.2 「全PDF化でリスク減」→ ⚠️ 半分正解（条件付き）

観点評価
✅ リスクが減る点フォーマットばらばら問題（PDF/PPTX/複雑PDF/スキャンPDF の4分岐）の入口を1本化できる。新規分のスキャンPDF混在をゼロにできる。文書抽出率 加重平均 74% → 約80%（+6pt）
⚠️ 条件「テキスト埋め込みPDF」であることが保証されて初めて成立。印刷スキャンPDFや画面スクショ貼付PDFが混ざると元の74%問題に逆戻り。運用徹底に依存する
❌ 失うものPPTXネイティブグラフの数値抽出ルートを自ら捨てる。前回リサーチ唯一の朗報（python-pptx で埋め込みExcelから数値を完全取得）が、PDF化でグラフが画像化して消える
❌ 残るリスク複雑レイアウトの読み順崩れ・図表内数値・ネスト表の構造崩れはフォーマットを変えても直らない。DR の捏造リスクとは完全に無関係
🆕 新難所既存1,500件への案件番号遡及付与。新規案件より歩留まりが低く（50-70%）、誤マッチがデータ汚染を生む

推奨：グラフ数値が業務上重要なら「PDF＋元PPTXのデュアル保管」。PPTX からは python-pptx で数値だけ別途抽出。

1.3 「子会社 OpenClaw が IG スクレイピング」→ 🔴 高リスク

詳細は第2章。要約すると三重リスク：(1) IG 規約違反（グレー〜アウト）、(2) OpenClaw が2026年2月に Google から無警告大量 BAN された実績、(3) OSS エージェント自体の脆弱性（CVE-2026-25253 ワンクリック RCE・脅威アクターによるボットネット化）。

2.IGスクレイピングの実態（海外事例・AIブーム）

2.1 法的状況（2026年時点）

地域整理
米国hiQ・Bright Data 判決により「未ログイン公開データのスクレイピングは CFAA 違反ではない」。ただし規約違反・著作権・動産侵害のリスクは残る「グレー」。Meta はログイン型・転売型には連勝、未ログイン型には連敗（Bright Data 訴訟は Meta が2024年に取下げ）
日本「グレー」。スクレイピング自体を直接禁止する法律はない（著作権法30条の4＝情報解析目的の例外あり）。ただし規約違反・不正競争防止法・著作物の大量複製蓄積・再配信は違法化しうる
状態該当行為
✅ 比較的セーフ寄り未ログイン・公開データのみ・サーバ過負荷を与えない・転売しない
⚠️ グレー規約違反（契約違反責任は残りうる）・著作権・動産侵害
❌ アウト寄りログイン状態でのスクレイピング、偽アカウント使用、技術的保護回避、収集データの大規模転売

トレンド内容

2.5 子会社 OpenClaw の IG スクレイピングへの評価

🔴 子会社 OpenClaw による IG スクレイピング = 高リスク（三重苦）
① IG 規約違反（グレー〜アウト）
② OpenClaw が Google から無警告大量 BAN された実績（2026年2月・OAuth トークン悪用で Antigravity/Gemini を不正廉価利用、Gmail/Workspace ごと凍結）
③ OpenClaw 自体の脆弱性（CVE-2026-25253 ワンクリック RCE、脅威アクターによるスクレイピングボットネット化の実態 — DataDome 調査）

プラットフォームは OSS エージェント経由のアクセスを無警告で一括遮断する姿勢。業務利用としては推奨できない。最優先で運用停止または代替手段への切替を検討すべき。

2.6 TeamAgent の動画分析データ層への教訓

  • dpro・Buzzmiru がスクレイピング系なら、IG 事例がそのまま当てはまる：いつ壊れてもおかしくない前提でアーキを組む
  • VSEO（自社）+ dpro + Buzzmiru の3本立ては、3本ともスクレイピング系なら「同じリスクの3重化」にすぎず冗長性として機能しない
  • データ層は「サービスが入れ替え可能」な抽象化レイヤーにし、特定スクレイパーにロックインしない設計が必須
  • 推奨：公式 API・公式データパートナーシップを基本線に据える。スクレイピングは「いつ止まってもおかしくない繋ぎ」と位置づける

効果内容

3.3 既存1,500件への遡及付与（新難所）

⚠️ 前提②の最大の難所：1,500件への案件番号 遡及付与
過去の提案書には案件番号が存在しない。LLM がファイル名・本文から抽出 → CRM 案件リストと fuzzy match → 人手確定、というフローが必要。
・遡及自動マッチ歩留まり：50-70%（新規案件の70-85%より低い）
・誤マッチは別案件のデータ汚染を生む（Salesforce 公式も「fuzzy match は慎重に」と警告）
・割り切り推奨：直近2-3年・受注済み・高価値案件を優先。古い低価値案件は「案件未特定」プールで全文検索だけ効かせる

5.統合結論と推奨アクション

5.1 監査役向け 統合サマリー

統合結論
① 案件番号管理は正解 — そのまま進めてよい。Salesforce/Glean/Snyk の成功パターン。BUコードを採番に内包させると懸念③のBU分岐にもそのまま使える。

② PDF統一は条件付きで有効 — 「テキスト埋め込みPDF」を運用で保証し続けられれば、文書抽出率は 74%→約80% に改善。ただし「PDF化すれば全部解決」は楽観バイアス。(a) DR の捏造には無効 (b) PPTXグラフ数値の抽出ルートを失う (c) 既存1,500件の遡及付与という新難所が生まれる。

③ 子会社 OpenClaw の IG スクレイピングは高リスク — IG規約違反 + OpenClaw の Google BAN実績 + OSS脆弱性の三重苦。業界は「無断スクレイピング→公式API・ライセンス」へ移行中。TeamAgent の dpro/Buzzmiru 依存にも同じ教訓が当てはまる。

2つの新前提（PDF統一・案件番号）はリスクを「消す」のではなく「可視化・追跡可能にする」性質のもの。PoC の必要性は変わらない。
