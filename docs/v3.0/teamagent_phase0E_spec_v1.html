1.本書について

本書は、TeamAgent 機能③ 提案コンテンツ生成（営業ヒアリングで「20時間/案件」の最大ペインとして特定された業務）の Phase 0 から Phase E まで の全工程について、営業のみなさんに 確認・合意 をいただくための資料です。

確認してほしい 6 つの大項目
  1. 入力 5 フィールド（商品名・CL名・URL・目的・その他）でカバーできるか？
  2. Phase A の確認ステップ（「この条件で進めます」）は必要か？（精度 vs 営業の手間）
  3. Phase B の所要時間（5〜10分）は許容範囲か？
  4. Phase D の対話往復数（5往復で8割）は妥当か？
  5. 出力フォーマット（MD + HTML）で OK か？（pptx は v3.0 で対応予定）
  6. Phase E（最後の2割）として残す範囲は妥当か？
本書で「変更不可」の前提
以下の 2 点はすでに技術側で決定済みです（営業判断の対象外）：
  • 「統合派」採用：Gemini Deep Research（DR）は外部サービスを買うのではなく、TeamAgent 内部の Phase B サブエージェントとして組み込みます。これにより営業さんは @TeamAgent ひとつで完結します。
  • Pattern Bravo+ 4経路冗長化：DR が Vertex AI のクォータ切れや障害で失敗しても、自動で 3 つの予備経路に切り替わります。営業さんから見ると「DR が落ちる」状況はほぼ発生しません。

本書の読み方

  • 各 Phase の 色分け（Phase 0 紫 / A 青 / B 緑 / C 黄 / D オレンジ / E 赤）を目印にしてください
  • 各 Phase の末尾に「✋ 営業さんに確認したいこと」ボックスがあります（オレンジの帯）。そこで Yes/No や選択肢を記入してください
  • Slack モックは「実際の Slack でこう見えます」のイメージです。文言・ボタン配置は調整可能
  • 所要時間は仮の目標値。社内ベータでの計測後に確定します

4.1TeamAgent が解釈する内容

営業が入れた 5 フィールドは「ざっくりした自由記述」ですが、Phase B の並列リサーチを精度高く動かすには「もう一段、構造化された検索条件」が必要です。Phase A では TeamAgent（Sonnet 4.6）が以下を 30 秒で推定 します。

解釈項目例（ACME 美容液 X / 30代女性ターゲットの場合）使い道
業界カテゴリBtoC 美容 / スキンケア / 美容液セグメント過去案件メタの絞り込み
商材特性消費財・継続購入型・客単価 ¥5,000〜¥15,000 帯競合動画の選定
ターゲット像30 代女性／美容意識高め／SNS リテラシー中〜高界隈分析・タグ調査
想定 KPI販売促進（販売件数 / 売上 / コンバージョン）提案書のゴール設定
想定ペイン「シミ・くすみへの即効性が見えない」「価格に対する効果実感」DR と X 検索の方向性
類似過去案件Gamma 健食 PR（類似度 0.82）／ Delta 化粧水 PR（0.71）pgvector Top5 を Phase B に injection

Phase B

並列リサーチ（4 つの Tier）

Tier 0 → Tier 1〜3 並列でリサーチ。動画分析データ＋過去案件メタが差別化の本丸、Gemini DR は外部文脈の中核
5〜10分
並列リサーチ

Phase B は機能③ の 最大の差別化ポイント です。「業界一般論を吐く ChatGPT」と違い、TeamAgent は 自社の動画分析データ層（VSEO + dpro + Buzzmiru）と 過去案件メタ（pgvector）を最優先で読みに行きます。そのうえで Gmail / Slack / Drive / DR / X を並列で読み込み、データドリブンな提案の土台を作ります。

4 つの Tier 構造

Tier 0 ｜ 差別化の本丸（最優先）
動画分析データ層 + 過去案件メタ
ここが 競合 AI ツールにはない 差別化要素。VSEO + dpro + Buzzmiru の 3 社統合データに、pgvector に蓄積された自社過去案件のメタ（消化額 / 保存率 / フック / 訴求軸 / CTA / 表現）を組み合わせて「うちで実施した案件パターンに基づく具体的な数値根拠」を Phase C に渡す。
🎬 VSEO Analytics（自社） 📊 dpro（広告消化額） ⭐ Buzzmiru（保存数） 🗄 pgvector cases（過去案件 200+ 件）
Tier 1 ｜ 案件文脈データ
この案件の本人視点・社内合意の引き継ぎ
営業本人の Gmail と案件 Slack チャンネルから「すでに決まっていること」「クライアントが言ったこと」「過去のキャッチボール」を拾う。営業が自分でやると 1 案件 30〜60 分かかる作業。
📧 Gmail（本人スレッド） 💬 Slack（案件チャンネル）
Tier 2 ｜ 横展開データ
社内のナレッジから類似事例を引く
他の営業が過去に同種クライアントを扱ったメール / Slack 議論 / Drive の過去提案 PDF を引く。「会社の脳」としての TeamAgent の本領発揮ゾーン。
📧 Gmail（他人横断） 💬 Slack（横断） 📁 Google Drive（過去提案 PDF/pptx）
Tier 3 ｜ 外部文脈データ（DR 中核）
Gemini Deep Research の 8 セクション JSON
市場・ソーシャル・ハッシュタグ・パブリシティ・界隈・競合・インサイト・イベントの 8 セクション構造化 JSON を生成。「業界一般論」ではなく「URL 実在確認 + 件数厳格化」のルールで動く。X はオンデマンドで補強検索。
🔍 Gemini DR（8 セクション JSON） 🐦 X（補助検索）

図 5.1Tier 構造。Tier 0 から先に実行し、その出力を Tier 1〜3 のクエリに inject して並列実行する。これにより「Tier 0 で見つかった類似案件パターン」が Tier 3 の DR テーマ設定にも反映される。

5.1Tier 0：差別化データ取得（最優先）

動画分析データ層（VSEO + dpro + Buzzmiru）

サービス取得する情報差別化価値
VSEO Analytics動画内容分析（Sonnet 4.6 ビジョン）・3シークレット重複度自社運用 / ◎ 安定
dpro広告消化額・配信期間・カテゴリ別ランキング・台本/ナレーション外部にも提供されているが統合層で参照
Buzzmiru保存数（save count）— 他で代替不可 の質的指標「効率（保存率）」を測れる唯一のソース

過去案件メタ（pgvector cases テーブル）

これまで TeamAgent が記録した 200+ 件の過去案件 をベクトル検索で類似度照合します。各案件には以下のメタが紐付いています：

  • クライアント業種 / 商材 / KPI / 予算規模
  • 採用したインフルエンサー構成（Top / ミドル / マイクロの内訳）
  • 動画 6 観点（EG情報 / メディア界隈 / フック / 訴求軸 / CTA / 表現）
  • 消化額・保存率・受注角度（成功 / 微妙 / 失敗）
  • 類似案件として参照可能な「成功パターンと失敗パターン」

  • Q2. 顧客固有表現（口癖 / NG ワード / 政治）は TeamAgent が学習・蓄積していくべきですか？

  • Q3. Phase E で MD/HTML を使うのは現実的ですか？
    HTML をブラウザで開いてコピペで OK MD をエディタで編集 pptx じゃないと作業できない（v3.0 まで待つ）
  • Q4. 最終提出形式は？
    PDF pptx HTML をそのまま共有 クライアントによる

10.データドリブン度を担保する仕組み

10.1 すべての主張に「データソース引用」を明示

提案書のすべての主要文に 引用元（データソース） を付ける運用にします。クライアントへの提出時は引用を非表示にする 営業向け表示 と 顧客向け表示 の 2 モードを持たせる予定。

主張の種類引用元の表記例
市場規模「DR セクション A・出典 a / 経路1 Vertex AI」
ターゲット行動「DR セクション G・X 補強 + 弊社過去案件 12 件平均」
動画戦略「Tier 0 動画分析 / VSEO + Buzzmiru 保存数 Top5%」
過去類似案件「pgvector 類似 0.82 / Gamma 健食 PR 2025/12」
媒体戦略「Drive 提案 ID:1247 / 2025-09 Delta 化粧水」

10.2 業界一般論 vs データドリブンの違い（実例）

例 1：ターゲット行動

ChatGPT 一般論

例 2：競合差別化

ChatGPT 一般論

10.3 「要確認」表示の運用

データが取れなかった項目は 「要確認」 と明示します。絶対に捏造しない ことが営業の信頼を担保します（DR の絶対遵守ルール）。

11.よくある質問（FAQ）

Q1. DR で取れないクライアントの場合は？（マイナー商材・BtoB ニッチ・新カテゴリ）
Pattern Bravo+ の経路 1（Vertex AI）が「該当データなし」を返した場合、自動で経路 2〜4 にフォールバックします。それでも取れない項目は 「要確認」 表示で残し、Tier 0（自社過去類似案件）と Tier 1（営業の Gmail / Slack）の比重を上げて提案を組みます。「外部 DR で取れないことが分かっているクライアント」は事前に営業がフラグ立てできる設計も検討中。
Q2. 自社データ（過去案件）が少ない案件の場合は？（新カテゴリ進出など）
Tier 0（自社過去）が手薄になるので、Tier 3（DR）の比重を上げて補います。pgvector 類似度が 0.5 未満 しか出ない場合、提案書冒頭に 「本案件は弊社未経験カテゴリのため、DR 中心の構成」 と明示します。これも捏造防止の一環。
Q3. 誤情報が混ざっていたら？
すべての主張に「データソース引用」が付くため（10.1）、誤情報は引用を辿って即時に確認できます。Phase D で営業さんから「これ違うよ」と指摘されたら、Sonnet 4.6 が該当章を書き直します。さらに、誤情報の元になったソース（特定の Gmail スレッドや過去案件）に「不正確タグ」を打って次回以降の参照から除外する機構（v2.0）も検討。
Q4. 出力フォーマットを pptx にできる？
v1.0 では MD + HTML のみ。pptx は v3.0 で対応予定。理由：python-pptx の placeholder 制御は実装工数の大部分を占めるため、まず MD/HTML で「中身の精度」を社内検証する段階的アプローチを採用しています。営業から強い要望があれば優先度を上げます。
Q5. Phase D の対話を途中で切って Phase E に行ける？
はい。いつでも [これで仕上げる] ボタン を押せます。3 往復で「もう十分」と感じたら Phase E に行って OK。
Q6. 同じ案件で 2 回目以降の編集（提出後の改訂版作成）は？
過去のアーティファクト（v0.5〜v0.9）は S3 に保管されており、@TeamAgent 続きから ACME美容液X で再開できる設計（v1.1 で対応予定）。Phase 0〜B はスキップしていきなり Phase D に入る流れになります。
Q7. 16 名の営業が同時に Phase B を走らせたら？
Phase B は Composer Fargate Task（1 task = 1 案件）で動くため、並列実行可能。同時 16 案件 = 16 Task。コスト面では各 Task のリサーチ時間 5〜10 分 × Sonnet 4.6 ＋ DR API の従量課金。詳細は実装計画書 v2.1 のコスト試算参照。
Q8. クライアントが秘密保持を強く要求する案件で、外部 DR を使いたくない場合は？
DR を OFF にして Tier 0〜2（自社データのみ）で組むモードを用意する予定。Phase A のモーダルに [ ] 外部 DR を使わない チェックボックスを追加（営業から要望が多ければ即時対応）。
Q9. 過去のキャッチボール（メール / Slack）の参照範囲はどこまで？
本人 Gmail は OAuth で全期間。他人 Gmail（横展開）は会社方針合意の範囲内（直近 24 ヶ月）。Slack は Bot Token で参加チャンネルのみ（チャンネル外は不可）。すべて監査ログ（CloudWatch Logs / 90 日保持）に記録されます。
Q10. Phase A の「条件確認」をスキップしたい営業と、必要な営業が分かれそう。
個人設定で切り替え可能にする予定（Q1 で確認）。デフォルトは「確認あり」、慣れた営業は /teamagent settings auto-confirm on でスキップ可能。
