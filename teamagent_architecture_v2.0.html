1.本書について

1.1目的

本書は TeamAgent v2.0 のシステムアーキテクチャを C4 model（Context / Container / Component / Code） の C1〜C3 レベルで記述する設計図書である。仕様書 v2.0 が「何を作るか（What）」を、実装計画書 v2.0 が「いつ・いくらで作るか（When / How much）」を扱うのに対し、本書は 「どう構築するか（How）」 を担当する。AWS 上の物理配置・コンポーネント分割・データフロー・運用設計までを一貫して示す。

1.2対象読者

FDE / 実装担当第3〜5章（AWS / 共通基盤 / 機能別）→ 第8〜9章（運用・セキュリティ）
SRE / インフラ担当第3章（AWS） → 第8章（監視） → 第9章（セキュリティ）
セキュリティ・監査担当第3.3章（IAM）→ 第9章（セキュリティ全般）
プロダクトオーナー第2章（全体像）→ 第11章（拡張性・ロードマップ）

1.3記述レベル（C4 model）

本書では Simon Brown が提唱する C4 model に従い、抽象度の異なる複数のビューを段階的に提示する。実装の詳細（クラス図・コードレベル C4）は実装フェーズで別途記述する。

レベルビュー本書での扱い
C1System Context（システム境界とアクター）第2.1章で記述
C2Container（実行単位・データストア・外部連携）第2.2章 + 第3章 で記述
C3Component（コンテナ内のコンポーネント分割）第4章 + 第5章 で記述
C4Code（クラス・関数レベル）本書ではスコープ外（実装フェーズで作成）

3.3IAM 設計（Bot SA / OAuth client / Service Roles）

最小権限（Least Privilege）原則に基づき、各実行コンテキストに専用の IAM Role を付与する。16名分の Gmail OAuth は Google 側のクライアント1個 + 16名分の refresh token という構成（Google Cloud OAuth Client は社内 Workspace 内アプリのため 外部公開不要、verification 不要）。

ID 種別名称権限スコープ
IAM RoleSlackReceiverLambdaRoleSecrets Manager: slack/* 読取 / SQS: SendMessage / CloudWatch Logs
IAM RoleSuggesterLambdaRoleSecrets: slack/* anthropic/* / RDS: pgvector 読取 / Bedrock invoke（fallback）
IAM RoleMailWorkerLambdaRoleSecrets: gmail-oauth/*（16名分）slack/* anthropic/* / RDS 読書
IAM RoleVideoAnalyzerLambdaRoleSecrets: vseo/* dpro/* buzzmiru/* / S3 PutObject / RDS 読書
IAM RoleComposerFargateRole上記すべて + Drive API 用 SA + Gemini DR 4経路（Vertex / Bedrock / Tavily / Brave / A6X）
OAuth ClientGoogle Workspace Internal Appgmail.readonly / gmail.compose / drive.readonly — 内部アプリのため verification 不要
Slack AppTeamAgent BotBot Token のみ。chat:write chat:write.public commands im:history channels:history
Service AccountDrive Service Account共有設定された社内 Drive フォルダのみ readonly（営業16名の OAuth とは独立）
設計判断：Gmail OAuth は internal app
Google Workspace 管理者により 社内ドメイン限定の internal アプリとして登録するため、Google verification プロセス（通常 4〜6週間）が不要。Phase 1 完了前に管理者と連携して登録を進めることでリスク R1（OAuth 遅延）を回避する。

4.2Gmail OAuth フロー（16名分の token 管理）

Google Workspace の internal app として OAuth クライアントを1個登録し、営業16名がそれぞれ初回ログイン時に同意する。各営業の refresh_token は Secrets Manager に gmail-oauth/<user_id> のキーで保管。Mail Worker は実行時にユーザー ID から refresh token を取得し、access token を都度発行する。

#ActorTarget動作 / 結果
1営業（初回）→Slack: /teamagent connectOAuth 開始コマンド
2Slack Receiver→API GW: /oauth/startGoogle 同意 URL を生成して DM 送信
3営業→Google 同意画面internal app のため警告なしで同意
4Google→API GW: /oauth/callbackcode を返す
5OAuth Lambda→Google: /tokencode → refresh_token + access_token 取得
6OAuth Lambda→Secrets Managergmail-oauth/U12345 として KMS 暗号化保管
7OAuth Lambda→RDS: users登録完了フラグを更新
8OAuth Lambda→Slack: 営業 DM「接続完了 ✅」を通知

表 4.2Gmail OAuth 初回接続フロー（営業1名分）。8 ステップ・所要 1〜2 分。

4.3Slack Bolt（イベント受信 / Block Kit / ephemeral）

Slack Bolt for Python を Lambda 上で動作させ、Slack Events API の検証・ルーティング・ボタン応答を処理する。3秒以内に ack を返す制約 を満たすため、長時間処理は SQS / Step Functions に逃がす非同期パターンを採用する。

受信イベント処理応答形態
app_mention（@TeamAgent + URL）Video Analyzer or Composer に SQS 経由で fan-out進捗 thread 投稿 + 完了時 chat.update
message（チャンネル）Suggester（機能①）の正規表現フィルタを発火chat.postEphemeral で本人にだけ表示
message（DM, button_click）機能② のボタン操作（深掘り / 編集 / 下書き保存）blocks 更新 + Gmail drafts.create
view_submission（モーダル）機能③ Phase A の検索条件確認モーダル送信Phase B 起動 + thread 投稿
scheduled（EventBridge）Cron 起動（朝バッチ / 週次バッチ / 死活監視）各営業 DM / マネージャー DM

Block Kit パターン

12.用語集

C4 model
Simon Brown が提唱するソフトウェアアーキテクチャ記述法。Context（C1）→ Container（C2）→ Component（C3）→ Code（C4）の4階層で抽象度を整理する。本書は C1〜C3 を扱う。
VPC / サブネット / NAT GW
AWS のネットワーク構成要素。VPC = 仮想ネットワーク、サブネット = VPC を分割した論理区画、NAT GW = プライベートサブネットからインターネットへ片方向通信を可能にするゲートウェイ。
Multi-AZ
RDS や Fargate を複数のアベイラビリティゾーンに分散配置する構成。AZ-a 障害時に AZ-c へ自動フェイルオーバーする。
pgvector
PostgreSQL の拡張機能。vector 型を持ち、コサイン距離・L2距離・内積による類似検索（HNSW / IVFFlat 索引）を提供する。本書では事例・動画・メールの埋め込みを格納する。
Embedding
テキストや画像を固定次元のベクトルに変換した表現。本書では 1536 次元（OpenAI text-embedding-3-small 互換）を採用。
Claude Agent SDK
Anthropic が提供する Agent 構築用の SDK。サブエージェント・ツール呼び出し・対話履歴管理を抽象化する。
サブエージェント
Orchestrator から呼び分けられる個別の Agent。本書では Suggester / Mail Worker / Composer / Video Analyzer の4種を定義する。
Pattern Bravo+
Gemini Deep Research の冗長化パターン。Vertex AI を主経路、Bedrock Claude WebSearch / Tavily+Brave / A6X を fallback とする4経路構成。
ephemeral
Slack chat.postEphemeral API で送信される「特定ユーザーにのみ表示」される一時メッセージ。機能① で活用。
Block Kit
Slack のリッチメッセージ記述形式。section / actions / divider / file / button などのブロックを組み合わせて UI を構築する。
サーキットブレーカー
連続失敗時に後続呼び出しを即時失敗にする耐障害パターン。Buzzmiru の不安定対策（§7.2）で採用。
5フェーズ生成ワークフロー
機能③ の動作を Phase A〜E の5段階で表現したもの。検索条件設計 → 並列リサーチ → 5割版生成 → 対話型仕上げ → 営業最終調整。
Internal App（Google Workspace）
Google Workspace の内部アプリ登録。組織ドメイン内ユーザーのみ利用可能で、Google 側の verification プロセスが不要。
Service Account（GCP）
人間ユーザーではないアプリケーション用の Google Cloud アカウント。本書では Drive 共有フォルダの読取に使用する。
SLO / SLI
Service Level Objective / Indicator。SLI = 計測可能なメトリクス（例：応答時間 p95）、SLO = SLI の目標値（例：≤ 5秒）。本書 §8.2 で定義。
map-reduce
分散処理パターン。Phase B の6並列リサーチ（map）→ Phase C で統合（reduce）の構造で採用。
動画分析データ層
VSEO Analytics（自社運用）+ dpro + Buzzmiru の3サービスを束ねた論理レイヤー。本書では1つの外部データソースとして扱う。
Adapter Pattern / Facade
3つの異種 API を1つの統一インターフェース（Video Analyzer Lambda）として扱う設計パターン。Buzzmiru の不安定さを呼び出し側から隠蔽する。

12.3 一行サマリー

TeamAgent v2.0 は AWS ap-northeast-1 上に Lambda × 4 + Fargate × 1 + RDS db.r7g.large（Multi-AZ）+ EventBridge + Secrets Manager + S3 + SQS + CloudWatch の9主要サービスで構築する。Slack を唯一の入口とし、Claude Agent SDK が4種のサブエージェント（Suggester / Mail Worker / Composer / Video Analyzer）をオーケストレートする。Gmail OAuth は Internal App として16名分管理し、pgvector が社内ナレッジ DB（cases / video_meta / mail_excerpts / audit_log）の中核を担う。Pattern Bravo+ により Gemini DR を4経路冗長化し、Buzzmiru の不安定性は5分死活監視 + サーキットブレーカーで耐性を確保する。
