1.本書について

1.1目的

本書は、TeamAgent v2.0 の 機能要件・非機能要件・受入基準 を体系的に定義する。v2.0 仕様書（構想・システム概要）が「何を実現するか」「なぜ必要か」を語るのに対し、本書は 「どのレベルで満たされていれば完了とみなすか」 を測定可能な形で定める。FDE および開発担当者が実装の Definition of Done（DoD）として参照する。

1.2対象読者

FDE / 実装担当第4〜10章（要件本体）を主に参照。各 Phase 終了時に DoD を満たしているかを照合
QA / テスト担当第4〜5章の受入基準と第9章のテスト要件を参照
マネージャー / プロダクトオーナー第2章スコープ、第3章ユーザーストーリー、第10章受入基準を参照
セキュリティ / 法務担当第8章セキュリティ・プライバシー要件を参照

1.3関連文書

文書名役割
teamagent_overview_v2.0.html構想・システム概要仕様書（上位文書）。本書のすべての要件はこの仕様書の記述と矛盾しない。
teamagent_implementation_plan_v2.0.html実装計画書（コスト + スケジュール）。本書の Phase 構成と整合する。

1.4要件記法と凡例

本書の各要件は、ID・タイトル・詳細・優先度・受入基準の5列表で記述する。

FR-X-YY機能要件。X は機能番号（1〜5）、YY は連番
NFR-YY非機能要件（性能・可用性・拡張性・運用性・ユーザビリティ）
SEC-YYセキュリティ・プライバシー要件
DR-YYデータ要件（pgvector・各データ層の格納形式）
INT-YY外部連携要件（API 認証・レート制限・フォールバック）
TST-YYテスト要件（単体・結合・受入・性能・セキュリティ）
DoD-PXPhase X の Definition of Done
必須 Must — リリース判定の必達条件
推奨 Should — 強く推奨。リリース時に 80% 以上満たすこと
任意 May — 望ましい / 将来対応可

3.2営業のユーザーストーリー

US-S-01 ／ 営業
朝出社したら、その日返信が必要なメールが Slack DM に整理されているので、5件すべてに対して案 A/B/C を見ながら判断したい。
US-S-02 ／ 営業
競合動画を Slack に貼って @TeamAgent で 2分以内に 6観点の構造化分析を受け取り、自分の案件に活かしたい。
US-S-03 ／ 営業
クライアント名・商材・URL を投げるだけで、10〜30分で提案書 5割版（MD/HTML）が出来上がり、対話で 8割まで仕上げたい。
US-S-04 ／ 営業
同僚に @田中さん 事例ある？ と聞いた瞬間、自分にだけ過去事例3件が表示され、田中さんを煩わせずに済むようにしたい。
US-S-05 ／ 営業
朝サマリーで気になるメールに「💬 もっと深掘り」を押し、「価格根拠を入れて」と指示するだけで、改訂版の本文が即時に提示されてほしい。
US-S-06 ／ 営業
プライベートな（家族・私信）メールは、AI には絶対に読まれたくない。Gmail ラベルでオプトアウトできるようにしてほしい。
US-S-07 ／ 営業
提案書の改訂対話は、5往復程度で完了し、最終調整は自分の手で行いたい。AI が勝手に完成版を出してこないでほしい。
US-S-08 ／ 営業
作成された返信ドラフトは Gmail 下書きフォルダに保存され、自分で内容確認の上、Gmail から送信したい（自動送信はされたくない）。

3.3マネージャーのユーザーストーリー

US-M-01 ／ マネージャー
月曜朝に 営業16名の案件状況が🟢🟡🔴で集約された DM が届き、1on1 の準備時間を 1 時間 → 10 分に短縮したい。
US-M-02 ／ マネージャー
2週間返信なしなどのリスク案件が🔴で自動的に浮上し、見逃しを防ぎたい。
US-M-03 ／ マネージャー
気になる案件は 🔍 個別案件を深掘り ボタンで その営業のメール履歴・Slack 文脈を一画面で確認したい。
US-M-04 ／ マネージャー
受注角度判定の根拠（どのメール・どの発言を見て判定したか）を 説明可能な形で提示してほしい。
US-M-05 ／ マネージャー
サマリー配信頻度を 週次 / 日次で切り替えられ、繁忙期は日次に切り替えたい。

3.4FDE / 運用のユーザーストーリー

US-F-01 ／ FDE
月次 API コストを 機能別・営業別に把握でき、上限超過の予兆をアラートで受け取れるようにしたい。
US-F-02 ／ FDE
営業から「変な提案書が出た」と報告された時、該当の対話履歴とプロンプトを辿って原因を特定できるようにしたい。
US-F-03 ／ 運用
Buzzmiru などの外部 API がダウンしても、機能全体は縮退運用で稼働し、自動で復旧時に取得を再開してほしい。
US-F-04 ／ 運用
退職者が出た場合、OAuth refresh token の即時失効と関連レコードのマーキングを 1 コマンドで実行したい。
US-F-05 ／ セキュリティ
「誰のメールが、いつ、誰の依頼で、何のために読まれたか」を 監査ログから 90日間遡って確認したい。

4.機能要件（FR）

各機能の要件を ID 体系（FR-X-YY）で列挙する。各要件には 優先度（必須 / 推奨 / 任意）と 受入基準（測定可能な完了条件）を付与する。Phase 順序は ④ → ② → ③ → ①（+ Phase 2.5 で⑤）。

4.1機能① Slack 自動サジェスト（FR-1-XX）

Phase 4受動・ephemeral所要 3〜5 秒

営業同士の Slack メッセージを受動的に監視し、@メンション付きの質問を検出した時のみ、質問者本人にだけ過去事例 Top3 を ephemeral で配信する。

ID タイトル 詳細 優先度 受入基準
FR-1-01 Slack message イベント受信 対象チャンネル（明示的に Bot を追加されたチャンネル）の message イベントを Slack Bolt で受信する。 必須 登録済み15チャンネルでメッセージ送信時に 100% Lambda が起動。CloudWatch Logs に受信ログが残る。
FR-1-02 トリガー条件の正規表現フィルタ (a) @メンション含む、(b) 質問キーワード（事例 / 参考 / 知ってる？等）含む、(c) 業界・案件・商材キーワード含む の3条件を AND 判定する Stage 1 軽量フィルタ。 必須 テストケース50件（陽性25 / 陰性25）に対し精度 90% 以上、再現率 85% 以上。
FR-1-03 Claude SDK 助け要請判定 Stage 1 を通過したメッセージを Claude SDK 軽量モデルで「助けを求めているか Yes/No」判定する Stage 2。 必須 判定 1.5秒以内・トークン使用量 1リクエスト 200 token 以下・Yes/No 構造化出力 100%。
FR-1-04 pgvector 類似事例検索 Top3 判定が Yes のメッセージを query embedding 化し、pgvector で過去事例（提案書・成功事例）を Top3 取得。 必須 検索レイテンシ 500ms 以下。コサイン類似度 0.7 以上の結果のみ返却（該当なしの場合はサジェスト送信を抑止）。
FR-1-05 ephemeral 配信 Slack chat.postEphemeral API で、質問者本人にだけ表示されるメッセージを送信する。 必須 メンション先や他のチャンネルメンバーには 100% 表示されない（Slack UI 検証で確認）。送信遅延 5秒以内。
FR-1-06 独り言・挨拶の除外 メンションなしの独り言、挨拶、お礼、クロージング、TeamAgent 自体への質問はサジェストしない。 必須 陰性テストケース25件のうち 23件以上（92%以上）でサジェストが発火しないこと。
FR-1-07 サジェスト件数上限 同一スレッドで連続発火しないよう、5分以内の再サジェストは抑制する。 推奨 同一ユーザー × 同一スレッドの 5分以内サジェストは 0件であること。
FR-1-08 フィードバック収集 サジェストカードに「役に立った / 邪魔だった」のリアクションボタンを添付する。 推奨 クリックされた値が pgvector の suggestion_feedback テーブルに記録される。
FR-1-09 機能オプトアウト 個別ユーザー単位で「サジェスト機能を無効化」できる設定。 推奨 DM コマンド /teamagent suggest off でオプトアウト。記録は users.suggest_enabled = false。
FR-1-10 深掘りへの誘導 サジェスト末尾に「詳しく作るなら @TeamAgent と呼んでください」と機能③（提案生成）への動線を提示する。 任意 ephemeral カードの末尾に文字列が含まれていること。

4.2機能② Mail ワークフロー（FR-2-XX）

Phase 2毎朝バッチ対話型深掘り

毎朝の Cron で全営業の Gmail を並列スキャンし、返信が必要なメールに対して案 A/B/C を生成、Slack DM で配信。深掘り対話を経て Gmail 下書きに保存する。

ID タイトル 詳細 優先度 受入基準
FR-2-01 EventBridge Cron 朝バッチ 営業日（月〜金）の 8:00 JST に EventBridge から Lambda を起動する。 必須 cron 式 cron(0 23 ? * SUN-THU *)（UTC）で起動。営業日のみ実行。土日祝日は停止可能。
FR-2-02 Gmail スキャン（営業16名並列） 各営業の本人 OAuth で「昨日0時以降の新着・自分宛・返信未済」メールを取得。Lambda 並列実行で 16名同時。 必須 16名のスキャンを 5分以内で完了。Gmail API rate limit に抵触しない（1ユーザー 250 quota units/sec 以下）。
FR-2-03 返信要否判定 Claude SDK で「返信が必要そうか」を判定。ニュースレター・社内連絡・自動配信は除外。 必須 テストデータ100件（要返信50・除外50）で精度 85% 以上、誤除外率 10% 以下。
FR-2-04 案 A/B/C ドラフト生成 各メールに対して「丁寧お礼系 / 早期ピッチ系 / 日程調整系」など3案を生成する。 必須 3案ともに件名・本文・差出人署名が揃っている。各案の文体が異なり、ボタン UI で選択可能。
FR-2-05 本人文体学習 本人の過去送信メール（直近30件）から文体パターン（敬称・改行・締めの言葉）を学習し、ドラフトに反映する。 推奨 同一営業の過去メール5件と AI 生成ドラフト5件を並べ、文体一致度を主観評価で 70% 以上。
FR-2-06 Slack DM 配信 各営業の Slack DM に「メール件数 + 案 A/B/C カード + ボタン」を集約配信する。 必須 他営業の DM には絶対に配信されない。Slack Block Kit でカード表現。
FR-2-07 深掘り対話モード 「💬 もっと深掘り」ボタン押下時、Slack スレッド内で TeamAgent との対話モードに入る。 必須 スレッド内で営業が自然言語指示を投げると、30秒以内に改訂版本文が返却される。
FR-2-08 文脈拡張（過去スレッド・他営業同クライアント） 深掘り時、過去のメールスレッド・他営業の同クライアント対応・Drive 関連提案を参照する。 必須 参照したソース（メール ID / Drive ファイル ID）を出力末尾に明記する。
FR-2-09 Gmail 下書き保存 「📝 Gmail 下書きに保存」ボタンで Gmail API drafts.create を呼び、本人 Gmail の下書きフォルダに保存。 必須 Gmail Web UI の下書きフォルダに 5秒以内に出現。差出人・宛先・件名・本文が揃っている。
FR-2-10 送信権限の不付与 OAuth スコープに gmail.send を含めない。送信は必ず本人 Gmail で実施。 必須 Google Cloud Console の OAuth 同意画面で gmail.send が要求されないこと。
FR-2-11 Gmail ラベル opt-out Gmail ラベル TeamAgent-Exclude が付いたスレッドは AI スキャン対象外。 必須 ラベル付きスレッドが Gmail 検索結果から除外される。CloudWatch Logs に「除外件数」が記録される。
FR-2-12 配信時刻のカスタマイズ 営業個人ごとに配信時刻を 7:00〜10:00 で 30分刻みで設定可能。 推奨 /teamagent mail time 8:30 等のコマンドで設定可能。設定値は users.mail_dispatch_time に保持。

4.3機能③ 提案コンテンツ生成（FR-3-XX）

Phase 35フェーズ対話型所要 10〜30 分

5フェーズ生成ワークフロー（A: 検索条件設計 → B: 並列リサーチ → C: 5割版生成 → D: 対話型仕上げ → E: 営業最終調整）で、Markdown / HTML 提案コンテンツを対話的に生成する。

ID タイトル 詳細 優先度 受入基準
FR-3-01 Phase A 検索条件設計（30秒） クライアント名・商材・URL から、ターゲット・競合分析範囲・DR テーマ・過去事例検索条件を文脈推定し、営業に確認する。 必須 推定結果が4要素（ターゲット / 競合 / DR / 過去事例）すべて埋まる。OK ボタンで Phase B に進む。

8.セキュリティ・プライバシー要件（SEC-01〜12）

v2.0 仕様書 第11章「プライバシー & セキュリティ」を要件レベルに展開する。「会社の脳として AI を活用する」会社方針を前提とし、他人メール参照を許容する代わりに監査・フィルタ・opt-out の3層で保護する。

ID タイトル 詳細 優先度 受入基準
SEC-01 Gmail OAuth スコープ最小化 スコープは gmail.readonly + gmail.compose のみ。gmail.send は付与しない（送信は必ず本人）。 必須 OAuth 同意画面に gmail.send が表示されないこと。
SEC-02 Slack 権限最小化 Bot Token のみ。User Token 不使用。チャンネルは明示的に Bot を追加されたものに限定。 必須 Slack App の OAuth スコープが最小権限（Tier 1 のみ）に設定されている。
SEC-03 他人メール参照の社内合意 「会社の脳として AI を活用する」方針について、書面または社内通達で全従業員に周知済み。 必須 社内ガイドライン文書 + OAuth 同意時の同意文 + 営業オンボーディング資料の3点が揃っている。
SEC-04 監査ログの完全性 誰のメールが、いつ、誰の依頼で、何のために読まれたかを CloudWatch Logs に記録（90日保持）。 必須 audit_log テーブルで accessor / accessed / purpose / timestamp の4必須項目が 100%埋まる。
SEC-05 個人 vs 業務メールの分離 件名・送信元ドメインで「家族・私信・社内雑談」を初期フィルタ。誤判定時に Gmail ラベルでカスタマイズ可能。 必須 プライベートメール100件のサンプリングで検出率 80% 以上、誤検知率 10% 以下。
SEC-06 Gmail ラベル opt-out 営業が Gmail ラベル TeamAgent-Exclude を付けたスレッドは AI 参照対象外。 必須 ラベル付きスレッドが mail_corpus に取り込まれない。既存レコードは is_excluded = true に即時更新。
SEC-07 退職者対応 OAuth refresh token を即時失効・関連 pgvector レコードに退職者マーク（参照禁止）。 必須 退職処理 1コマンドで完了。退職者のメールが検索結果に出現しない。
SEC-08 機密情報のマスキング クレジットカード番号・マイナンバー・SSN など PII を自動検出して LLM 入力前にマスキング。 推奨 PII 検出ライブラリ（Presidio 等）で 95% 検出率。マスキング後 LLM に送信。
SEC-09 マネージャー権限の制限 機能⑤ はマネージャーロールのみ実行可能。一般営業からは参照不可。 必須 非マネージャーが機能⑤ コマンド実行時に「権限がありません」エラー。
SEC-10 監査ログ閲覧権限 audit_log の閲覧は FDE / セキュリティ担当のみ。営業・マネージャーは自身関連分のみ閲覧可能。 必須 RDS IAM ロール分離。営業ロールから audit_log への SELECT が拒否される。
SEC-11 外部送信防止 顧客メールの内容を外部 LLM API（Anthropic / Vertex AI）に送る前に、PII マスキング + 件名/本文の要約を行う。 推奨 外部 API への送信トークン量がメール原文の 30% 以下（要約効果）。
SEC-12 セキュリティ訓練 四半期ごとにペネトレーションテスト + Runbook 机上訓練を実施。 推奨 四半期報告書に訓練結果が記載される。Critical 指摘は 30日以内に修正。

12.用語集

TeamAgent
本要件定義の対象となる Slack AI エージェントの総称。営業16名 + マネージャー数名が利用する社内専用エージェント。
5機能（機能① 〜 ⑤）
① Slack 自動サジェスト ／ ② Mail ワークフロー ／ ③ 提案コンテンツ生成 ／ ④ 動画 URL 分析 ／ ⑤ 営業進捗サマリー（②に内包）
5+1 データ層
📧 Gmail / 💬 Slack / 📁 Google Drive / 🎬 動画分析データ（VSEO + dpro + Buzzmiru）/ 🔍 Gemini DR / 🐦 X — 6つの読み取り対象。
4共通基盤
Claude Agent SDK / Gmail OAuth / Slack Bolt / pgvector — 5機能すべてが再利用するインフラ。
FR（Functional Requirement）
機能要件。FR-X-YY 形式（X = 機能番号、YY = 連番）。
NFR（Non-Functional Requirement）
非機能要件。性能・可用性・セキュリティ・拡張性・運用性・ユーザビリティの 6カテゴリ。
SEC / DR / INT / TST
セキュリティ要件 / データ要件 / 外部連携要件 / テスト要件。
DoD（Definition of Done）
各 Phase の受入基準。すべて満たした時点で Phase 完了。
SLO / SLA
SLO = Service Level Objective（社内目標）／ SLA = Service Level Agreement（外部約束）。本書ではコア機能 SLO 99.5% を採用。
RTO / RPO
RTO = Recovery Time Objective（復旧目標時間）4時間 ／ RPO = Recovery Point Objective（データ損失許容時間）1時間。
ephemeral
Slack の chat.postEphemeral API で送信される「特定ユーザーにのみ表示」される一時メッセージ。機能① で活用。
opt-out
営業が「これは AI に見せたくない」スレッドを Gmail ラベル TeamAgent-Exclude で除外する機構。
Pattern Bravo+
Gemini Deep Research の 4経路冗長化方針（Vertex AI 公式 / Bedrock Claude WebSearch / Tavily / Brave）。
5フェーズ生成ワークフロー
機能③ の動作。Phase A: 検索条件設計 → B: 並列リサーチ → C: 5割版生成 → D: 対話型仕上げ → E: 営業最終調整。
本人メール / 他人メール
本人メール = 当該営業自身の Gmail（要件定義・条件設定に利用）。他人メール = 同社他営業の Gmail（社内ナレッジ DB として利用）。
動画分析データ層
VSEO Analytics + 動画広告プロ（dpro）+ Buzzmiru の3サービスを束ねた論理レイヤー。
受注角度
機能⑤ で使う指標。クライアントの温度・進捗ステージから受注確度を「高（80%以上）/ 中（40〜79%）/ 低（40%未満）」で分類。
