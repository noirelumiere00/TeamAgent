1.本書について

1.1目的

本書は「TeamAgent」の構想とシステム概要を、5つの機能 × 5つのデータ層 × 4つの共通基盤という整理で記述する。v1.5 までの記述は提案書生成と Mail Agent を中心に組み立てられていたが、現場ヒアリングの結果、実際に必要とされている機能が 4つ＋マネージャー向けサマリー1つであることが判明したため、本書はそれをベースに全面改訂したものである。

1.2対象読者

営業メンバー第2章（5機能の使い方）→ 第6〜9章（個別機能詳細）
マネージャー第2章 → 第7章（Phase 2.5 進捗サマリー）→ 第12章（ロードマップ）
経営層第3章 → 第4章 → 第12章 → 実装計画書
推進オーナー / 実装担当（FDE）第5章 → 第6〜9章 → 第10〜12章 → 実装計画書

🆕 v2.1 の意図
IBM レビューにより、機能① の「受動監視によるサジェスト乱入」は (a) 誤検知時の信頼失墜リスクが高く、(b) Slack message event を全件監視するための pgvector 検索コストが嵩むことが指摘された。v2.1 では 営業が明示的に呼ぶ ＝ 信頼ゼロからやり直しではなく、価値を取りに行ったときだけ価値が返る 設計に変更。あわせて期間を現実的な 8.5ヶ月に延長し、各スプリント末のテスト・観察を明示する。

1.4用語定義

TeamAgent
本書で扱う Slack AI エージェントの総称。営業16名 + マネージャー数名が利用する社内専用エージェント。
5機能
① Slack 自動サジェスト（@TeamAgent メンション起動 / v2.1 で受動監視から変更）／② Mail ワークフロー（朝サマリー＋返信深掘り）／③ 提案コンテンツ生成（多段階対話型）／④ 動画 URL 分析（リアクティブ）／⑤ 営業進捗サマリー（マネージャー向け、② に内包）
5データ層
📧 Gmail / 💬 Slack / 📁 Google Drive / 🎬 動画分析データ（VSEO + dpro + Buzzmiru） / 🔍 Gemini DR / 🐦 X — のうち X はオンデマンド検索のため層数としては5＋1とも数える。
4共通基盤
Claude Agent SDK ／ Gmail OAuth ／ Slack Bolt ／ pgvector — 5機能すべてが再利用する共通レイヤー。
動画分析データ層
VSEO Analytics（自社運用）+ 動画広告プロ（dpro）+ Buzzmiru の3サービスを束ねた論理レイヤー。本書では1つのデータソースとして扱う。
本人メール / 他人メール
本人メール = 当該営業自身の Gmail（要件定義・条件設定に利用）。他人メール = 同社他営業の Gmail（過去事例ナレッジ DB として利用、会社方針として合意済み）。
5フェーズ生成ワークフロー
提案コンテンツ生成（機能③）の動作を Phase A〜E の5段階で表現したもの。検索条件設計 → 並列リサーチ → 5割版生成 → 対話型仕上げ → 営業最終調整。
ephemeral
Slack の chat.postEphemeral API で送信される「特定ユーザーにのみ表示」される一時メッセージ。機能① で活用。

2.営業向け使い方ガイド（5機能）

📌 一言で言うと

TeamAgent は Slack の中で動く5つの AI 機能です。営業は Slack だけ開いていれば、(1) 困っていれば自動でヒントが届き、(2) 朝にメール対応の整理が届き、(3) 頼めば提案書が出来上がり、(4) 動画 URL を投げれば構造化分析が返り、(5) マネージャーには営業進捗が自動で集約されます。新しいツールを覚える必要はありません。

2.15つの機能の俯瞰

🎬
機能 ④ ／ Phase 1
動画 URL 分析
TikTok / YouTube Shorts の URL を投げると、EG情報・メディア・界隈・フック・訴求軸・CTA・表現の 6観点で構造化分析。
所要 1〜3 分・リアクティブ
📬
機能 ② ／ Phase 2
Mail ワークフロー
朝にメール対応サマリーが DM で届き、返信を作りたい時はその場で 深掘り対話 → Gmail 下書きに保存。
毎朝バッチ + オンデマンド深掘り
📊
機能 ⑤ ／ Phase 2.5
営業進捗サマリー
マネージャー向け。各営業の 案件進捗・受注角度・リスク要因 を Mail / Slack から自動集約。
日次 or 週次バッチ（② に内包）
📝
機能 ③ ／ Phase 3
提案コンテンツ生成
クライアント名・商材・URL を投げると、5フェーズで Markdown / HTML 提案資料を5割→8割と対話的に作る。
所要 10〜30 分（対話型）
💡
機能 ① ／ Phase 4 🆕 v2.1
Slack 自動サジェスト
営業が @TeamAgent で事例を呼び出すと、本人だけに見える形で過去事例3件を ephemeral で提示。打鍵 2 秒で会社の記憶を引き出せる。
メンション起動・本人のみ表示

2.2機能ごとの体験例（Slack UI モック）

各機能を、実際の Slack 画面でどう見えるかを再現したモックアップで説明する。営業・マネージャーが触る場所はすべて Slack の中で完結する。各機能の末尾には 「フロー図」 を添付し、入力 → 処理 → 出力の流れを視覚化している。

3.背景と課題

3.14機能発生の経緯（営業ヒアリングより）

v1.5 までの仕様書は提案書生成と Mail Agent を中心に組み立てられていたが、現場ヒアリングを通じて営業が実際に必要とする機能が 4つ＋マネージャー向け1つ であることが判明した。各機能が解こうとしているペインは以下のとおり。

機能解くペイン頻度

② Mail ワークフロー「朝の返信の山が重い」「文体やトーンを毎回考えるのが疲れる」「返信遅延でクライアント温度が下がる」毎朝

3.2営業現場のペイン（再構成）

v1.5 で示した「営業20時間/案件」の問題は依然として最大ペイン（機能③）であるが、機能① ② ④ で 「日常の小さな摩擦」 も併せて解くことで、TeamAgent は「資料生成ツール」から「営業の AI 同僚」へと位置付けが進化する。マネージャー向け機能⑤ により、組織レベルの可視化も同時に進む。

7.機能② Mail ワークフロー

本機能は 朝のサマリー → 返信深掘り → 下書き保存 の一連のワークフローと、その延長として 営業進捗サマリー（Phase 2.5、機能⑤） をマネージャー向けに提供する。

7.1朝のサマリー

毎朝（推奨：8:00 〜 10:00）に EventBridge Cron で発火し、営業16名分の Gmail を並列スキャン。「返信が必要そうなメール」を Claude SDK が判定し、各営業の Slack DM に集約配信する。

項目仕様
発火時刻cron(0 8 ? * MON-FRI *) JST 想定（営業日のみ）
スキャン対象各営業の Gmail で「昨日0時以降の新着・自分宛・返信未済」
判定Claude SDK で「返信が必要そうか」を判定（ニュースレター・社内連絡は除外）
配信先各営業の Slack DM（他人には見えない）
配信内容メール件数 + 案A/B/C ドラフト + ボタン（下書き保存・編集・深掘り）

7.2返信深掘りモード

朝サマリーで「もっと深掘り」ボタンを押した瞬間に、Slack スレッドで TeamAgent との対話が始まる。これは 機能③（提案コンテンツ生成）の縮小版 と捉えるべきで、データソースの参照範囲は限定されるが Phase D（対話型仕上げ）と同質の体験。

段階動作所要
① 文脈拡張過去スレッド・他営業の同クライアントメール・Drive の関連提案を参照30秒
② ドラフト改訂営業の指示（「数字根拠を入れて」「価格に触れて」など）を反映1往復30秒
③ 文体調整本人の過去メールから文体パターンを学習・反映同上
④ 下書き保存Gmail API drafts.create で本人 Gmail に保存1秒

7.3営業進捗サマリー（Phase 2.5、機能⑤）

②のインフラ（Gmail OAuth + Slack DM + pgvector）を流用し、マネージャー向けのレポート機能を Phase 2.5 として段階的に追加 する。実装コストは最小（① ② ③ で構築済みの基盤を集約するだけ）。

項目仕様
発火頻度週次（月曜 9:00）or 日次選択可能
データソース全営業の Mail スレッド（直近2週間）+ 案件 Slack チャンネル
判定軸受注角度（高 / 中 / 低）・リスク要因・推奨アクション
判定ロジックClaude SDK が文脈から「クライアントの温度」「進捗ステージ」を分類
配信先マネージャー個別 DM（営業情報の共有レベルは会社方針に従う）
形式角度別グルーピング（🟢 高 / 🟡 中 / 🔴 要フォロー）
⚠️ プライバシー前提
本機能は他営業のメールを横断参照するため、第11章「プライバシー & セキュリティ」で扱う社内合意（会社の脳として AI を活用する方針）に基づく。監査ログとフィルタ機構が前提条件。

8.機能③ 提案コンテンツ生成

本機能は当初構想の本丸であり、TeamAgent の最大価値ドライバー。多段階対話型で動作し、AI が一発で完成版を出すのではなく、5割版を起点に対話的に8割まで仕上げる「Co-Authoring」スタイルを採用する。出力は初期実装で Markdown + HTML の2形式（pptx は将来対応）。

9.機能① Slack 自動サジェスト 🆕 v2.1 で全面書き換え

本機能は @TeamAgent 直接メンションで起動するリアクティブな機能。機能③（提案コンテンツ生成）と同じ起動方式だが、軽量な事例検索（pgvector Top3）にフォーカスする。Phase 4 で実装し、それまでに整備された pgvector ナレッジ DB を活用する。

v2.0 → v2.1 の設計変更
v2.0 までは「同僚への @メンション + 質問形」を Slack message event で受動監視し、ephemeral でサジェストを横入りさせる設計だった。IBM レビューで以下の指摘を受けた：(1) 全 message event を pgvector で類似度検索するコストが高い、(2) 誤検知時に「変なタイミングで AI が乱入する」体験が信頼を損なう、(3) ステルス起動はプライバシー懸念を生む。v2.1 では 営業が明示的に呼ぶときだけ動く 設計に変更し、誤検知ゼロ・打鍵 2 秒・応答 3 秒の軽量機能とする。

9.1動作条件と ephemeral 配信

営業が @TeamAgent を呼んでクエリ（例：「ACME 美容液の事例ある？」「過去の SaaS 採用ブランディング思い出して」）を投げると、3 秒以内に過去事例 Top3 を ephemeral で本人にのみ表示 する。送信は Slack chat.postEphemeral で行い、チャンネルの他メンバーには見えない。DM でもチャンネルでも動作する。

動作する条件動作しない条件
@TeamAgent + クエリ（事例 / 過去 / 思い出して 等）同僚への @メンション（v2.0 で動いていた条件は廃止）
業界・案件・商材キーワードを含む明示的な呼びかけ@TeamAgent なしの独り言・チャンネル雑談
DM チャンネルでの直接対話挨拶・お礼・クロージング（クエリでない）

9.2検出ロジック

v2.1 では 受動監視を撤廃 したため、検出ロジックは大幅に簡素化された。app_mention イベントで起動するため、誤起動の心配がない。

[app_mention event @TeamAgent] ↓ [Stage 1: クエリ抽出（Claude SDK 軽量）] メンションテキストから検索意図を抽出 "ACME 美容液の事例" / "BtoB SaaS 採用" など ↓ [Stage 2: pgvector 類似度検索 Top3] 過去案件・提案資料・Mail スレッドから関連 3 件 ↓ [Stage 3: chat.postEphemeral で本人にだけ送信] "過去事例 3 件、関連性高めで [リスト] [📁 案件1] [📁 案件2] [📁 案件3] [📝 これで提案書つくる] ← 機能③ にルーティング"

3 ステージに簡素化（v2.0 では 4 ステージ）。応答時間は 3 秒以内（pgvector Top3 検索 + Claude SDK 軽量 1 回）。

9.3機能①と機能③の意図ルーター 🆕 v2.1 新規

v2.1 で機能① も @TeamAgent 起動になったため、機能③（提案コンテンツ生成）と 起動方式が同じ になった。営業が @TeamAgent ACME 美容液の事例ある？ と書いた場合と @TeamAgent ACME 美容液の提案書つくって と書いた場合は、システム側で意図を判別して振り分ける必要がある。

分類キーワード（例）振り分け先動作
事例リコール「事例」「過去」「思い出して」「recall」「教えて」機能①（軽量）pgvector Top3 → ephemeral（3 秒）
提案コンテンツ生成「提案」「資料」「作って」「compose」「クライアント=」機能③（重い）5 フェーズ生成（10〜30 分の対話型）

判定は Claude SDK の軽量プロンプト（< 100 tokens）で行う。confidence < 0.7 の場合は確認 ephemeral を返す：「『事例検索』として動きますか？それとも『提案書生成』として動きますか？」と本人にだけ確認し、ボタン選択で確定させる。これにより誤ルーティングを最小化する。

[@TeamAgent + メッセージ受信] ↓ [Claude SDK で意図分類（軽量プロンプト）] confidence ≥ 0.7：そのまま振り分け confidence < 0.7：確認 ephemeral を本人に返す [事例検索] [提案書生成] ← ボタン選択 ↓ [機能① または 機能③ のサブエージェントへ]

9.4slash command オプション 🆕 v2.1 新規

@TeamAgent + 自由文の代わりに、打鍵負担を減らしたい営業向けに Slack slash command も提供する：

コマンド動作応答
/teamagent recall <キーワード>機能①（事例リコール）に直接ルーティング、意図判定をスキップephemeral で Top3
/teamagent compose <クライアント> <商材>機能③（提案コンテンツ生成）に直接ルーティングPhase A（検索条件設計）から起動

slash command を使うと意図ルーターを通らないため、応答が常に高速・確定的になる。営業が機能を覚えてきたタイミングで使い始めるオプションとして提供する（強制ではない）。

11.プライバシー & セキュリティ

11.1会社方針

TeamAgent が他営業のメールを横断参照する設計について、「会社の脳として AI を駆使し、使えるものは使う」 方針が会社として合意済み。これを前提に設計を進める。ただし実装上は以下の最低要件を満たす。

11.2監査・フィルタ・opt-out

要件仕様
監査ログ誰のメールが、いつ、誰の依頼で、何のために読まれたかを CloudWatch Logs に記録（90日保持・SOC2 相当）
個人 vs 業務メールの分離件名・送信元ドメインで「家族・私信・社内雑談」を初期フィルタ。誤判定時のラベル設定機能
opt-out 機構営業が「これは AI に見せたくない」スレッドを Gmail ラベル TeamAgent-Exclude で除外可能
退職者対応OAuth refresh token を即時失効・関連 pgvector レコードに退職者マーク（参照禁止）
Gmail OAuth スコープgmail.readonly + gmail.compose（送信権限は付与しない＝送信は必ず本人）
Slack 権限Bot Token のみ。User Token は不使用。チャンネルは明示的に追加されたものに限定

12.実装ロードマップ

12.1Phase 順序の論理

順序：④ → ② → ③ → ①（+ Phase 2.5 で⑤）。理由：各 Phase が次 Phase のインフラを段階的に積み上げ、最終 Phase（①）の時点でほぼ全データソースと共通基盤が揃っている状態にする。v2.1 では各 Phase 内を 2 週間スプリント（合計 17 Sprint）で反復し、各 Sprint 末にテスト・観察・修正の時間を確保する。

🆕 v2.1 のスケジュール変更
v2.0 の 6ヶ月想定に対し、IBM レビューで「並行化前提が楽観的すぎる」「テスト工程が不可視」「Phase 終了直後の観察期間がない」と指摘され、v2.1 では 8.5ヶ月（17 Sprint） に再構成した。各 Phase の最後に観察スプリント（テスト・修正・営業フィードバック反映）を追加し、実装 55% / テスト 25% / 修正・安定化 20% の配分を実装計画書に明示する。
